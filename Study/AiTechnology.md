# AI 技术沉淀：从 AI 运价项目理解 Agent 设计

## 文档定位

这份文档的目标不是复述项目做了什么，而是把这个项目里真正有技术价值的部分抽出来，沉淀成一套可持续复用的 Agent 认知框架。

它服务两个目的：

1. 帮我基于这个项目系统理解 Agent，不只停留在“会调模型、会接工具”的层面。
2. 把项目里的技术判断抽象成未来可复用的设计方法，用于后续做别的 Agent 项目、技术分享、面试复盘、咨询陪跑。

这份文档分两条线来讲：

1. 项目内：这个 AI 运价项目是怎么设计的，为什么这样设计。
2. 项目外：抛开这个项目，一般业务 Agent 应该怎么设计，哪些设计是通用的，哪些是强业务定制的。

---

## 一、先给一个总判断

### 1. 这个项目不是“Prompt 工程项目”

它的主体不是一堆 Prompt，而是一个带状态、带路由、带工具、带结果后处理的受控对话系统。

如果只把它理解成：

- 用 DeepSeek 做意图识别
- 抽几个参数
- 调一次运价接口

那就低估了这个项目的技术含量。

### 2. 这个项目也不是“自由 Agent”

它不是那种完全开放的 autonomous agent，不会让模型自己决定：

- 是否调用工具
- 调哪个工具
- 工具参数是什么
- 什么时候结束

它更准确的定位是：

> 一个受控工作流型 Agent。

这类系统的关键不是“让模型更自由”，而是：

1. 用状态机守住业务边界
2. 用规则和结构化状态控制执行
3. 让模型主要参与“理解”和“补全”，而不是参与“不受控执行”

### 3. 这类业务 Agent 的核心矛盾

一个业务 Agent 最难的点，不是接模型，不是写接口，而是平衡这两件事：

1. 智能性
   - 能理解自然语言
   - 能承接上下文
   - 能追问和解释
2. 可控性
   - 不乱查
   - 不误调工具
   - 不污染状态
   - 不破坏前后端协议

这个项目的全部技术价值，本质都围绕这对矛盾展开。

## 二、这个项目里的 Agent 架构到底是什么

结合当前代码与文档，这个项目可以抽象成 7 层。

## 1. 协议层

职责：

1. 定义前后端如何对接
2. 保证系统不是“黑盒对话”，而是一个有输入输出约束的服务

本项目对应：

- `POST /api/chat`
- SSE 输出：`text / context / done / error`
- 前端每轮必须回传 `context`

为什么重要：

如果没有协议层，后端无法稳定维护会话状态，前端也无法承接补参、澄清、结果分析这些多轮能力。

这个项目里一个很对的判断是：

> 把 `context` 当成协议的一部分，而不是临时拼接的内部变量。

## 2. 会话状态层

职责：

1. 保存当前轮和跨轮运行所需的最小状态
2. 把“这轮对话现在进行到哪一步”显式化

本项目对应：

- [graph/state.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\graph\state.py)

当前状态大致分成 6 组：

1. 基础对话状态
   - `messages`
   - `intent`
   - `query_subtype`
2. 运价槽位状态
   - `sfg`
   - `mdg`
   - `inputWeight`
   - `inputVol`
   - `hbrq`
   - 以及可选筛选字段
3. 控制流状态
   - `missing_slots`
   - `query_ready`
   - `query_completed`
4. 待澄清 / 待动作状态
   - `pending_clarify_*`
   - `pending_action_*`
   - `pending_reuse_confirmation`
5. 结果分析状态
   - `latest_quote_result`
   - `result_analysis_intent`
   - `result_reference_field`
   - `response_mode`
   - `quantity_mode`
6. RAG 状态
   - `retrieval_query`
   - `retrieved_docs`
   - `rag_answer`

为什么重要：

很多人做 Agent 项目会把“上下文”理解成一段聊天记录。这个项目提醒了一个更成熟的认识：

> 真正可维护的 Agent，不是只记聊天文本，而是把关键运行状态结构化。

## 3. 会话语义层

职责：

1. 判断用户这句话当前在做什么
2. 区分“改条件”“答澄清”“问结果”“问能力”“闲聊”等不同行为

本项目的演进过程很能说明问题。

一开始只有粗粒度意图：

- `rate_query`
- `rag`
- `unknown`

后面逐渐扩展成：

- `result_analysis`
- `result_reference`
- `support_info`

再往下一层继续抽象：

- `query_subtype`
- `response_mode`
- `quantity_mode`

这里最重要的认识是：

> 用户一句话的“领域意图”不够用，还必须进一步识别“会话行为类型”。

比如下面这些都属于运价域，但行为完全不同：

1. `上海到洛杉矶多少钱`
   - 新询价
2. `改成900公斤`
   - 条件更新
3. `这周六`
   - 澄清回答
4. `最便宜的是哪个`
   - 结果分析
5. `这是多少号的`
   - 结果引用解释
6. `你有哪些功能`
   - 能力说明
7. `你好`
   - 闲聊/非业务短句

如果没有会话语义层，系统就会把这些全都粗暴塞进：

- 继续询价
- 结果分析
- 兜底

然后表现得“像工作流、不像 Agent”。

## 4. 动作选择层

职责：

1. 在理解了用户行为之后，决定系统下一步该做什么
2. 区分“理解”和“执行”

本项目对应：

- [graph/agent.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\graph\agent.py)
- [graph/nodes.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\graph\nodes.py)

当前主路由是：

1. `intent_node`
2. 根据 `intent` 进入：
   - `slot`
   - `result_analysis`
   - `result_reference`
   - `support_info`
   - `rag_retrieve`
   - `fallback`

这一层的价值在于：

> 模型负责帮助理解，但真正的执行路线由代码决定。

这就是“受控 Agent”的核心。

## 5. 执行层

职责：

1. 真正完成某个动作
2. 动作可以是抽槽位、追问、调用工具、检索知识库、分析结果

本项目里的执行节点包括：

1. `slot_node`
2. `ask_node`
3. `tool_node`
4. `result_node`
5. `result_analysis_node`
6. `result_reference_node`
7. `rag_retrieve_node`
8. `rag_answer_node`

为什么这种拆法合理：

1. 检索和生成分开，便于调试
2. 分析结果和继续查价分开，避免误重查
3. 结果引用解释和结果分析分开，便于未来增强

## 6. 结果表达层

职责：

1. 决定系统如何把结果给用户
2. 不只是“答什么”，还包括“怎么答”

本项目后期开始明确这个抽象，主要体现在：

- `response_mode`
- `quantity_mode`

这两个字段非常关键。

### `response_mode`

它决定：

1. `summary_only`
2. `summary_plus_table`
3. `summary_plus_clarify`
4. `single_result`

也就是：

1. 只总结
2. 总结 + 表格
3. 总结 + 反问
4. 只给一条结果

### `quantity_mode`

它决定：

1. 用户要一条
2. 还是要一组

这就是为什么：

- `最便宜的一条`

和

- `最便宜的有哪些`

虽然都包含“最便宜”，但结果完全不同。

这层抽象的价值很大，因为它说明：

> 一个成熟的 Agent，不只是要知道“要做什么”，还要知道“该怎么把结果交付给用户”。

## 7. 记忆与生命周期层

职责：

1. 管理哪些状态只在本轮有效
2. 哪些状态需要跨轮保留
3. 哪些结果应该进入后端运行时记忆

本项目对应：

- `context`
- `SESSION_RUNTIME_STORE`
- `latest_quote_result`
- `reset_quote_context`

这个层面有两个特别好的设计：

### 设计 1：前端只回传必要上下文，不回传整批结果

前端回传的是槽位状态、澄清状态、展示模式等轻量上下文，而不是整个报价结果集。

完整结果留在后端内存：

- `SESSION_RUNTIME_STORE[session_id]`

这样做的好处：

1. 前端负担更小
2. 结果分析链更稳定
3. 不需要在请求体中传大对象

### 设计 2：显式的会话结束信号

通过：

- `reset_quote_context`

告诉后端：

1. 当前这轮报价分析结束
2. 清空活跃上下文
3. 但不清聊天展示

这解决的是 Agent 项目里非常容易忽略的一个问题：

> 一轮业务对话的逻辑生命周期，不等于聊天窗口的视觉生命周期。

## 三、这个项目里最值得沉淀的几个设计点

## 1. 用状态机守住业务边界

这个项目最值得学的第一点，就是没有把执行权直接放给模型。

为什么：

1. 运价查询有真实业务成本
2. 参数不全时不能乱查
3. 状态继承错了会直接查错
4. SSE 协议和前端上下文不能乱

所以系统采取的不是：

- 模型说该查就查

而是：

1. 先识别
2. 再补槽
3. 再判定是否 ready
4. ready 才允许进入 `tool_node`

这是一种非常典型、非常正确的业务 Agent 设计思路。

### 通用原则

如果你的 Agent 会调用真实业务工具：

1. 工具执行必须有代码护栏
2. 模型可以帮助理解，但不能直接决定危险或高成本动作
3. `state -> route -> guard -> execute` 的链路要清晰

## 2. 把“继续查询”和“结果分析”分成两条链

这是本项目中后期最有价值的架构升级之一。

很多项目会犯一个错误：

用户只要继续说话，就默认重新查一次。

这会导致：

1. 成本浪费
2. 语义错乱
3. 用户感觉系统“没听懂”

本项目通过 `latest_quote_result` 和 `result_analysis_node`，明确分出：

1. 继续查询链
   - 改日期
   - 改重量
   - 改包装
2. 结果分析链
   - 最便宜
   - 直达有哪些
   - CZ 有没有
   - 只看托盘

### 通用原则

只要你的 Agent 有“先查数据，再围绕数据提问”的场景，就应该拆出：

1. 查询链
2. 分析链
3. 引用解释链

否则所有追问都会退化成重复查询。

## 3. 把澄清问题显式建模

这个项目后期开始从：

- `pending_clarify_slot`
- `pending_clarify_message`

进一步升级到：

- `pending_action_type`
- `pending_action_prompt`
- `pending_action_payload`

这说明项目已经开始意识到：

> 澄清不是一个临时问题，而是一类正式的系统状态。

### 为什么澄清状态要显式建模

因为系统不仅要会问：

- 这周六还是下周六？
- 起运港还是目的港？
- 货好日期还是航班日期？

还必须会消费下一轮回答。

如果不把“我现在在等用户回答什么”这个状态存下来，第二轮回复就很容易掉到 `fallback`。

### 通用原则

任何需要追问的 Agent，都应该统一设计一个 pending action 模型，而不是每个问题各自发明一套字段。

一个通用结构可以是：

```python
pending_action_type
pending_action_prompt
pending_action_payload
pending_action_retry_count
```

## 4. 输出策略必须从业务逻辑里抽出来

这是很多 Agent 项目经常忽略，但实际很关键的一点。

用户并不总是需要：

1. 一大段自然语言
2. 或一大张表格

很多时候真正需要的是：

1. 一句总结
2. 一句总结 + 一张小表
3. 一句总结 + 一句反问
4. 一条结果

本项目后期开始抽象 `response_mode`，就是在解决这个问题。

### 通用原则

无论做哪个领域 Agent，只要有结构化结果，最好单独建模：

1. `answer_intent`：做什么
2. `response_mode`：怎么答
3. `quantity_mode`：答几条

如果不拆这三层，系统很容易出现：

1. 识别对了，但输出形式很蠢
2. 明明用户只要一条，系统却给一堆
3. 明明该追问，却直接硬答

## 5. 协议层和运行态记忆要分清

本项目里一个很成熟的设计，是把：

1. 前端上下文
2. 后端运行时记忆

分开了。

### 前端上下文负责什么

负责跨轮续接最小必要状态，例如：

1. 当前槽位
2. 是否查询完成
3. 待澄清状态
4. 展示模式

### 后端运行态负责什么

负责留存不适合前端回传的大对象，例如：

1. `latest_quote_result`
2. 活跃结果分析会话

### 通用原则

做多轮 Agent 时，不要把所有东西都放进聊天消息里，也不要把所有东西都扔给前端。

应该至少分三类记忆：

1. 对话消息记忆
2. 结构化会话状态
3. 后端运行态对象记忆

## 四、抛开这个项目，一般业务 Agent 应该怎么设计

下面给出一套更通用的设计框架。

## 1. 一个通用业务 Agent 的推荐分层

推荐从上到下分 8 层：

1. 协议层
2. 状态层
3. 会话语义层
4. 动作选择层
5. 工具执行层
6. 结果分析层
7. 输出策略层
8. 记忆与观测层

### 1. 协议层

设计内容：

1. 输入格式
2. 输出事件类型
3. 会话 ID
4. 上下文回传机制

如果协议层不先定，后面所有多轮能力都会变脆。

### 2. 状态层

设计内容：

1. 领域槽位
2. 控制流标记
3. 澄清状态
4. 结果对象状态
5. 展示状态

状态层不要等需求变复杂了再补，一开始就要留出清晰边界。

### 3. 会话语义层

设计内容：

不要只分“大意图”，还要分“行为类型”。

典型行为类型包括：

1. 新任务
2. 条件更新
3. 回答澄清
4. 询问当前结果
5. 要求筛选/排序
6. 要求解释字段
7. 询问系统能力
8. 闲聊

这一层是决定 Agent “聪不聪明”的核心。

### 4. 动作选择层

设计内容：

1. 什么时候查
2. 什么时候问
3. 什么时候解释
4. 什么时候过滤
5. 什么时候兜底

推荐原则：

- LLM 可以判断
- 代码决定执行

### 5. 工具执行层

设计内容：

1. 工具 schema
2. 执行前校验
3. 超时/错误处理
4. 多工具编排

这里的原则是：

> 工具层必须 deterministic，不能过度依赖自然语言模糊执行。

### 6. 结果分析层

设计内容：

1. 排序
2. 过滤
3. 分组
4. TopN
5. 解释字段

只要工具会返回结构化结果，这一层就非常值得单独建设。

### 7. 输出策略层

设计内容：

1. 回答形式
2. 粒度
3. 条数
4. 是否追问

很多系统“功能做出来了但体验差”，问题都不在识别，而在这里。

### 8. 记忆与观测层

设计内容：

1. 会话状态如何保存
2. 运行对象放在哪
3. 怎么打印日志
4. 怎么排查误路由

一个 Agent 项目做大以后，能不能 debug，取决于这层。

## 2. 业务 Agent 的推荐建模顺序

一个新项目一般不要一上来就追求“像人一样自然”，推荐顺序是：

### 第一步：把业务主链打通

先解决：

1. 能不能完成主要任务
2. 参数齐不齐
3. 工具能不能稳定执行

### 第二步：把会话边界打清楚

再解决：

1. 什么时候复用旧状态
2. 什么时候重置
3. 什么时候要澄清

### 第三步：把结果对象抽出来

再解决：

1. 查询完成后，结果能不能复用
2. 用户能不能围绕当前结果继续提问

### 第四步：再提升“Agent 感”

最后再解决：

1. 更细的会话语义
2. 更自然的输出策略
3. 更强的引用解释能力

这比一开始就追求“全能智能体”稳得多。

## 3. 什么场景不适合做自由 Agent

以下场景通常不适合一开始就做自由 Agent：

1. 工具调用成本高
2. 参数必须准确
3. 错一次就有明显业务损失
4. 前后端协议要求稳定
5. 结果必须结构化可审计

像运价、金融、审批、报关、客服工单这些，都更适合：

> 受控 Agent，而不是自由 Agent。

## 五、设计 Agent 时最容易踩的坑

## 1. 把“聊天记录”当成全部上下文

问题：

聊天记录只是原始语料，不等于可执行状态。

后果：

1. 系统很难知道当前缺什么
2. 很难知道是不是在等用户回答某个问题
3. 很难知道当前这轮围绕的是哪一个结果对象

正确做法：

1. 聊天记录保留
2. 结构化状态并存

## 2. 所有追问都临时写 if/else

问题：

时间一长，追问逻辑会爆炸。

后果：

1. 规则互相冲突
2. 很难复用
3. 第二轮回答很容易断链

正确做法：

统一 pending action 模型。

## 3. 只做“查”，不做“结果对象复用”

问题：

系统每轮都重查，用户追问一个字段也查一次。

后果：

1. 成本高
2. 用户觉得系统没理解
3. 无法做结果分析能力

正确做法：

把结构化结果对象保存下来，拆出结果分析链和结果引用链。

## 4. 识别层和展示层耦合在一起

问题：

识别到“最便宜”就直接走某个固定模板，识别到“包装”就直接走另一套模板。

后果：

很难灵活支持：

1. 一条 vs 多条
2. 总结 vs 表格
3. 结果解释 vs 结果筛选

正确做法：

至少拆开：

1. `analysis_intent`
2. `response_mode`
3. `quantity_mode`

## 5. 一开始就追求“全自动 Agent”

问题：

这会让系统看起来很智能，但很容易变成不可控。

后果：

1. 误调工具
2. 错误继承状态
3. 难以 debug
4. 用户体验不稳定

正确做法：

先做受控 Agent，再逐步在判断层增加智能。

## 七、如果以后我再做一个 Agent，我会怎么设计

如果抛开这个项目，让我从头做一个业务 Agent，我会优先按下面的顺序设计：

## 1. 先定义协议，不先写 Prompt

包括：

1. 请求格式
2. SSE/HTTP 返回格式
3. 会话 ID
4. 上下文回传结构

## 2. 先定义状态，不先堆节点

包括：

1. 业务槽位
2. 控制状态
3. pending action
4. 结果对象状态
5. 输出模式状态

## 3. 先定义主链，再补“像 Agent 的部分”

先回答：

1. 主任务怎么闭环
2. 什么条件下允许执行
3. 失败怎么兜底

再去做：

1. 结果分析
2. 结果引用
3. 更自然的追问

## 4. 一开始就预留结果对象

哪怕第一版不做结果分析，也建议预留：

- latest_result

因为大部分业务场景迟早会出现“查完以后继续问结果”的需求。

## 5. 一开始就预留输出模式

至少预留：

1. `summary_only`
2. `summary_plus_table`
3. `summary_plus_clarify`

否则后期很容易写成一堆耦合逻辑。

## 八、一句话收束

这个 AI 运价项目最有价值的技术沉淀，不是“我做了一个能查价的聊天系统”，而是它逼着我理解了一个业务 Agent 真正的骨架：

1. 协议
2. 状态
3. 会话语义
4. 动作选择
5. 工具执行
6. 结果对象
7. 输出策略
8. 生命周期

也让我更清楚地看到：

> Agent 项目真正的难点，通常不在模型本身，而在如何让“理解、状态、执行、结果、输出”这五件事既协同，又可控。

如果把这套框架吃透，这个项目就不再只是一个完成过的业务需求，而会变成一套你以后可以持续复用的 Agent 技术资产。
