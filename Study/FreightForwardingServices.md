# 货代业务沉淀：从 AI 询价项目理解空运询价业务

## 文档定位

这份文档的目标不是解释代码，而是把这个 AI 运价项目背后的货代询价业务抽出来，沉淀成一份可复用的业务资产。

它服务两个目的：

1. 帮我把“AI 询价”背后的业务逻辑真正讲清楚，避免只懂系统，不懂业务。
2. 把这套业务理解整理成以后可以复用的咨询、陪跑、产品设计、需求访谈框架。

这份文档重点不讲模型、不讲 LangGraph，而是专讲：

1. 询价业务本质
2. 用户真实问法
3. AI 询价链路背后的业务判断
4. 结果分析如何服务业务决策
5. 以后做其他货代/物流询价项目时，可以复用的业务框架

---

## 一、先给一个总判断

### 1. AI 询价业务的本质，不是“回答一个价格”

空运询价看起来像一个问题：

- 上海到洛杉矶多少钱？

但业务上它其实至少包含四步：

1. 收集条件
2. 把自然语言转成可查询字段
3. 查询并整理候选方案
4. 帮客户做初步决策或引导下一步动作

所以 AI 询价本质上不是一个“价格问答器”，而是一个：

> 条件收集器 + 结果整理器 + 初步报价顾问 + 业务分流器

### 2. 货代询价不是单轮动作，而是一段会话

真实客户很少一次把所有条件说完整。

更常见的是：

1. 先说航线
2. 再补重量
3. 再补体积
4. 再说日期
5. 查完以后继续问：
   - 最便宜的是哪个
   - 直飞有哪些
   - 某航司有没有
   - 只要一条托盘的

所以这类业务天然是：

> 会话型业务，而不是表单型业务。

### 3. 货代询价不是“查到就结束”，而是“查到以后才开始真正比较”

客户很多时候不是缺一个数字，而是缺一个决策支持。

他真正关心的可能是：

1. 哪条最便宜
2. 哪条更稳
3. 是直飞还是中转
4. 哪个航司更适合
5. 有没有托盘/散货可走
6. 今天没价，明后天有没有类似机会

因此，AI 询价系统如果只会把结果吐出来，那只是“半成品”。

真正有业务价值的，是：

> 查价 + 整理 + 比较 + 引导

## 二、空运询价业务的最小业务模型

为了把 AI 询价讲清楚，先把这个业务抽成一个最小模型。

## 1. 询价对象是什么

在这个项目里，询价对象不是“订单”，而是一个待报价的运输需求片段。

它至少包含：

1. 起运港
2. 目的港
3. 日期
4. 重量
5. 体积

这是当前系统里的核心必填字段。

如果这 5 个没有凑齐，严格意义上就还不是一个完整的可报价请求。

## 2. 可选但重要的业务约束是什么

在项目中，这些不是硬必填，但会显著影响结果：

1. 航班类型
   - 直达 / 中转
2. 包装类型
   - 散货 / 托盘
3. 货类
   - 普货等
4. 航司二字码
   - CA / CZ / MU / CK 等
5. 客户报价 ID / 公布运价

这说明业务上存在两层条件：

1. 让系统“能查”的最低条件
2. 让结果“更贴近客户真实需求”的筛选条件

## 3. 报价结果的最小业务对象是什么

基于当前接口与结果分析能力，一个候选报价方案大致可以抽象成：

1. 航司
2. 航线
3. 直达/中转
4. 货类
5. 包装
6. 预估运费单价
7. 预估运费总价
8. 预估卡车费
9. 合计
10. 航班日期

这说明：

> 用户拿到的不是一个价格，而是一组可比较的候选方案。

## 4. 当前业务里非常关键的隐含规则

### 规则 1：城市不等于机场

客户问的是：

- 上海
- 广州
- 洛杉矶

系统真正要查的是：

- PVG
- CAN
- LAX

这意味着 AI 询价里最早的业务翻译动作，不是价格计算，而是：

> 城市/地区 -> 三字码机场

### 规则 2：重量和体积都不是装饰字段

当前系统要求同时收集：

1. `inputWeight`
2. `inputVol`

背后对应的是航空业务里的计费逻辑。

项目中的规则是：

```text
体积重 = CBM * 1000 / 6
计费重 = max(实重, 体积重)
```

这意味着业务上不是“知道重量就够了”，而是：

> 运价和计费重相关，计费重又由实重与体积共同决定。

也就是说，用户只说：

- 500 公斤

在业务上仍然不够，系统还必须知道体积，才能判断计费基础。

### 规则 3：日期也不是一个简单时间字段

项目里日期至少有三种不同语义：

1. 航班日期
2. 货好日期 / 备货日期
3. 模糊时间偏好
   - 尽快
   - 最近
   - 这周六

这里最容易踩坑的点是：

> 客户说的日期，不一定就是要查的航班日期。

例如：

- `货5月22日准备好`

这更可能是：

- 货好日期

而不是：

- 5月22日航班价格

### 规则 4：客户问的是“价格”，但业务真正交付的是“可选方案”

同一组条件下，接口返回的不是一个价，而可能是多条方案：

1. 不同航司
2. 不同 routing
3. 直飞 / 中转
4. 不同包装/货类

所以 AI 不能只回答：

- 多少钱

而应该回答：

- 有哪些可选方案
- 哪条更便宜
- 哪条更适合

## 三、AI 询价业务为什么天然适合做成对话系统

## 1. 因为客户表达天然不完整

真实客户经常这样问：

1. `上海到芝加哥多少钱`
2. `500公斤`
3. `1个立方`
4. `后天呢`
5. `那香港呢`

这些都不是完整请求，但在真实销售语境里完全正常。

如果系统要求一次性结构化表单输入，虽然规范，但体验上反而不像业务员。

所以对话系统的价值在于：

1. 顺着客户说法接
2. 缺什么补什么
3. 逐步把模糊条件收敛成可查询字段

## 2. 因为客户经常是边想边问

客户不是总能先想清楚所有条件再来问。

更常见的是：

1. 先看一条线有没有
2. 再决定重量体积
3. 再想要不要直飞
4. 再看看某个航司

这类业务天然适合“会话式条件收集”。

## 3. 因为查后追问比首问更接近真实业务

查完以后，客户很少就此结束。

他可能继续问：

1. `最便宜的是哪个`
2. `直达的有哪些`
3. `CZ 有没有`
4. `只给我一条托盘的`
5. `这个是哪个航司`
6. `这是多少号的`

这说明真正的业务价值，很多时候在“查后追问”里，而不是在第一次查价本身。

## 四、用户问法怎么拆：货代询价的 6 类高频会话

这是这份文档最值得长期复用的一部分。

如果以后你做任何货代 AI 询价项目，都可以优先用这 6 类来拆需求、做测试集、做产品设计。

## 1. 新询价类

含义：

- 用户在发起一票新的询价需求

典型问法：

1. `上海到洛杉矶多少钱`
2. `PVG 到 AMS 今天 500公斤 1个立方查下价格`
3. `我有一票货要从广州到悉尼空运`

业务特征：

1. 常常缺字段
2. 有时信息一次说全
3. 有时只说航线

系统目标：

1. 判断是不是新询价
2. 收集必填字段
3. 准备进入查询

## 2. 补参与条件更新类

含义：

- 用户不是发新询价，而是在继续完善或修改当前条件

典型问法：

1. `500公斤`
2. `1.5个立方`
3. `改成托盘`
4. `后天呢`
5. `改成900公斤`

业务特征：

1. 上下文依赖很强
2. 通常是局部字段变化
3. 如果系统不能稳接，会非常像“没听懂”

系统目标：

1. 识别这是续问，不是新单
2. 只更新明确变化的字段
3. 保留未变化的上下文

## 3. 新航线短追问类

含义：

- 用户在上一轮基础上，换航线继续问

典型问法：

1. `那青岛呢`
2. `换广州看看`
3. `从深圳出发呢`
4. `那香港多少钱`

业务特征：

1. 很短
2. 歧义强
3. 很像真人销售对话里的省略表达

系统难点：

1. 这是换起运港还是换目的港
2. 该不该沿用上一轮重量、体积、日期

业务沉淀点：

这类问题说明货代询价里非常重要的一点：

> 航线变化是强信号，比包装、货类、航司变化的业务权重高得多。

## 4. 结果分析类

含义：

- 用户已经拿到一批报价结果，想围绕结果继续筛选和比较

典型问法：

1. `最便宜的是哪个`
2. `直达的有哪些`
3. `只看托盘`
4. `KE 最低多少钱`
5. `给我最便宜的一条`

业务特征：

1. 不是重新查
2. 是对已有结果做分析
3. 对结果展示方式要求很高

系统目标：

1. 不重复调接口
2. 基于当前结果做排序、过滤、分组
3. 给用户可决策的结构化结果

## 5. 结果引用解释类

含义：

- 用户不是要重排结果，而是在问当前结果里的某个属性

典型问法：

1. `这是多少号的`
2. `这个是哪个航司`
3. `这个是直飞还是中转`
4. `最便宜那条的包装类型是什么`

业务特征：

1. 指代强
2. 上下文依赖极强
3. 很像真实业务员继续解释报价单

业务沉淀点：

这是业务 Agent 很像真人顾问的一层能力，因为它开始不是“再查”，而是“解释当前结果”。

## 6. 业务咨询 / 服务信息类

含义：

- 用户仍在当前业务上下文里，但问的不是价格，而是服务相关问题

典型问法：

1. `你有哪些功能`
2. `人工客服联系方式是`
3. `你们还能查什么`

业务特征：

1. 不是闲聊
2. 也不是询价
3. 但和成交、继续沟通、业务转人工有关

系统目标：

1. 不误触发查价
2. 不误进结果分析
3. 给出清晰、可行动的信息

## 五、货代 AI 询价里最容易被低估的业务难点

## 1. 日期语义不是单一字段，而是一类业务歧义

项目已经暴露出一个非常典型的问题：

- `货5月22日准备好`

这句话看起来像日期补充，但业务上其实不一定是航班日期。

所以做货代 AI 询价时，日期至少要拆成三类：

1. 明确航班日期
   - `5月22日飞`
   - `按5月22日航班查`
2. 货好日期 / 备货日期
   - `货5月22日准备好`
   - `5月27号能出货`
3. 模糊时间偏好
   - `越快越好`
   - `最近`
   - `这周六`

业务启发：

> 货代行业里，时间字段不是一个字段，而是一组不同语义的时间约束。

## 2. “最便宜”不是总是同一个意思

客户说“最便宜”，可能至少有三种意思：

1. 合计最低
2. 单价最低
3. 某一筛选条件下最低
   - `散货里最便宜`
   - `直达里最便宜`

所以咨询时、建系统时都要明确：

1. 默认排序依据是什么
2. 用户显式提到“单价”时是否切换口径
3. 筛选和排序的顺序是什么

## 3. “只要一条”是独立业务意图，不是展示偏好

用户说：

1. `给我一条最便宜的`
2. `只要一条直飞的`
3. `推荐一条托盘的`

这不只是“少显示一点”，而是：

> 结果数量目标发生了变化。

业务上，这意味着客户从“想看范围”切换到了“想拿建议”。

这个认知很适合以后做咨询时用：

1. 多条结果 = 浏览/比较
2. 单条结果 = 决策/推荐

## 4. “那青岛呢”这类短问法是高价值场景，不是噪音

很多系统会把这种话当边界 case，但业务上恰恰相反。

这种问法说明：

1. 客户已经进入比较阶段
2. 客户在快速试探不同航线/方案
3. 系统如果接得住，会非常像真人业务员

所以：

> 越口语、越短、越省略的表达，往往越接近真实业务价值场景。

## 六、询价结果为什么不能只“展示”，还要“分析”

## 1. 因为报价结果本身就是一个比较对象

在当前系统里，一次查询可能返回多条结果。

如果系统只是把结果吐出来，用户下一步仍然得自己做这些事：

1. 比总价
2. 比单价
3. 看是不是直达
4. 找某个航司
5. 按包装筛选

而这些恰恰是最适合 AI 来做的工作。

## 2. 当前项目里已经沉淀出的 A 类结果分析框架

基于 `docs/dev2.md` 和 `docs/RequestQuotation.md`，当前可以稳定复用的一套结果分析框架是：

### A1. 价格最低类

业务意图：

1. 最便宜的是哪个
2. 单价最低的是哪个
3. 前三最便宜有哪些

### A2. 航线筛选类

业务意图：

1. 直达有哪些
2. 中转有哪些
3. 直达和中转分别列出来

### A3. 航司筛选类

业务意图：

1. CZ 有没有
2. KE 最低多少钱
3. 不同航司分别怎么报价

### A4. 包装 / 货类筛选类

业务意图：

1. 散货有哪些
2. 托盘有哪些
3. 普货有哪些方案

### A5. 日期比较类

业务意图：

1. 哪天最便宜
2. 未来几天各自是什么价格
3. 今天没价，后面哪天有类似机会

### A6. 组合决策类

业务意图：

1. 直达里最便宜的是哪条
2. 散货里最低的是哪条
3. 某航司直达和中转分别多少钱

### A7. 结果摘要类

业务意图：

1. 帮我总结一下
2. 推荐一条
3. 只给我前 5 条

## 3. 货代业务视角下，“结果分析”比“首问查价”更接近价值创造

首问查价解决的是：

- 有什么价

结果分析解决的是：

- 哪个更合适

后者更接近：

1. 销售判断
2. 方案推荐
3. 客户决策支持

所以以后如果做对外咨询，这里可以明确讲：

> AI 询价项目真正有商业价值的，不是替代“查一次价”，而是把“查后比价、筛选、解释、推荐”这段顾问式工作吃掉一部分。

## 七、货代业务里“查不到”的处理为什么同样重要

很多人做 AI 询价，只盯着“查到有数据”的场景。

但在真实业务里，“没查到”往往更考验系统是否像业务员。

## 1. 当前项目里对无数据的业务处理思路

从 `AI运价.md` 和后续方案可以看出，项目把无数据分成两层：

1. 精确无结果，但有类似结果
2. 精确和类似都没有

对应的话术和动作也不同：

### 情况 1：精确无结果，但有类似结果

系统不应该只说“没有”。

而应该说：

1. 当前精确日期未满足
2. 给出未来几天类似参考方案
3. 让客户继续判断或联系人工

这背后对应的是一个很典型的业务认知：

> 报价业务里，参考价往往也有价值，不是非黑即白。

### 情况 2：完全无结果

系统应转向：

1. 人工联系
2. 服务承接
3. 邮箱/客服引导

这说明业务上“没结果”不是系统终点，而是转人工节点。

## 2. 为什么“类似运价”是一个好业务设计

项目中定义的类似运价策略是：

1. 保持航线不变
2. 保持重量和体积不变
3. 保持可选筛选项尽量不变
4. 只向后放宽日期 7 天

这个策略很有代表性，因为它说明：

> 类似运价不是随便给个差不多的价，而是控制变量，只放宽一个最可能导致无结果的字段。

这对以后做其他业务咨询也很有用：

### 通用咨询原则

当业务查询无结果时，优先考虑：

1. 哪个字段是“最容易导致空结果”的变量
2. 能否只放宽这个变量，而不破坏客户核心约束

## 八、把货代 AI 询价做成咨询资产时，可以复用的业务框架

这一部分是最适合以后拿去做咨询、陪跑、需求访谈、产品规划的。

## 框架 1：先问“询价业务的最小成交条件”是什么

你可以先用 4 个问题摸清一个客户或业务团队的询价业务：

1. 客户最少提供哪些字段，你们才会报价
2. 哪些字段不是必填，但会显著影响结果
3. 哪些条件错了，业务风险最高
4. 没有精确结果时，你们通常怎么处理

对于本项目，对应答案就是：

1. 起运港
2. 目的港
3. 重量
4. 体积
5. 日期

以及：

1. 航班类型
2. 包装类型
3. 货类
4. 航司

## 框架 2：先拆“用户问法类型”，再谈 AI 能力

不要一开始就问：

- 能不能上 AI

应该先问：

1. 新询价怎么问
2. 补参怎么问
3. 改条件怎么问
4. 查后怎么继续问
5. 没结果时客户会怎么追问
6. 客户什么时候需要人工接入

这套框架在这个项目里就对应：

1. 新询价
2. 补参
3. 新航线短追问
4. 结果分析
5. 结果引用
6. 服务信息 / 转人工

## 框架 3：把“查价能力”和“顾问能力”分开看

做咨询时可以直接用这个判断：

### 查价能力

解决：

1. 能不能把需求变成接口查询
2. 能不能拿回结构化结果

### 顾问能力

解决：

1. 能不能帮客户比
2. 能不能帮客户筛
3. 能不能帮客户解释
4. 能不能在无结果时给出下一步建议

真正能卖出价值的，通常是第二部分。

## 框架 4：把无结果处理当成业务设计的一部分

咨询时一定要问：

1. 没价时你们现在怎么处理
2. 是直接结束，还是给类似方案
3. 什么时候必须转人工
4. 转人工时给什么信息

很多系统失败，不是查不到，而是查不到以后太像“死机”。

## 框架 5：把结果展示当成业务设计，不只是 UI 设计

这点非常关键。

本项目里从“默认全量表格”变成“默认两类包装最低价 + 可展开全部”，本质上不是 UI 优化，而是业务表达优化。

咨询时可以直接问：

1. 用户第一次看到结果时，是更需要“全量浏览”还是“最优摘要”
2. 什么时候需要全部展开
3. 什么时候应该只给一条推荐
4. 什么时候该先总结再追问

这套思路可直接复用到其他报价、比价、方案推荐类业务。

> 货代 AI 询价真正有价值的地方，不在于替客户“问到一个价格”，而在于帮客户完成从模糊询问到条件收敛、从一批报价到初步决策、从无结果到下一步行动的整段业务过程。
