## 一、项目一句话定义

这是一个面向货代官网聊天场景的 AI 运价 Agent 项目。

它的核心目标不是做一个通用聊天机器人，而是围绕两个明确业务能力建立统一对话入口：

1. 空运运价查询
2. 物流业务知识问答（RAG，当前仍处于建设中/预埋中）

系统最终统一暴露 `POST /api/chat`，前端通过 SSE 接收流式文本，并通过 `context` 把会话状态原样带回后端，完成多轮询价、补参、结果追问与后续扩展。

## 二、整体时间线概览

从当前仓库材料看，这个项目大致经历了 5 个阶段：

1. 立项与骨架期
   - 时间参考：`2026-03-11` 到 `2026-03-20`
   - 核心是先把“AI 询价”这件事定义成一个可落地的后端 Agent 服务，而不是一个散乱的 Prompt Demo。
2. RAG 预埋与知识库探索期
   - 时间参考：`2026-03-24` 到 `2026-04-09`
   - 核心是先把知识库能力的边界、目录、配置和链路预埋好，但不破坏报价主链。
3. 询价主链稳定化期
   - 时间参考：`2026-05-06` 到 `2026-05-08`
   - 核心是把“能查”升级成“能稳定多轮补参、能给出可展示结果、能处理无结果与类似结果”。
4. 结果分析与会话控制期
   - 时间参考：`2026-05-13` 到 `2026-05-21`
   - 核心是让系统从“只会查价”变成“查完以后还能围绕结果继续分析”，并补足请求取消、会话重置、上下文边界控制。
5. Agent 化反思与下一轮规划期
   - 时间参考：`2026-05-25` 到 `2026-05-27`
   - 核心是从大量联调问题和业务反馈中反思：为什么系统已经能跑，但仍然“像工作流、不像 Agent”，并开始提出会话语义层、输出策略层、数量模式等更高层抽象。

## 三、阶段复盘

## 阶段 1：立项与骨架期

### 时间锚点

- `2026-03-11`：`chatUse.md`
- `2026-03-20`：`CLAUDE.md`

### 这一阶段在做什么

这一阶段的核心不是“优化效果”，而是把项目先立起来。

从现有材料看，项目最初就不是朝“开放聊天”去做的，而是很早就确定了一个非常工程化的方向：

1. 后端是一个独立服务
2. 入口是统一聊天接口
3. 输出是 SSE
4. 内部不是单次 Prompt，而是带状态的 Agent/工作流
5. 业务目标先聚焦空运询价，不先做过宽的能力铺设

### 已经形成的早期共识

从 `CLAUDE.md` 可以看出，项目在早期就已经明确了几件重要的事：

1. 技术栈以 `FastAPI + LangGraph + LangChain + DeepSeek` 为主
2. 项目结构不是临时脚本，而是：
   - `main.py`
   - `graph/`
   - `tools/`
   - `rag/`
3. 对话系统的本质是“槽位补全 + 工具调用 + 结果输出”
4. RAG 在早期只是预留，不抢主线

### 这一阶段的关键价值

这一步最重要的不是功能多少，而是方向定得比较对：

1. 没把它做成纯前端拼接的聊天页面
2. 没把它做成只有一个大 Prompt 的试验品
3. 很早就选择了“有状态的 Agent 编排”路线
4. 很早就把工具调用和对话状态分开了

### 我的复盘判断

这里有一个很值得沉淀的认知：

> 一个业务 Agent 项目的第一步，不是把模型接上，而是先决定系统是否有“状态、节点、工具、协议”这些骨架。

这个项目在建立初期，至少在架构心智上已经迈过了“玩具 Demo”阶段。

### 推断/归纳

基于 `chatUse.md`、`CLAUDE.md` 与项目结构约束，可以合理推断：

1. 初期主要压力不是模型效果，而是“系统怎么组织”
2. 团队一开始就很重视后端可维护性
3. 项目最早的目标，大概率就是服务前端官网对话窗，而不是做离线分析工具

## 阶段 2：RAG 预埋与知识库探索期

### 时间锚点

- `2026-03-24`：`plan/log.md`
- `2026-03-26`：`ProjectConfig.md`
- `2026-04-09`：`RAG.md`

### 这一阶段的背景

当报价主链骨架基本明确后，项目开始面对第二条长期能力线：物流业务知识问答。

但从文档看，这一阶段没有直接把 RAG 粗暴接进主链，而是采取了更稳的做法：

1. 先补配置
2. 先补目录与职责
3. 先把离线建库、在线检索、生成回答分层
4. 先让 `intent=rag` 具备未来接入位置
5. 暂时不改变 `/api/chat` SSE 协议

### 这一阶段做了什么

根据 `plan/log.md`，3 月 24 日的工作基本形成了一套完整的 RAG 设计闭环：

1. 配置层
   - 在 `config.py` 中预留 embedding、Chroma、RAG 参数
2. 状态层
   - 在 `graph/state.py` 扩展 RAG 中间状态字段
3. 模块层
   - 拆出 `metadata / loaders / cleaner / splitter / embeddings / vector_store / bm25_store / indexer / query_analyzer / retriever / generator / service`
4. 路由层
   - 计划把 `intent=rag` 从兜底提示改成真实检索链
5. 脚本层
   - 增加 `init_kb / rebuild_kb / test_rag`

### 这一阶段为什么重要

虽然从“用户可见功能”看，RAG 还没有真正上线，但这一步的价值很大：

1. 它没有污染报价主链
2. 它把知识问答定义成一条独立能力链，而不是“随手加个 Prompt”
3. 它把未来难题前置思考了：
   - 文档解析
   - metadata
   - 混合检索
   - 重排预留
   - 配置独立

### 这一阶段反映出的工程判断

这一阶段有两个判断值得特别记录：

1. 不是一开始就追求复杂度，而是先把 RAG 跑通
2. 不是直接做多个 Collection，而是先用单 Collection + metadata 过滤

这说明项目在这里的思路是“先最小闭环，再逐步变复杂”，而不是一上来做大而全的知识库系统。

### 项目心态的变化

从这个阶段开始，项目已经不再只是“AI 询价接口”，而是在向“统一物流 Agent”演进。

也就是：

1. 报价查询是一条链
2. 知识问答是一条链
3. 最终统一在一个聊天入口里路由

### 我的复盘判断

这一阶段很适合以后对外讲成一句话：

> 我们没有把 RAG 当成外挂，而是从一开始就把它当成未来会并入统一 Agent 的第二条业务链，所以在状态、目录、配置和路由上都提前预埋了接口。

## 阶段 3：询价主链稳定化期

### 时间锚点

- `2026-05-06`：`plan/ai_quote_plan_2026-05-06.md`
- `2026-05-06`：`plan/2026-05-06_10-57-29.md`
- `2026-05-08`：`plan/2026-05-08_15-14-54.md`

### 这一阶段的背景

到 5 月上旬，项目重心明显回到了运价主链，而且进入了“从能跑到能演示”的阶段。

领导/业务方提出的诉求已经很具体，不再只是“能查价格”，而是：

1. 接口字段变多了，要支持更多参数
2. 缺参时要一次性追问，不要问一次漏一点
3. 结果返回要稳定、可读、适合前端展示
4. 查询不到时不能只说没有，要给类似运价或人工引导
5. 完整查完一轮后，再问新航线时不能错误继承旧参数

### 这一阶段的核心变化

#### 1. 从“能查”升级到“有完整产品口径”

`ai_quote_plan_2026-05-06.md` 很关键，它相当于把 AI 询价从技术实现拉成了产品能力定义：

1. 必填字段和可选字段边界明确
2. 缺参提示方式明确
3. 精确结果、类似结果、完全无结果三层输出策略明确
4. 新完整询价与续问的判定规则开始被设计

#### 2. 多轮补参与上下文复用开始真正成体系

这一阶段的系统不再只是抽槽位，而是在处理一个更棘手的问题：

> 用户的新一句话，到底是在继续上一轮，还是在开始下一轮？

这直接催生了：

1. `query_completed`
2. 新航线不完整询价时清空旧参数
3. 参数合并优先级
4. 类似运价自动兜底

#### 3. 结果展示从 Prompt 转向代码控制

这一点很重要。

从方案文档能看出，项目逐渐意识到：

1. 结果表格结构
2. 类似运价的呈现方式
3. 无结果的固定话术

这些不应该交给模型自由生成，而应该更多由代码层稳定控制。

这其实是业务 Agent 项目很关键的一个进化：

> 把“确定性展示”从模型里拿出来，交给程序控制。

### 典型问题与修复

`2026-05-06_10-57-29.md` 记录了一个很典型的问题：

- 上一轮完整询价后
- 用户只说“青岛到莫斯科”
- 系统错误继承了上一轮重量、体积、日期，直接查价

这暴露出一个本质问题：

> 多轮上下文不是有了就行，关键是何时复用，何时清空。

这轮修复的价值，不只是解决一个 bug，而是开始建立“新询价边界判断”。

### 这一阶段项目心智的变化

从这里开始，项目不再只是“LLM + 工具”，而是明显出现了“会话产品设计”的味道：

1. 什么时候该追问
2. 什么时候该重用上下文
3. 什么时候该清空上下文
4. 什么时候该给类似结果
5. 返回给前端的是什么协议化上下文

### 我的复盘判断

这是项目第一次真正进入“像产品而不只是像脚本”的阶段。

如果以后我要对外讲这个项目，这一阶段可以总结成：

> 我们从 5 月上旬开始，真正把 AI 询价系统当成一个多轮状态机产品来打磨，而不是“用户说一句、模型回一句”的问答接口。

## 阶段 4：结果分析与会话控制期

### 时间锚点

- `2026-05-13`：`plan/2026-05-13_13-24-13.md`
- `2026-05-14`：`plan/2026-05-14_15-34-40.md`
- `2026-05-14`：`plan/2026-05-14_17-11-54.md`
- `2026-05-21`：`plan/2026-05-21_13-29-23.md`
- `docs/dev2.md`
- `Architecture.md`
- `chat.MD`

### 这一阶段的背景

当基础询价链开始稳定后，项目遇到了一个更高阶的问题：

> 用户查到价格以后，不会立刻结束，他们会继续围绕结果问一堆问题。

比如：

1. 最便宜的是哪个
2. 直达的有哪些
3. 某航司有没有
4. 散货有哪些
5. 帮我总结一下

这意味着系统不能只会“重新查”，还要能“围绕当前结果继续分析”。

### 这一阶段最关键的变化

#### 1. 正式拆出“结果分析链”

`2026-05-13_13-24-13.md` 是这一轮的核心节点。

系统从原来的单一询价链，演进成了两条链：

1. 继续查询链
   - 改重量
   - 改日期
   - 改包装
2. 结果分析链
   - 最便宜
   - 直达筛选
   - 航司筛选
   - 结果摘要

这是一次很重要的系统分层。

从这一步开始，项目已经不再是“报价工具封装器”，而是开始具备围绕结果进行后处理的能力。

#### 2. 增加会话生命周期控制

这个阶段引入了几个非常关键、很容易被忽略但很体现工程成熟度的能力：

1. `reset_quote_context`
2. `SESSION_RUNTIME_STORE`
3. `latest_quote_result`
4. 请求取消止损

这意味着系统开始认真处理：

1. 本轮询价何时结束
2. 结果分析上下文如何保留
3. 前端取消请求时后端如何尽快止损

这些能力本身不“炫技”，但非常接近真实产品落地。

#### 3. 系统不再只围绕槽位，而开始围绕“结果对象”工作

之前的多轮更多是：

- 补参数
- 改参数
- 再查询

这个阶段以后，多轮开始变成：

- 已经查到了结果
- 继续围绕这批结果做筛选、排序、解释

这是系统心智的一次升级：

> 从“询价状态机”扩展成“询价 + 结果分析”的复合会话系统。

### 中途暴露出的现实问题

#### 1. 工程兼容性问题

`2026-05-14_15-34-40.md` 记录了 `Asia/Shanghai` / `tzdata` 问题。

这件事的价值不在于技术难度，而在于提醒了一个事实：

> 项目从设计阶段走向联调阶段以后，很多问题不再来自业务逻辑，而来自真实运行环境。

这类问题是产品化项目一定会遇到的：

1. 环境不一致
2. 时区依赖
3. Windows 兼容性

#### 2. 澄清问题会问，但不会稳定消费回答

`2026-05-14_17-11-54.md` 暴露了另一个很典型的问题：

系统会问：

- 这周六还是下周六？

但用户下一轮回复：

- 这周六

系统却掉进了 `fallback`。

这说明项目第一次明显撞上了一个 Agent 级问题：

> 系统不只是要会提问，还要会消费用户对提问的回答。

这催生了：

1. `pending_clarify_slot`
2. `pending_clarify_message`
3. 待澄清状态跨请求回传

### 到 5 月 21 日的进一步反思

`2026-05-21_13-29-23.md` 说明项目已经从“修 bug”转向“讨论行为策略”：

核心问题是：

1. 参数到底该不该复用
2. 多窗口/多聊天时状态如何隔离

这说明项目已经不满足于“功能做出来”，而开始思考：

1. 业务体验是否合理
2. 状态边界是否清晰
3. 架构是否足够支撑下一阶段

### 我的复盘判断

这一阶段是项目真正开始有“系统产品”味道的阶段。

它不只是加了功能，而是开始正面处理几个真正艰难的问题：

1. 结果分析
2. 生命周期
3. 请求取消
4. 澄清状态
5. 会话隔离

如果以后要讲“项目难点”，这一阶段是素材最密集的地方。

## 阶段 5：Agent 化反思与下一轮规划期

### 时间锚点

- `2026-05-25`：`plan/2026-05-25_16-41-14.md`
- `2026-05-26`：一系列方案文档
- `2026-05-27`：`plan/2026-05-27_09-43-16.md`
- `2026-05-27`：`plan/2026-05-27_13-41-01.md`
- `2026-05-27`：`plan/2026-05-27_13-48-11.md`

### 这一阶段的背景

到这个阶段，项目出现了一个非常典型的现象：

1. 它已经能跑
2. 它已经有多轮
3. 它已经有结果分析
4. 但在真实演示和口语化交互里，仍然会显得“有点傻”

于是项目开始进入一个更上层的反思：

> 为什么系统看起来更像工作流，而不像一个真正会理解上下文的 Agent？

### 这一阶段的标志性变化

#### 1. 开始区分“做事能力”和“会话理解能力”

这一阶段的大量文档其实都在围绕同一件事打转：

系统已经会做很多事，但还不够会判断“用户现在到底在干什么”。

比如：

1. 用户是在改条件
2. 用户是在回答澄清
3. 用户是在问刚刚结果里的某个字段
4. 用户是在问系统能力
5. 用户是在闲聊

如果这层判断不稳，再多规则都会显得机械。

#### 2. 明确提出“会话语义层”

5 月 26 日的多份文档实际上逐步收敛出了一个非常重要的抽象：

不能只分：

1. `rate_query`
2. `result_analysis`
3. `unknown`

而应该在运价域内部进一步识别子类型，例如：

1. `new_quote`
2. `quote_update`
3. `clarify_reply`
4. `result_reference`
5. `result_expand`
6. `result_filter`
7. `service_info`
8. `capability_intent`

这意味着系统开始从“节点编排”往“会话语义建模”升级。

#### 3. 明确提出“输出策略层”

这一阶段另一个很关键的思想是：

系统的问题不仅是“判断错”，还有“输出形式太死”。

于是开始提出：

1. `summary_only`
2. `summary_plus_table`
3. `summary_plus_clarify`
4. `single_result`

这个抽象非常重要，因为它把“答什么”和“怎么答”拆开了。

#### 4. 明确提出“结果数量层”

从 `2026-05-27` 的文档可以看出，项目开始意识到：

用户说“给我一条”“推荐一条”“最便宜的一条”时，问题不只是排序，而是：

> 用户的结果数量目标没有被显式建模。

于是提出：

- `quantity_mode = single | multi`

这是一个很好的抽象，因为它把“单条 vs 多条”的问题从结果分析意图里拆了出来。

### 这一阶段暴露出的典型问题

#### 1. `你好` 会误触发询价

这个问题很有代表性：

有旧上下文时，系统把“你好”误判成城市 follow-up，重新进了询价链。

它说明一个本质问题：

> follow-up 规则已经开始互相挤压，系统缺少更高优先级的语义排除层。

#### 2. “一条数据”失效

用户明确说要一条，但系统还是输出多条。

这说明：

> 系统已经开始做结果分析，但还没有把“数量约束”真正打通到最终渲染层。

#### 3. 结果引用与结果分析边界不清

如：

- 最便宜那条的包装类型是什么

这类问题本质上应该是：

1. 先定位结果
2. 再解释字段

但系统还容易把它粗暴地打成 `lowest` 或 `summary`。

### 这一阶段最重要的认知产出

如果说前几个阶段沉淀的是“功能”，那这个阶段沉淀的是“方法论”。

这个阶段已经逐渐形成一个更成熟的判断：

1. 当前系统不是纯工作流
2. 但也还不是成熟 Agent
3. 更准确的定义是：

> 受控工作流型 Agent 雏形

这个定义非常有价值，因为它既不虚夸，也不贬低。

它准确描述了系统状态：

1. 已经有上下文
2. 已经有路由
3. 已经有多轮
4. 已经有结果分析
5. 但还缺“稳定的会话语义层和动作选择层”

### 我的复盘判断

这是当前项目最应该沉淀成个人资产的部分。

因为这一阶段已经不只是“修功能”，而是在形成一套很通用的 Agent 项目认知：

1. 为什么系统会显得像工作流
2. 为什么不能只继续堆规则
3. 为什么不能一下子改成自由 Agent
4. 为什么应该补：
   - 会话语义层
   - 输出策略层
   - 结果数量层
   - 统一澄清状态机

这部分以后无论是讲 AI Agent 架构、业务 Agent 产品化，还是做咨询陪跑，复用价值都很高。

## 四、这两个月项目演进的主线变化

如果把这次项目的变化压缩成几条主线，其实非常清楚：

### 1. 从“能查价”到“能多轮补参”

最开始的重点是：

1. 识别用户是不是在问报价
2. 抽取起运港、目的港、重量、体积、日期
3. 缺什么问什么
4. 参数齐了就查

这是基础 Agent 能力。

### 2. 从“多轮补参”到“会话边界控制”

系统开始面对：

1. 什么时候复用旧参数
2. 什么时候清空旧参数
3. 新航线短句怎么处理
4. 澄清状态怎么跨轮保留

这是会话系统能力。

### 3. 从“查完返回结果”到“围绕结果继续对话”

系统开始支持：

1. 最便宜
2. 直达/中转
3. 航司筛选
4. 包装/货类筛选
5. 结果摘要

这是结果分析能力。

### 4. 从“结果分析”到“结果引用与解释”

系统进一步暴露出一个更难的问题：

1. 这是多少号的
2. 这个是哪个航司
3. 最便宜那条的包装类型是什么

这是结果对象解释能力。

### 5. 从“功能堆叠”到“开始抽象方法论”

后期文档最有价值的地方在于，它已经不满足于继续补一个规则，而是开始抽象：

1. 会话语义层
2. 动作选择层
3. 输出策略层
4. 数量模式
5. 统一 pending action

这是从做项目走向沉淀方法论的信号。

## 五、当前项目阶段判断

基于现有材料，我对当前项目的阶段判断是：

### 不是

1. 不是一个简单的运价查询接口
2. 不是一个只有 Prompt 的 Demo
3. 不是一个成熟的开放式 Agent

### 更准确地说是

> 一个已经具备多轮状态管理、工具调用、结果分析、会话控制能力的受控工作流型 Agent 雏形。

它现在最强的地方是：

1. 工程结构清晰
2. 业务边界清晰
3. 主链稳定性优先
4. 已经开始沉淀中间层抽象

它现在最明显的短板是：

1. 会话语义层还不够稳
2. 澄清消费机制还不够统一
3. 结果引用/解释能力还在早期
4. 输出策略还没有完全从节点逻辑中抽离

## 六、这次项目对我最有价值的沉淀点

这部分不是仓库事实，而是我基于整个时间线对项目价值的复盘归纳。

### 1. 我学到的不是“怎么写一个聊天接口”

真正学到的是：

1. 怎么把一个业务问题抽成 Agent 状态机
2. 怎么让 LLM 处在受控的工程边界内工作
3. 怎么把工具调用、槽位状态、结果对象、前端协议拆开
4. 怎么在多轮对话中处理“会话边界”

### 2. 我开始真正碰到 Agent 项目里的中间层问题

比如：

1. 什么时候该澄清
2. 什么时候该继续执行
3. 什么时候该看作新任务
4. 什么时候用户是在问结果而不是在改条件

这些问题不是“模型好不好”能单独解决的，而是 Agent 设计问题。

### 3. 我开始意识到业务 Agent 的真正难点不在工具调用，而在状态与语义

这个项目一开始看起来像是“接个运价接口，再让模型抽参数”。

但往后做会发现，真正难的是：

1. 状态如何存
2. 状态何时清
3. 上下文何时继承
4. 用户一句短话到底依附于什么对象
5. 输出到底该是表格、总结，还是反问

### 4. 我开始形成一种更成熟的 Agent 认知

不是：

- 让模型更自由

而是：

- 在关键执行点保留护栏
- 在中间理解层逐步放开一点智能

也就是：

> 先理解，再执行；理解可以更灵活，执行必须受控。

## 七、后续可继续补写的内容

这份文档现在已经能作为第一版完整迭代记录，但后面还可以继续补两个方向：

### 1. 增加“版本视角”的里程碑表

例如：

1. `v0`：只支持单轮查价
2. `v1`：支持多轮补参
3. `v2`：支持结果分析
4. `v3`：支持澄清状态跨轮
5. `v4`：开始 Agent 化抽象

这样以后对外讲会更清楚。

### 2. 增加“问题-决策-结果”的索引页

例如：

1. 为什么要引入 `latest_quote_result`
2. 为什么要做 `reset_quote_context`
3. 为什么不能只靠 Prompt
4. 为什么要抽 `response_mode`
5. 为什么 `quantity_mode` 是必要的

这会更适合以后做咨询或陪跑时直接引用。
