# 服务器更新与手动同步清单

## 1. 结论先说

你现在的想法整体是对的，而且如果后续一直遵循
[本地迭代、GitHub与服务器协作说明.md](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\Study\本地迭代、GitHub与服务器协作说明.md)
里的边界，正常情况下大概率不会出现代码冲突。

核心原因很简单：

- 代码从本地提交到 GitHub
- 服务器只拉代码，不长期手改源码
- 生产专属文件不提交 GitHub
- Docker 只负责构建和运行
- 日志、知识库索引、缓存走宿主机持久化目录

只要你不在服务器上直接改 Git 跟踪的源码文件，这套模式是稳定的。

---

## 2. 你当前理解里哪些是对的

你提的这四步里，方向基本没问题：

1. `git pull` 项目
2. 把一些 `.env` 手动拖到服务器项目目录下
3. 启动 Docker，完成依赖环境和项目运行
4. 启动成功后访问测试

这里我只补两个关键修正：

### 2.1 不是每次都要手动拖 `.env`

更准确地说：

- 只有当生产环境变量变更时，才需要手动覆盖服务器上的 `deploy/.env.production`
- 如果这次只是业务代码改动，`.env.production` 没变，就不需要每次都重新拖

### 2.2 “RAG 重新构建”不应该默认每次都做

更合理的规则是：

- 只改业务代码、接口、Prompt、日志，不需要每次重建 RAG
- 只有 `data/docs` 资料变了，或者索引策略变了，才需要重建知识库

也就是说，`docker compose up -d --build` 和 `RAG 重建` 是两件不同的事，不能绑定成“每次部署都一起跑”。

## 3. 每次服务器更新建议做的事情

后续每次服务器部署，建议固定成下面这套顺序。

## 3.1 第一步：先确认本地已经完成这些事

本地先完成：

- 功能开发
- 本地验证
- 代码提交
- 推送 GitHub

例如：

```bash
git add .
git commit -m "feat: xxx"
git push origin main
```

---

## 3.2 第二步：SSH 登录服务器

通过 `MobaXterm` 登录服务器。

建议进入项目目录：

```bash
cd /data/apps/freight-agent
```

---

## 3.3 第三步：拉取 GitHub 最新代码

```bash
git pull origin main
```

如果你后续不是用 `main` 分支，就替换成实际分支名。

这一步只负责同步代码，不负责生产配置。

---

## 3.4 第四步：确认或覆盖服务器专属文件

不是每次都拖全部文件，而是只处理“这次确实变了”的生产专属文件。

优先检查：

- `deploy/.env.production`
- `deploy/nginx/default.conf`
- 证书文件
- 资料文件 `data/docs/`

如果这些没变，这一步可以跳过。

---

## 3.5 第五步：构建并启动 Docker

进入项目目录后执行：

```bash
docker compose up -d --build
```

这一步会做的事：

1. 重新读取 `docker-compose.yml`
2. 重新构建应用镜像
3. 安装项目依赖
4. 用新镜像替换旧容器
5. 启动 `freight-agent-app`
6. 启动 `freight-agent-nginx`

这一步不会主动帮你重建 RAG，除非你另外执行知识库初始化或重建命令。

---

## 3.6 第六步：检查容器状态

```bash
docker compose ps
```

重点确认：

- `freight-agent-app` 是 `Up`
- `freight-agent-nginx` 是 `Up`
- health check 正常

---

## 3.7 第七步：检查日志

先看 Docker 容器日志：

```bash
docker compose logs --tail=100 freight-agent-app
docker compose logs --tail=100 freight-agent-nginx
```

再看宿主机落盘日志：

```bash
ls -lah /data/logs/freight-agent
tail -n 100 /data/logs/freight-agent/freight-agent-app.log
```

---

## 3.8 第八步：做访问验证

先测本机健康检查：

```bash
curl http://127.0.0.1/health
```

再测公网访问：

```bash
curl http://你的域名/health
```

如果暂时还没绑定域名，也可以先测 IP：

```bash
curl http://47.101.141.117/health
```

## 4. 哪些文件不提交 GitHub，需要手动同步

这一部分最重要，因为它直接决定你后续会不会乱。

## 4.1 生产环境变量文件

这类文件不应该提交 GitHub：

- `deploy/.env.production`
- 其他生产专用 `.env` 文件

这些文件应该：

- 只保存在服务器
- 本地保留模板或示例
- 变更时手动上传或覆盖

---

## 4.2 证书与私钥文件

这类文件不应该提交 GitHub：

- `deploy/nginx/certs/`
- `*.pem`
- `*.key`

这些文件属于纯生产资产，只能放服务器。

---

## 4.3 运行日志

这类文件不应该提交 GitHub：

- `/data/logs/freight-agent/freight-agent-app.log`
- `/data/logs/freight-agent/freight-agent-app.jsonl`
- `/data/logs/freight-agent/freight-agent-error.log`

这些日志应该只保留在服务器或后续日志平台，不属于源码仓库。

---

## 4.4 RAG 运行产物

这类文件不应该提交 GitHub：

- `data/chroma/`
- `data/cache/`
- `data/exports/`

原因：

- 这是运行态生成物
- 不是源码
- 可能随版本与数据变化

---

## 4.5 Docker 本地临时覆盖文件

这类文件一般也不应该提交：

- `docker-compose.override.yml`

除非你明确决定把它作为正式部署文件的一部分。

---

## 4.6 服务器上可能手动维护的资料文件

这类情况要分两种：

### 情况 A：资料文件也走 Git 管理

例如 `data/docs` 下的正式知识库源文件，你决定跟着仓库走。

那就：

- 本地更新资料
- 提交 GitHub
- 服务器 `git pull`

### 情况 B：资料文件只在服务器更新

那就：

- 不放进 GitHub
- 由你手动上传到服务器
- 上传后手动重建知识库

如果你希望资料也纳入版本管理，我更推荐走情况 A。

---

## 5. 建议你手动同步的文件清单

以后每次部署，不是“把很多东西都拖上去”，而是只维护下面这些需要人工控制的文件。

建议长期手动同步或手动确认的内容：

1. `deploy/.env.production`
2. SSL 证书文件
3. Nginx 证书目录
4. 服务器专属域名配置
5. 如未纳入 Git 的知识库资料文件

不建议手动拖的内容：

1. `main.py`
2. `graph/`
3. `rag/`
4. `tools/`
5. `Dockerfile`
6. `docker-compose.yml`

这些都应该通过 `git pull` 同步，而不是人工拖文件覆盖。

## 6. Docker 在服务器启动项目时到底做了什么

这部分建议你后续心里要非常清楚。

当你执行：

```bash
docker compose up -d --build
```

实际发生的是：

### 6.1 Compose 读取编排文件

读取：

- `docker-compose.yml`
- `deploy/.env.production`

### 6.2 构建应用镜像

按 `Dockerfile`：

- 选择基础镜像
- 安装 Python 依赖
- 复制项目代码
- 生成应用镜像

### 6.3 启动应用容器

启动：

- `freight-agent-app`

应用容器里会运行：

- FastAPI
- LangGraph Agent
- RAG 模块
- 应用自己的日志系统

### 6.4 启动 Nginx 容器

启动：

- `freight-agent-nginx`

Nginx 负责：

- 对外监听 `80`
- 反向代理到 `freight-agent-app:8012`

### 6.5 连接挂载目录

当前最关键的挂载包括：

- `./data/docs:/app/data/docs`
- `./data/chroma:/app/data/chroma`
- `./data/cache:/app/data/cache`
- `./data/exports:/app/data/exports`
- `/data/logs/freight-agent:/app/runtime/logs`

这意味着：

- 容器删了，宿主机的数据还在
- 日志不会因为容器重建而丢失
- Chroma 索引不会因为应用重启而丢失

---

## 7. 日志会不会受 Docker 影响

会，但现在这类影响是正向可控的。

### 7.1 Docker 不会阻止宿主机保存日志

因为当前已经把：

```text
/data/logs/freight-agent:/app/runtime/logs
```

挂载进容器了。

同时生产环境变量当前已经配置为：

```text
APP_LOG_DIR=/app/runtime/logs
```

这表示：

- 应用在容器内写 `/app/runtime/logs`
- 实际会落到宿主机 `/data/logs/freight-agent`

所以按当前配置，Docker 不会把你的日志“锁死在容器里”。

### 7.2 以后日志仍然会按天滚动

项目日志系统本身已经实现按天滚动。

所以后续重点不是“Docker 会不会挡住日志”，而是：

- 宿主机挂载目录是否存在
- 容器内 `APP_LOG_DIR` 是否指向挂载路径

只要这两个条件成立，日志就会继续在 `/data/logs/freight-agent` 下按天滚动存储。

## 8. 什么时候需要重建 RAG

这一点建议你单独记住，不要和 Docker 启动混为一谈。

### 不需要重建 RAG 的情况

- 改聊天逻辑
- 改 Prompt
- 改日志
- 改 Docker 文件
- 改前端接口适配

### 需要重建 RAG 的情况

- `data/docs` 文档变了
- metadata 变了
- chunk 策略变了
- embedding 或索引策略变了

这时再执行类似：

```bash
python -m scripts.rebuild_kb
```

或者你后续如果改成容器内执行，再走容器命令。

## 9. 以后推荐的最小服务器更新流程

你后续完全可以把每次更新收敛成下面这套最小动作。

### 普通代码更新

```bash
cd /data/apps/freight-agent
git pull origin main
docker compose up -d --build
docker compose ps
curl http://127.0.0.1/health
```

### 生产配置变更

```bash
cd /data/apps/freight-agent
# 手动覆盖 deploy/.env.production
docker compose up -d --build
docker compose ps
```

### 资料文件更新并重建知识库

```bash
cd /data/apps/freight-agent
# 更新 data/docs
docker compose up -d --build
# 再执行知识库重建
```

## 10. 最终建议

你现在的思路整体是对的，而且足够实用：

1. 本地负责开发
2. GitHub 负责代码真源
3. 服务器负责生产配置和运行
4. Docker 负责构建与启动
5. 日志落到宿主机目录

我对你这套思路的补充只有三点：

1. 不要每次都机械地重新拖 `.env`，只有变了才覆盖
2. 不要把 “Docker 重建” 和 “RAG 重建” 绑死成同一个步骤
3. 不要在服务器直接改 Git 跟踪源码

只要守住这三条，后续基本不会乱。
