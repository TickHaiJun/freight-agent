# 阿里云云服务器采购与部署建议

## 文档信息
- 创建时间：2026-05-27
- 适用对象：当前 `freight-agent` 项目，以及后续计划继续部署的 4-5 个 AI 系统
- 目标：给出可执行的阿里云服务器采购建议、部署前准备清单、备案准备清单

---

## 这份文档解决什么
这份文档主要回答 4 个问题：

1. 当前这类 AI 系统，买什么样的阿里云服务器配置更合适
2. 如果后续要跑 5-6 个 AI 系统，什么配置更稳
3. 为什么推荐这些配置，而不是只看 CPU 核数随便买
4. 如果要部署到中国大陆 Linux 服务器并对外可访问，需要提前准备哪些内容和备案材料

---

## 摘要结论
如果按你现在这类系统的形态来看，我的建议很明确：

1. **当前不是 GPU 优先场景，先买高主频 CPU + 足够内存 + ESSD 磁盘**
   - 你现在这类系统主要是：
     - Python / FastAPI
     - SSE 长连接
     - 调外部大模型 API
     - 本地保存一些会话状态、向量索引、日志
   - 这类负载更吃：
     - CPU
     - 内存
     - 磁盘 IOPS
     - 网络稳定性
   - **不是先买 GPU**

2. **如果短期内要跑 5-6 个 AI 系统，单机建议至少从 `16 vCPU / 64GB` 起步**
   - `8 vCPU / 32GB` 可以做前期验证或 1-2 个系统
   - 但如果真要承载多个 AI Web/API 系统，`16C64G` 会稳很多

3. **当前代码结构更适合“单机强配置”先跑稳，再做多机**
   - 你现在的系统里，有一部分状态是进程内存态
   - 这意味着如果直接上多台 ECS 做负载均衡，后面会遇到：
     - 会话状态不一致
     - 上下文串话
     - 分布式状态同步问题
   - 所以现阶段更建议：
     - **先单机强配置**
     - 再逐步往多机高可用演进

4. **中国大陆正式对外访问，建议买大陆地域 ECS，并按企业主体备案**
   - 如果要正式对公网开放给客户或公司内部长期使用，备案基本是要做的
   - 如果只是临时测试、又想尽快上线，不备案的替代方案通常是中国香港地域，但大陆访问体验和后续规范性不如大陆地域

## 当前系统负载假设
以下建议建立在这个前提上：

### 当前系统特点
- Python + FastAPI + SSE
- 多轮会话
- 调外部 LLM API，不在本机跑大模型推理
- 本机需要承载：
  - Web/API 服务
  - Nginx
  - Python 运行时
  - 日志
  - 可能的向量库 / 文档索引 / 本地缓存
- 后续还会部署 4-5 个类似 AI 系统

### 这意味着什么
这类系统不是典型的“重 GPU 推理机”，而是典型的：

- 多个 Python 服务常驻
- 并发请求不一定特别大，但会有：
  - 较多网络等待
  - SSE 长连接
  - 内存占用叠加
  - 多项目共机资源争用

所以采购重点应放在：

- 足够的 CPU
- 足够的内存
- 较好的系统盘 / 数据盘性能
- 稳定公网出口

## 采购建议总原则

## 1. 实例类型优先级
优先考虑：

- **通用型实例**
- **企业级实例**

不建议一开始就买：

- 便宜但性能波动大的极低配突发型
- GPU 实例

原因：
- 你现在的系统主要是 API 编排与业务逻辑，不是本地大模型推理
- GPU 成本高，但当前收益不大

## 2. 内存比你想象的重要
后面 5-6 个 AI 系统一起跑时，真正容易先出问题的不是“CPU 跑满”，而是：

- 多个 Python 进程
- 各自依赖
- 向量索引
- 本地缓存
- 日志
- Nginx / 监控 / Docker

一起把内存吃掉。

所以：
- `32GB` 是前期可用
- `64GB` 是更稳的生产起点
- `128GB` 是多系统长期共机更舒服的档位

## 3. 磁盘不要只买系统盘
建议至少拆成：

- 系统盘
- 数据盘

数据盘建议放：
- 项目代码
- Docker 数据
- Chroma / 向量数据
- 日志
- 上传文件
- 备份

## 4. 带宽要按“公网访问体验”来选
如果只是后台服务调用，带宽压力不一定大。
但你这里会有：

- Web 页面访问
- SSE 实时输出
- 多个系统共用

所以不建议买太低的公网带宽。

## 三个推荐配置方案

## 方案 A：前期验证 / 成本优先
### 推荐用途
- 当前先部署这个 AI 运价系统
- 后续再加 1-2 个 AI 系统
- 团队先跑通、先对外试运行

### 推荐配置
- **实例规格**：`8 vCPU / 32GB`
- **实例族方向**：阿里云 ECS 通用型实例
- **系统盘**：ESSD 云盘 `200GB`
- **数据盘**：ESSD 云盘 `500GB`
- **公网带宽**：`10 Mbps`
- **操作系统**：`Ubuntu 22.04 LTS` 或 `Alibaba Cloud Linux 3`

### 为什么推荐
- 这个配置跑 1-2 个 Python AI 服务是够的
- 成本可控
- 适合作为第一台正式云主机

### 不足
- 如果后续很快涨到 5-6 个系统，会开始吃紧
- 内存余量不够从容
- 磁盘空间和 IOPS 也会比较早到瓶颈

### 结论
适合：
- **试运行**
- **前期上线**
- **预算紧**

不适合：
- 直接作为后续 5-6 个系统的长期主生产机

---

## 方案 B：推荐生产单机方案
### 推荐用途
- 未来 5-6 个 AI 系统主要先跑在一台机器上
- 先求稳定、先求省心
- 当前架构还不适合马上做多机会话一致性

### 推荐配置
- **实例规格**：`16 vCPU / 64GB`
- **实例族方向**：阿里云 ECS 通用型实例
- **系统盘**：ESSD 云盘 `300GB`
- **数据盘**：ESSD 云盘 `1TB`
- **公网带宽**：`20-30 Mbps`
- **操作系统**：`Ubuntu 22.04 LTS`

### 为什么推荐
这是我最推荐的主方案，原因有 5 个：

1. **匹配当前架构**
   - 当前系统还存在会话状态保存在进程内存里的情况
   - 单机方案最简单、最稳，不会先引入分布式问题

2. **内存足够健康**
   - 5-6 个 AI 系统一起跑，`64GB` 明显比 `32GB` 从容
   - 留得出：
     - Python 服务
     - Nginx
     - Docker
     - 向量索引
     - 日志
     - 监控
     - 系统缓存

3. **CPU 更合适**
   - 这类系统不是 GPU 型，而是多个 API 服务并发型
   - `16C` 比较适合多项目共机

4. **磁盘更稳**
   - `1TB` 数据盘后续放多个系统、日志、向量库更安心

5. **带宽更合理**
   - `20-30M` 对外访问体验明显比 `5-10M` 更稳

### 结论
如果你现在是认真要买一台、后面还要放 4-5 个 AI 系统，**优先推荐这个方案**。

---

## 方案 C：稳定生产 / 后续可扩展方案
### 推荐用途
- 你希望这台云上环境不仅跑 5-6 个 AI 系统
- 还希望后面更容易往高可用、多机扩展
- 愿意为稳定性多花一些预算

### 推荐配置

#### 当前可先买的单机规格
- **实例规格**：`32 vCPU / 128GB`
- **实例族方向**：阿里云 ECS 通用型实例
- **系统盘**：ESSD 云盘 `300GB`
- **数据盘**：ESSD 云盘 `2TB`
- **公网带宽**：`30-50 Mbps`
- **操作系统**：`Ubuntu 22.04 LTS`

#### 后续升级方向
后面再演进到：
- 2 台应用 ECS
- 1 个负载均衡
- 1 个 Redis / 共享状态层
- 日志、备份、监控分离

### 为什么推荐
这个方案的意义不是“现在一定要用满”，而是：

- 你买一次，短期不用太快再升配
- 后续多个系统进来后，资源余量更健康
- 即使某些 AI 系统后来增加本地向量库、任务调度、后台处理，也更扛得住

### 不足
- 成本更高
- 当前如果项目量还没真正起来，前期会有资源冗余

### 结论
适合：
- 你们已经确定会继续在这台机子上长期堆多个 AI 系统
- 更看重后续省心，而不是当前最低成本

## 我最终建议买哪个

## 最推荐：方案 B
如果你要一个最现实、最稳、最适合当前系统状态的答案：

### 建议采购
- `16 vCPU / 64GB`
- ESSD 系统盘 `300GB`
- ESSD 数据盘 `1TB`
- 公网带宽 `20-30Mbps`
- Ubuntu 22.04 LTS

### 推荐原因
因为它在这几个目标之间最平衡：

- 当前单机部署稳定
- 未来 5-6 个系统还能扛住
- 不会过早遇到内存瓶颈
- 不会因为上多机而马上踩会话一致性坑
- 预算通常也比方案 C 友好

## 什么情况下需要 GPU
当前不建议买 GPU。

只有在下面情况出现时，才考虑单独上 GPU 机：

1. 你们要在服务器本地跑开源大模型
2. 本地跑 embedding / reranker / OCR / 视觉模型
3. 不再主要依赖外部 API，而是改成本地推理

如果后面真的要走本地模型路线，建议是：

- **单独买 GPU 机**
- 不要把当前这台 API/业务编排服务器直接升级成 GPU 主机

因为两类机器的目标完全不同：

- 当前这台：业务编排、Web/API、状态、索引
- GPU 机：模型推理

## 地域建议

## 如果要备案并正式对外提供服务
建议选：

- 中国大陆地域

地域优先原则：

1. 用户主要在华东  
   - 选上海 / 杭州一带更合适

2. 用户主要在华北  
   - 选北京一带更合适

3. 如果公司总部、网络、运维都在华东  
   - 优先选华东地域，后续排障更方便

## 如果想临时免备案快速上线
替代方案通常是：

- 中国香港地域

但要明确：

- 不走大陆备案
- 大陆访问体验不一定比大陆地域稳定
- 后续正式对外时，还是大概率要回到大陆地域

所以如果你现在已经在做正式项目，我建议还是**直接走中国大陆地域 + 备案**。

## 部署前需要准备什么

## 一、域名
至少准备：

1. 一个正式域名
2. 域名已完成实名认证
3. 提前想好：
   - 主站域名
   - API 子域名
   - 未来多个系统是否按二级域名拆分

例如：
- `ai.xxx.com`
- `freight-ai.xxx.com`
- `rag.xxx.com`

## 二、服务器基础资源
至少准备：

1. ECS 实例
2. 安全组规则
3. 公网 IP / 弹性公网
4. 系统盘 + 数据盘
5. 快照策略

## 三、部署运行环境
建议准备：

1. `Python 3.11`
2. `Docker` / `Docker Compose`
3. `Nginx`
4. `systemd` 或 Docker 自启动
5. `Git`
6. `HTTPS 证书`

## 四、运维基础
建议至少有：

1. 日志目录规划
2. 定时备份
3. 云监控告警
4. 磁盘空间告警
5. CPU / 内存告警
6. 访问日志保留策略

## 当前项目部署时要特别注意的点

## 1. 当前更适合单机
这是最重要的工程边界。

你现在这类 AI 系统里，存在一些会话或运行态存在进程内存里的情况。  
这意味着：

- 单机部署：问题最少
- 多机负载均衡：后面要解决状态一致性

所以采购时不要幻想：
- 今天直接买两台挂负载均衡就万事大吉

如果代码没先改造，共机多实例会有稳定性风险。

## 2. Nginx 反向代理要做
建议结构：

- `Nginx :80/:443`
- 转发到各个本地 Python 服务端口

这样后面多个 AI 系统可以统一通过域名和路径管理。

## 3. 建议容器化
后面会有 5-6 个 AI 系统，建议尽量：

- 每个系统独立容器
- 独立环境变量
- 独立日志
- 独立重启

不建议所有系统直接手工跑在宿主机里。

## 备案前需要准备哪些东西

## 说明
如果你买的是中国大陆地域云服务器，并且要让网站/系统正式对外访问，一般要做 ICP 备案。  
企业场景通常按企业主体备案。

## 通常要准备的材料

### 1. 主体资料
- 企业营业执照
- 企业统一社会信用代码
- 企业通信地址

### 2. 负责人资料
- 法人身份证信息
- 网站负责人身份证信息
- 手机号
- 邮箱

有些公司会让：
- 法人做主体负责人
- 具体业务同事做网站负责人

### 3. 域名资料
- 域名实名认证已完成
- 域名持有者信息与备案主体信息一致或可对应

### 4. 服务器资料
- 阿里云中国大陆地域服务器
- 可用于备案的云资源
- 对应备案服务码或备案接入信息

### 5. 网站信息
- 网站名称
- 服务内容
- 域名
- 语言版本
- 网站用途说明

### 6. 特殊行业前置审批（如涉及）
如果内容属于特殊行业，可能需要前置审批。  
你们当前这种企业 AI 业务系统，一般先按普通企业网站/业务系统去看，但正式备案时仍要按当时页面内容判断。

---

## 备案前的实际准备建议
为了少返工，建议你们在备案前先准备这几项：

1. 确定备案主体  
   - 用公司主体，不要用个人主体

2. 确定主域名和网站名称  
   - 不要今天一个名字、明天一个名字

3. 确定系统对外描述  
   - 例如：
     - 智能运价查询系统
     - 物流业务 AI 助手

4. 确定页面内容不要出现敏感偏差  
   - 备案阶段尽量保持页面内容与申报用途一致

## 上云后建议的部署结构

## 推荐结构（当前阶段）
- 1 台 ECS
- 1 个 Nginx
- 多个 Docker 容器
- HTTPS
- 定时快照
- 云监控

### 示例
- `Nginx`
- `freight-agent`
- `ai-system-2`
- `ai-system-3`
- `ai-system-4`
- `ai-system-5`
- `ai-system-6`

### 推荐端口方式
对外：
- `80/443`

对内：
- 每个服务独立本地端口

例如：
- `127.0.0.1:8012`
- `127.0.0.1:8013`
- `127.0.0.1:8014`

Nginx 再按域名或路径转发。
