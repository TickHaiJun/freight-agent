## 4. 为什么我不推荐 5 Mbps

这个问题要拆开看，不然很容易误判。

### 4.1 阿里云文档里，5 Mbps 到底代表什么

阿里云官方文档明确说明，公网带宽控制的是公网出方向速率，单位通常是 `Mbps`。按照官方换算：

- `5 Mbps = 0.625 MB/s = 640 KB/s`

参考：

- 阿里云《网络带宽核心机制与计费详解》
  - https://help.aliyun.com/zh/ecs/user-guide/network-bandwidth/

这个换算很重要，因为很多人看到 `5 Mbps` 会误以为是 `5 MB/s`，实际不是。

### 4.2 对你这种 AI API 服务，带宽消耗来自哪里

你这个项目不是下载大文件型业务，它的公网出流量主要来自：

1. 浏览器或前端项目访问 `/api/chat`
2. SSE 持续向前端推文本
3. 健康检查、管理接口等少量返回

注意：

- `DeepSeek`、`DashScope`、运价接口调用，很多是你的服务器主动出站请求
- 但真正让公网“出网带宽”吃紧的，通常是你给外部客户端返回的内容

单看 `freight-agent` 这个服务本身，纯文本 SSE 并不算重带宽业务。

### 4.3 为什么我还是不建议你买 5 Mbps

不是因为单个 `freight-agent` 必然吃满 5 Mbps，而是因为你已经明确后续要跑 `4-5` 个 AI 项目。

到时候一台机器上很可能同时存在：

1. 多个前端或管理页面
2. 多个 API 服务
3. 多个 SSE 或流式输出服务
4. 可能还有文件上传、知识库调试、内部联调

`5 Mbps` 在单个轻服务阶段也许能跑，但对于“多项目共机”来说，余量太小。

它的风险不是“完全不可用”，而是：

1. 高峰时响应变慢。
2. 多个项目并发时相互抢公网出口。
3. 你排查问题时会分不清是代码问题还是带宽瓶颈。
4. 一旦后续加静态资源或文件下载，紧张会更明显。

### 4.4 我的带宽建议怎么选

如果你现在就要买，我建议这样选：

#### 方案 1: 成本和稳定性平衡

- `10 Mbps`

适合：

- 当前以 API 为主
- 浏览器前端请求量中等
- 暂时没有大文件下载

#### 方案 2: 我更推荐的起步值

- `20 Mbps`

适合：

- 明确会跑 `4-5` 个 AI 项目
- 有多个前端项目同时联调
- 希望减少“网络余量不够”的概率

#### 方案 3: 偏稳妥的单机生产值

- `30 Mbps`

适合：

- 这台机器未来承担统一 AI 接入层角色
- 还会接更多网页、管理后台、静态内容、下载流量
- 你想尽量晚一点再做带宽升级

### 4.5 固定带宽还是按流量

阿里云官方文档说明，固定公网 IP 带宽支持：

- 按固定带宽计费
- 按使用流量计费

参考：

- 阿里云《ECS 公网带宽计费》
  - https://help.aliyun.com/zh/ecs/public-bandwidth/
- 阿里云《修改固定公网带宽》
  - https://help.aliyun.com/zh/ecs/user-guide/modify-the-bandwidth-configurations

你的场景我更建议：

1. 如果公网访问模式比较稳定，先用 `按固定带宽`。
2. 带宽值先从 `10 Mbps` 或 `20 Mbps` 起。
3. 观察一段时间后，再决定是否调整。

原因很简单：

- 你现在更需要稳定的上限和可预期的访问体验
- 而不是一开始就在计费模式上做复杂优化

## 5. CPU / 内存 / 磁盘，我建议怎么升级

### 5.1 CPU 与内存

你当前打算买的 `8核32G`，对于单项目是合理的，但对于你已经明确的“后面会跑 4-5 个 AI 项目”，我建议一步到位上：

- `16核64G`

原因不是因为单个模型推理重，而是因为你未来会同时承载：

1. 多个 Python 服务
2. 多个虚拟环境或容器
3. Nginx
4. 日志
5. RAG 索引
6. 缓存和后台任务

你现在这些 AI 项目大概率都属于“编排型 AI 服务”，也就是：

- API 网关
- Prompt 编排
- 向量检索
- 外部模型调用

这种系统不一定最吃 GPU，但很吃“多服务并存时的 CPU/内存余量”。

### 5.2 系统盘

系统盘 `100G` 不是不能用，但我建议你至少重新评估一下：

- 如果你后续会装 Docker、构建缓存、多个 Python 环境
- 如果你还会在系统盘上临时存更新包、备份包

那 `200G` 会更省心。

如果你确定严格执行“代码、日志、知识库、备份都放数据盘”，那：

- `100G 系统盘` 也能接受

我的建议是：

- 保守值：`100G`
- 更省心：`200G`

### 5.3 数据盘

如果你真的是为多项目准备，不建议停留在 `500G`。

更推荐：

- `1TB ESSD`

原因：

1. 每个 AI 项目都可能有自己的日志目录
2. 未来多个 RAG 项目会有自己的文档、向量库、缓存
3. 数据盘除了存代码，还会存备份、临时文件、上传文件

`500G` 对单项目很够，但对多项目统一应用机来说，偏紧。

### 5.4 ESSD 还要注意什么

阿里云官方文档说明，ESSD 的性能和：

1. 云盘类型/级别有关
2. 容量有关
3. 实例规格上限也有关

参考：

- 阿里云《ESSD 云盘》
  - https://help.aliyun.com/zh/ecs/user-guide/essds
- 阿里云《块存储性能规格》
  - https://help.aliyun.com/zh/ecs/user-guide/block-storage-performance

对你这种应用来说，不需要一开始追求很高的 PL 档位。第一阶段重点不是极限 IOPS，而是：

1. 容量够
2. 数据盘单独挂载
3. 代码和日志分层清楚

如果没有特殊高吞吐场景，普通 ESSD 就够用了。

## 7. 目录规划到底是什么，为什么要规划

目录规划，本质上是给服务器上的文件“分区分层”。

不是为了好看，而是为了下面这些现实问题：

1. 出问题时能快速定位。
2. 多项目共机时不互相污染。
3. 备份时知道该备份什么。
4. 清理磁盘时不容易误删。
5. 日志、缓存、代码、上传文件职责清晰。

如果你不做目录规划，最容易出现的情况是：

1. 代码随便放在某个用户目录
2. 日志散落在系统目录、nohup 输出、终端历史里
3. 数据盘和系统盘混着写
4. 后面一多项目就乱

## 9. 每个目录是干什么的

### 9.1 `/data/apps`

这是放项目代码的地方。

例如：

```text
/data/apps/freight-agent
```

里面放：

- 项目代码
- `.venv`
- `.env`
- `data/chroma`
- `data/cache`
- `data/docs`

你可以把它理解成“项目自己的家目录”。

### 9.2 `/data/logs`

这是放运行日志的地方。

例如：

```text
/data/logs/freight-agent
```

里面放：

- 应用日志
- 错误日志
- 按天切分的运行日志

为什么单独拿出来：

1. 日志增长快
2. 日志最适合统一清理和压缩
3. 多项目共机时日志必须独立

### 9.3 `/data/backups`

这是放备份的地方。

例如：

- 每次发版前打包
- 知识库索引快照
- 配置文件备份

它的意义是：不要把备份和生产代码放在同一个目录里。

### 9.4 `/data/runtime`

这是放运行时文件的地方。

通常会包括：

- PID 文件
- UNIX socket
- 临时状态文件

如果未来你某些服务改成 `gunicorn + unix socket`，这个目录会用得上。

### 9.5 `/data/tmp`

这是临时文件区。

例如：

- 上传文件中转
- 解析文档的临时缓存
- 临时压缩包

临时文件不要随便写到项目代码目录里。

### 9.6 `/data/shared`

这是多个项目共用的资源目录。

例如：

- 公共脚本
- 公共证书
- 公共部署包

它的目的是让“项目私有资源”和“公共资源”分开。

## 11. 日志能不能按天自动落盘

可以，完全可以。

而且这件事我建议你不要靠“终端重定向”去碰运气，应该正式设计。

### 11.1 你现在要的效果是什么

你要的其实是下面这件事：

1. 运行时控制台输出照常有
2. 同样的内容自动写入文件
3. 每天凌晨切一份新日志
4. 每天一整天的日志归到一个文件里
5. 文件落到：

```text
/data/logs/freight-agent
```

6. 文件命名类似：

```text
2026-06-02_00-00-00.log
2026-06-03_00-00-00.log
```

这其实是“控制台日志 + 文件日志 + 按天轮转”的组合。

### 11.2 推荐方案: 应用层直接做双通道日志

我更推荐的方案不是只靠 `systemd/journalctl`，而是：

1. 控制台继续输出
2. Python `logging` 再增加一个 `TimedRotatingFileHandler`

这样有几个好处：

1. 终端里看到什么，文件里就能基本保留什么
2. 日志目录和命名完全可控
3. 不依赖你后面再把 `journalctl` 导出来
4. 多项目时最清晰

### 11.3 为什么不推荐只靠 `nohup > xxx.log`

因为它的问题很明显：

1. 不会自动按天切
2. 重启后文件容易混乱
3. 很难保留规范命名
4. 错误日志和标准日志不好分

### 11.4 为什么不把“每天日志文件”完全交给 Nginx

因为你要的是应用运行日志，不是访问日志。

Nginx 只能很好地记录：

- 请求访问日志
- 反代错误日志

但它不能替代 Python 应用自身的业务日志。

## 12. 我建议的日志方案长什么样

### 12.1 应用日志

放在：

```text
/data/logs/freight-agent/
```

命名：

```text
2026-06-02_00-00-00.log
2026-06-03_00-00-00.log
```

建议规则：

1. 每天 `00:00:00` 切分
2. 保留最近 `30~90` 天
3. 超过保留期自动清理或压缩

### 12.2 Nginx 访问日志

放在：

```text
/data/logs/nginx/
```

记录：

- 哪些 IP 来访问
- 哪些路径被调用
- 状态码、耗时等

### 12.3 错误日志

可以单独留一个：

```text
/data/logs/freight-agent/error.log
```

它适合快速看严重异常，不用翻整天业务日志。

## 13. 这件事和当前代码的关系

这里要讲清楚边界，避免误解。

当前项目代码里，`main.py` 现在只是：

- 用 `logging.basicConfig(...)` 把日志输出到控制台

也就是说：

1. 当前代码并没有实现“按天写文件”
2. 当前也没有实现“详细时间命名日志文件”
3. 这需要在代码层增加文件 handler，或者由外部日志系统接管

如果你问“能不能做到”：

- 能

如果你问“现在项目是不是已经做了”：

- 不是

### 13.1 最推荐的落地方式

后续真正落地时，建议对项目做一处小改动：

1. 保留控制台 handler
2. 再加文件 handler
3. 文件 handler 用按天轮转

这是最稳、最清晰的方案。

### 13.2 如果你后面要我帮你改代码

那我会优先这样改：

1. 在项目里新增统一日志初始化模块
2. 从 `.env` 或配置文件读取日志目录
3. 输出到 `/data/logs/freight-agent`
4. 用 `TimedRotatingFileHandler` 实现每天一份

这会比继续堆在 `main.py` 的 `basicConfig` 里更干净。

## 14. 前端项目访问这台服务器接口，应该怎么理解

你这次边界里最关键的一句是：

- 公司的另一个服务器前端项目访问这个服务器接口

这句话在工程上要拆成两层。

### 14.1 如果是浏览器直接请求

假设前端页面域名是：

- `https://portal.company.com`

AI 接口域名是：

- `https://api.company.com`

那就是浏览器跨域访问。

这种场景你要关注：

1. CORS
2. HTTPS
3. SSE 在浏览器端能否稳定消费

### 14.2 如果是前端所在服务器先做代理

如果门户服务器先把请求代理到 AI 服务器，再由浏览器请求门户自己的路径，那跨域问题基本就绕开了。

这仍然是我更推荐的接入方式。

### 14.3 当前项目该怎么收口

你前面那份部署手册已经讲过：当前 `allow_origins=["*"]` 能先跑通。

这份补充只补一句最实际的建议：

如果未来是公司前端正式接入，建议上线前把 CORS 收敛成实际域名，不要长期保持 `*`。
