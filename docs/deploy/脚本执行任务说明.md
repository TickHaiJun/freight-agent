# 脚本执行任务说明

本文说明脚本：

[bootstrap_aliyun_docker_host.sh](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\docs\deploy\bootstrap_aliyun_docker_host.sh)

在服务器上实际会做哪些事情，以及在什么前提下你可以直接在 `MobaXterm` 里执行它。

---

## 1. 现在可以直接跑吗

可以，但有前提。

你现在已经确认：

1. 服务器基础环境已经准备好了
2. 本地项目的 Docker 部署文件已经补齐

所以这个脚本现在不是单纯“初始化宿主机”了，而是：

- 宿主机环境校验 + 补齐
- 项目部署文件校验
- 直接执行 `docker compose up -d --build`

也就是说，当前版本的脚本已经接近“部署脚本”，不只是“环境准备脚本”。

---

## 2. 在 MobaXterm 里运行它的前提条件

你要先满足这几个前提，再执行。

### 2.1 已经能 SSH 登录服务器

也就是你已经用 MobaXterm 成功连上服务器。

### 2.2 `/data` 已挂载

脚本会检查 `/data` 是否存在。  
如果你还没挂载数据盘，脚本会直接退出。

### 2.3 项目代码已经上传到服务器

默认脚本要求项目路径是：

```text
/data/apps/freight-agent
```

这里至少要有：

1. `Dockerfile`
2. `.dockerignore`
3. `docker-compose.yml`
4. `deploy/nginx/default.conf`
5. `deploy/.env.production`
6. `requirements.txt`
7. `main.py`

如果这些文件不在服务器上，脚本会直接报错退出。

### 2.4 `deploy/.env.production` 不能再是占位值

脚本会主动检查：

```text
fill_me
```

如果你还没把 `.env.production` 里的占位值改成真实值，脚本会直接退出，不会继续启动容器。

这一步是故意加的保护，避免你一键跑完但服务实际上不可用。

---

## 3. 脚本会执行哪些任务

当前版本的脚本会做下面这些事。

### 任务 1：打印当前部署目标信息

会输出：

1. 公网 IP
2. 部署用户
3. 项目根目录
4. Docker 部署目录

目的是让你执行脚本时先确认变量没有跑偏。

### 任务 2：检查 `/data` 是否存在

如果 `/data` 不存在，脚本会直接失败退出。

因为整个部署方案都建立在数据盘已经挂载到 `/data` 的前提下。

### 任务 3：更新系统软件包

执行：

```bash
apt update
apt upgrade -y
```

### 任务 4：安装基础工具

会安装：

1. `curl`
2. `git`
3. `vim`
4. `htop`
5. `tree`
6. Docker 依赖项

### 任务 5：配置 Docker 官方仓库

如果 Docker 官方仓库还没加，脚本会自动补上。

### 任务 6：安装 Docker 与 Compose

会安装：

1. `docker-ce`
2. `docker-ce-cli`
3. `containerd.io`
4. `docker-buildx-plugin`
5. `docker-compose-plugin`

### 任务 7：启动并设置 Docker 开机自启

执行：

```bash
systemctl enable docker
systemctl start docker
```

### 任务 8：创建标准目录

会确保这些目录存在：

```text
/data/apps/freight-agent
/data/logs/freight-agent
/data/backups/freight-agent
/data/docker/freight-agent
/data/tmp
```

### 任务 9：创建项目运行数据目录

会确保这些目录存在：

```text
/data/apps/freight-agent/data/docs
/data/apps/freight-agent/data/chroma
/data/apps/freight-agent/data/cache
/data/apps/freight-agent/data/exports
/data/apps/freight-agent/deploy/nginx
```

### 任务 10：校验部署文件是否齐全

它会检查下面这些文件是否存在：

1. `Dockerfile`
2. `.dockerignore`
3. `docker-compose.yml`
4. `deploy/.env.production`
5. `deploy/nginx/default.conf`
6. `requirements.txt`
7. `main.py`

只要缺一个，脚本就会退出。

### 任务 11：检查生产环境变量是否还在占位

如果 `.env.production` 里还有：

```text
fill_me
```

脚本会退出，不会执行 Compose 启动。

### 任务 12：校验 Docker Compose 配置

执行：

```bash
docker compose config
```

如果 Compose 配置有语法问题，这一步就会报错退出。

### 任务 13：执行正式启动

执行：

```bash
docker compose up -d --build
```

这一步会：

1. 构建应用镜像
2. 启动应用容器
3. 启动 Nginx 容器

### 任务 14：输出容器状态和最近日志

执行：

```bash
docker compose ps
docker compose logs --tail=80
```

### 任务 15：本机健康检查

执行：

```bash
curl http://127.0.0.1/health
```

这里验证的是：

- Nginx 是否正常
- Nginx 到应用容器的反代是否正常

### 任务 16：查看宿主机日志目录

执行：

```bash
ls -lah /data/logs/freight-agent
```

便于你确认应用日志是否开始落盘。

---

## 4. 脚本不会做什么

为了安全和可控，当前脚本不会做这些事情。

### 4.1 不会自动 SSH 登录

你必须先用 MobaXterm 登录成功，再手动执行脚本。

### 4.2 不会自动输入密码

包括：

1. SSH 密码
2. Git 私有仓库密码
3. `.env.production` 密钥

### 4.3 不会自动格式化数据盘

因为这是高风险操作，不能在部署脚本里自动做。

### 4.4 不会自动帮你上传项目代码

你必须先把项目代码同步到服务器。

### 4.5 不会自动帮你填真实环境变量

你需要自己编辑：

```text
deploy/.env.production
```

---

## 5. 正确执行顺序

你现在应该按这个顺序来。

1. 用 MobaXterm SSH 登录服务器
2. 确认 `/data` 已挂载
3. 把本地项目完整上传到：

```text
/data/apps/freight-agent
```

4. 编辑：

```text
/data/apps/freight-agent/deploy/.env.production
```

把里面的 `fill_me` 全部替换成真实值

5. 把脚本上传到服务器

6. 执行：

```bash
bash /path/to/bootstrap_aliyun_docker_host.sh
```

7. 然后检查：

```bash
docker compose ps
curl http://127.0.0.1/health
curl http://47.101.141.117/health
```

---

## 6. 是否可行

结论是：

可以这样跑，而且这比你手工一条条敲命令更稳。

原因是：

1. 它会先校验前置条件
2. 它会在发现 `.env.production` 还是占位值时主动停下
3. 它会在正式启动前先做 Compose 配置校验
4. 它跑完后还会帮你直接看容器状态和健康检查

但它可行的前提不是“脚本万能”，而是：

- 代码已经上传到服务器
- `.env.production` 已填好
- 数据盘已挂载

满足这三个前提，它就适合在 MobaXterm 里执行。
