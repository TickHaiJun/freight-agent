# AI Freight Agent 阿里云 Ubuntu 22.04 手动部署指南

## 1. 文档目标

这份文档面向你当前已经购买好的阿里云 ECS：

- `16 vCPU / 64 GB`
- `系统盘 100G ESSD`
- `数据盘 1TB ESSD`
- `公网带宽 15 Mbps`
- `Ubuntu 22.04`

目标是从 0 到 1，把当前 `freight-agent` 项目手动部署到云服务器，并满足下面几件事：

1. 你能通过 SSH 连上服务器
2. 服务器目录结构合理
3. 项目代码、日志、知识库索引有明确落点
4. 服务能通过 `Nginx` 对外访问
5. 后续便于维护、扩容、加域名、加 HTTPS、加自动化部署

这份文档按你刚确认的 3 个边界来写：

1. `FREIGHT_API_BASE` 云服务器可达
2. 外网不放 `8012`，统一走 `Nginx`
3. 当前阶段先用 SSH 登录和手动部署

默认方案如下：

- 先不用 Docker
- 先不用 CI/CD
- 先走手动部署
- `uvicorn` 只监听 `127.0.0.1:8012`
- 外部访问统一通过 `Nginx -> 127.0.0.1:8012`

---

## 2. 先给结论

如果让我给你一条最稳的落地路线，我建议这样做：

1. 先在阿里云控制台放通安全组端口
2. 用 Windows 自带 `ssh` 或 `MobaXterm` 连上 Ubuntu
3. 格式化并挂载 `1TB` 数据盘到 `/data`
4. 在 `/data/apps/freight-agent` 放项目代码
5. 在项目目录创建 Python 虚拟环境
6. 配置 `.env`
7. 重建知识库
8. 先用 `uvicorn` 本机监听做联通验证
9. 再用 `systemd` 托管应用
10. 用 `Nginx` 反向代理到 `127.0.0.1:8012`
11. 最后通过公网 `80/443` 访问

第一轮成功的目标形态是：

```text
Windows 本地
  -> SSH 连接 Ubuntu
  -> 项目放 /data/apps/freight-agent
  -> systemd 托管 uvicorn
  -> Nginx 反代到 127.0.0.1:8012
  -> 外网访问 80/443
```

## 4. 系统盘和数据盘怎么分

这是你这台机器部署里非常关键的设计点。

### 4.1 系统盘做什么

系统盘 `100G ESSD` 建议只承担这些职责：

1. Ubuntu 系统本身
2. `apt` 安装的软件
3. 系统日志
4. SSH、systemd、Nginx 这类系统级服务
5. 少量临时文件

不要把项目代码、知识库索引、应用日志长期堆在系统盘。

### 4.2 数据盘做什么

数据盘 `1TB ESSD` 建议承担这些职责：

1. 项目代码
2. Python 虚拟环境
3. 应用日志
4. 知识库源文件
5. Chroma 索引
6. BM25 缓存
7. 备份文件
8. 后续多个 AI 项目的统一落点

### 4.3 为什么这样分

这样做的好处是：

1. 系统盘更干净
2. 应用迁移和备份更清晰
3. 后续多项目共机时更不容易乱
4. 磁盘扩容和日志治理更容易
5. 即使以后重装系统，数据盘也更容易保留

## 7. 第一步：阿里云控制台基础检查

你现在机器已经买好了，先不要急着 SSH，先在阿里云控制台确认这几件事。

### 7.1 记录基础信息

记下：

1. 公网 IP
2. 实例名称
3. 实例登录用户名
4. 登录密码
5. 安全组 ID

### 7.2 安全组放行端口

当前方案下，建议至少放通这些端口：

1. `22/TCP`
2. `80/TCP`
3. `443/TCP`

含义：

- `22` 用于 SSH
- `80` 用于 HTTP
- `443` 用于 HTTPS

这里明确说明：

- 不建议在安全组里开放 `8012`
- `8012` 只作为服务器本机应用监听端口
- 外部用户不应该直接访问 `8012`

### 7.3 登录方式建议

当前阶段你已经确认先用 SSH 登录。  
第一次可以先用密码登录，正式稳定后再切 SSH 密钥登录。

## 8. 第二步：从 Windows 连接 Ubuntu

### 8.1 用 Windows Terminal 连接

在本地 PowerShell 里执行：

```powershell
ssh root@你的公网IP
```

如果阿里云默认不是 `root`，而是普通用户，例如 `ubuntu`，就执行：

```powershell
ssh ubuntu@你的公网IP
```

首次连接会提示确认指纹，输入：

```text
yes
```

然后输入密码。

### 8.2 如果连接失败，排查顺序

按这个顺序查：

1. 安全组是否放行 `22`
2. 公网 IP 是否正确
3. 实例是否处于运行中
4. 用户名是否正确
5. 本地网络是否拦截 SSH

## 9. 第三步：登录后先做基础初始化

连接上 Ubuntu 之后，先执行：

```bash
whoami
pwd
uname -a
lsblk
df -h
```

你要先确认两件事：

1. 你当前是不是有 `sudo` 权限
2. 数据盘有没有被系统识别

然后更新系统：

```bash
sudo apt update
sudo apt upgrade -y
sudo apt install -y curl wget git unzip vim htop tree build-essential nginx
```

这里顺手把 `nginx` 装上，因为当前部署方案就是要由它对外代理。

## 10. 第四步：格式化并挂载 1TB 数据盘

这一步非常关键，因为你后续的项目、日志、知识库索引都建议落在数据盘。

### 10.1 查看磁盘

执行：

```bash
lsblk
sudo fdisk -l
```

通常你会看到类似：

```text
/dev/vda   系统盘
/dev/vdb   数据盘
```

实际设备名以你机器输出为准，不要死记 `vdb`。

### 10.2 如果数据盘是全新空盘

假设数据盘是 `/dev/vdb`，可以这样格式化为 `ext4`：

```bash
sudo mkfs.ext4 /dev/vdb
```

注意：  
这条命令会清空该盘数据。  
如果这块盘上已经有数据，不能直接执行。

### 10.3 挂载到 /data

创建挂载点：

```bash
sudo mkdir -p /data
```

先临时挂载：

```bash
sudo mount /dev/vdb /data
```

验证：

```bash
df -h
ls /data
```

### 10.4 配置开机自动挂载

先查 UUID：

```bash
sudo blkid /dev/vdb
```

复制 UUID，然后编辑：

```bash
sudo vim /etc/fstab
```

追加一行：

```text
UUID=你的UUID /data ext4 defaults,nofail 0 2
```

保存后验证：

```bash
sudo mount -a
df -h
```

如果 `mount -a` 没报错，说明自动挂载配置基本正确。

## 16. 第十步：先前台启动验证

先不要一上来就配 systemd。  
先用最直接的方式确认项目能不能跑起来。

```bash
cd /data/apps/freight-agent
source .venv/bin/activate
uvicorn main:app --host 127.0.0.1 --port 8012
```

这里明确说明：

- 只监听 `127.0.0.1`
- 不监听 `0.0.0.0`
- 因为应用只给本机 `Nginx` 转发，不直接暴露给外网

然后在服务器本机另开一个终端，执行：

```bash
curl http://127.0.0.1:8012/health
```

预期返回：

```json
{"status":"ok"}
```

这一步只验证“应用本机可用”，不验证外网直连。

---

## 17. 第十一步：用 systemd 托管应用

前台验证没问题后，再做正式托管。

创建 service 文件：

```bash
sudo vim /etc/systemd/system/freight-agent.service
```

写入：

```ini
[Unit]
Description=AI Freight Agent
After=network.target

[Service]
User=ubuntu
Group=ubuntu
WorkingDirectory=/data/apps/freight-agent
Environment="PYTHONUNBUFFERED=1"
ExecStart=/data/apps/freight-agent/.venv/bin/uvicorn main:app --host 127.0.0.1 --port 8012
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

如果你的用户不是 `ubuntu`，记得改成真实用户名。

然后执行：

```bash
sudo systemctl daemon-reload
sudo systemctl enable freight-agent
sudo systemctl start freight-agent
```

查看状态：

```bash
sudo systemctl status freight-agent
```

查看最近日志：

```bash
journalctl -u freight-agent -n 100 --no-pager
```

## 18. 第十二步：配置 Nginx 反向代理

既然你明确不希望开放 `8012`，那正式对外访问必须走 Nginx。

### 18.1 创建站点配置

执行：

```bash
sudo vim /etc/nginx/sites-available/freight-agent
```

写入：

```nginx
server {
    listen 80;
    server_name _;

    client_max_body_size 20m;

    location / {
        proxy_pass http://127.0.0.1:8012;
        proxy_http_version 1.1;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_set_header Connection "";
        proxy_buffering off;
        proxy_cache off;
        chunked_transfer_encoding off;
        proxy_read_timeout 3600;
        proxy_send_timeout 3600;
    }
}
```

这里把整站都反代给应用，适合当前单服务形态。

### 18.2 启用配置

执行：

```bash
sudo ln -s /etc/nginx/sites-available/freight-agent /etc/nginx/sites-enabled/freight-agent
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t
sudo systemctl restart nginx
sudo systemctl enable nginx
```

如果 `nginx -t` 返回成功，说明配置基本正确。

## 22. 第十六步：HTTPS 什么时候再上

当前阶段可以先只开：

- `22`
- `80`

确认整条链路稳定后，再考虑：

- 域名解析
- SSL 证书
- `443` HTTPS
- Nginx HTTPS 配置

也就是说，你第一次部署不需要把 HTTPS 一次做满。

## 25. 当前阶段我对你的部署建议

基于你现在的情况，我的判断是：

1. 你现在最适合手动部署，不适合一上来 Docker 化
2. 你这次坚持不开放 `8012` 是对的，应用层应该藏在 Nginx 后面
3. 先把数据盘规划好，比先折腾自动化更重要
4. 第一轮成功标准不是“架构优雅”，而是“服务能稳定跑、Nginx 能代理、日志能落盘、报价链路能联通”

## 26. 还需要你后续重点确认的 3 件事

虽然这份文档可以直接执行，但真正上线前，你还需要确认：

1. 是否要尽快绑定域名
2. 是否要尽快上 HTTPS
3. 是否要把 SSH 改成密钥登录并收紧 root 登录策略

这三件事决定的是“上线质量”和“安全水平”，不是“能不能启动”。
