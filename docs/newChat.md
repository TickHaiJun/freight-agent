# AI 运价前后端对接说明（chat）

## 文档信息

- 文件名：`docs/chat.md`
- 创建时间：`2026-05-13 Asia/Shanghai`
- 文档定位：前后端交接流程说明
- 使用场景：前端项目接入当前 AI 运价系统的新一轮后端能力

---

## 1. 这份文档要解决什么问题

这份文档不是后端设计文档，也不是前端页面设计文档。  
它的目标只有一个：

**把前端和当前后端新能力之间的交接流程写清楚。**

本轮新增交接重点主要有：

1. 前端支持“中断请求”按钮
2. 前端支持“结束本次对话”按钮
3. 前端在原有请求参数基础上，新增 `reset_quote_context: true`
4. 前端继续使用同一个 `/api/chat` 接口，不改 SSE 协议
5. 前端继续按 Markdown 渲染 AI 回复内容

---

## 2. 摘要结论

前端这次接入后端时，最重要的变化只有两点：

### 变化一：新增“中断请求”按钮

用户在 AI 还没回复完成时，可以手动停止本轮请求。

前端动作：

- 中断当前请求
- 停止流式渲染
- 当前消息标记为已停止

后端动作：

- 感知连接断开
- 尽快停止无意义资源消耗

### 变化二：新增“结束本次对话”按钮

这个按钮不是退出聊天，而是：

- 结束当前这轮运价分析会话
- 保留聊天记录
- 清空当前活跃询价上下文
- 后续新问题进入新一轮查询

前端在点击这个按钮后，下一次发请求时，需要在原有请求参数基础上新增：

```json
"reset_quote_context": true
```

---

## 3. 当前接口不变的部分

本轮对接里，下面这些保持不变：

- 接口仍然是 `POST /api/chat`
- 响应仍然是 `text/event-stream`
- SSE 事件类型仍然是：
  - `text`
  - `context`
  - `done`
  - `error`
- 前端仍然要缓存并回传 `context`
- AI 返回内容仍然按 Markdown 渲染

也就是说：

**这次不是改协议，而是在原协议上增加一个新的控制参数和两个前端按钮。**

---

## 4. 最新请求参数格式

前端后续调用 `/api/chat` 时，请求体格式建议统一为：

```json
{
  "session_id": "web_xxx",
  "message": "用户输入内容",
  "context": {
    "sfg": "pvg",
    "mdg": "ams",
    "inputWeight": 500,
    "inputVol": 1,
    "hbrq": "2026-05-08",
    "hbrqBegin": null,
    "hbrqEnd": null,
    "flightType": null,
    "packageType": null,
    "cargoType": null,
    "twoCode": null,
    "gid": null,
    "query_completed": true
  },
  "reset_quote_context": false
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `session_id` | `string` | 是 | 当前聊天会话唯一标识 |
| `message` | `string` | 是 | 用户本轮输入 |
| `context` | `object \| null` | 否 | 上一轮返回的上下文 |
| `reset_quote_context` | `boolean` | 否 | 是否结束当前运价分析会话并重置本轮询价上下文 |

默认规则：

- 正常对话时：`reset_quote_context = false`
- 用户点击“结束本次对话”后，下一次发请求时：`reset_quote_context = true`

---

## 5. 新增按钮一：中断请求

## 5.1 按钮出现时机

这个按钮只在 AI 正在回复中时出现。

也就是：

- 已发请求
- 正在流式接收 `text`
- 还没收到 `done`

建议按钮文案：

- `中断请求`

或者：

- `停止回答`

---

## 5.2 点击后的前端行为

用户点击“中断请求”后，前端应立即执行：

1. 中断当前请求
2. 停止继续读取 SSE
3. 停止继续渲染 AI 文本
4. 当前 assistant 消息状态改为：
   - `已中断`
   - 或 `已停止`
5. 不更新本轮尚未完整结束时的 `context`

---

## 5.3 对前端状态的影响

点击“中断请求”后：

### 保留

- 用户已输入消息
- AI 已经返回的那部分文本
- 当前聊天记录

### 不应继续更新

- 当前请求未完成前的剩余文本
- 本轮最终完成态
- 本轮未确认完成的 `context`

---

## 5.4 重要说明

“中断请求”按钮的语义是：

- 停止这一次还在进行中的请求

它不是：

- 清空当前会话
- 开始新一轮查询
- 删除历史消息

所以它和“结束本次对话”是两个不同动作，前端不能混用。

---

## 6. 新增按钮二：结束本次对话

## 6.1 这个按钮的目的

这个按钮用于解决一个非常明确的问题：

- 用户已经完成一次完整运价查询
- 并围绕这批结果连续问了很多轮
- 现在希望结束这轮分析，开始下一票新询价

这个按钮的作用是：

- 结束当前这一轮运价分析会话
- 保留历史聊天记录
- 清空当前活跃询价上下文
- 后续新问题作为新的会话段开始

---

## 6.2 按钮展示位置

你已经明确产品要求：

- 这个按钮要在 AI 回复信息之后
- 居中保留
- 形式为：`icon + 结束本次对话`

建议展示规则：

1. 只在当前这一轮已经拿到完整 AI 回复后显示
2. 位置在本轮 AI 回复卡片下方居中
3. 在本轮尚未结束前，这个按钮一直可见
4. 用户点击后，该按钮不再继续作为“活跃操作”按钮显示，而是切换为“本轮已结束”状态

建议按钮文案固定为：

- `结束本次对话`

---

## 6.3 按钮点击后的前端行为

用户点击“结束本次对话”后，前端应立即做两件事：

### 第一件事：本地 UI 状态更新

1. 当前这轮会话区块标记为已结束
2. 当前这轮会话视觉上封箱
3. 当前活跃上下文 `context` 清空
4. 后续输入不再归入当前活跃会话块

### 第二件事：下一次请求时带上控制参数

用户下一次输入并真正发起请求时，请求体中增加：

```json
"reset_quote_context": true
```

这是这次前后端交接中最关键的新增参数要求。

---

## 6.4 重要说明

这里不要在用户点击按钮当下立刻单独发一个空请求给后端。  
更推荐的方式是：

1. 用户点击“结束本次对话”
2. 前端先本地标记当前会话已结束
3. 当用户下一次真正输入新问题时
4. 再把 `reset_quote_context: true` 一起带上发给后端

这样更自然，也更符合用户心智。

---

## 7. 结束本次对话后的下一轮请求示例

假设用户刚结束上一轮运价分析，随后开始问：

- `青岛到莫斯科空运多少钱`

则前端下一次请求建议这样发：

```json
{
  "session_id": "web_xxx",
  "message": "青岛到莫斯科空运多少钱",
  "context": null,
  "reset_quote_context": true
}
```

说明：

- `reset_quote_context: true`
  表示告诉后端：上一轮分析已经结束
- `context: null`
  表示当前前端不再继续带旧询价上下文

建议规则：

- 这次请求发出后，前端即可把本地“待重置”状态清掉
- 后续继续正常回到 `reset_quote_context: false`

---

## 8. 建议的前端消息区块行为

前端后续建议不要把所有聊天内容平铺成一整条时间线，而是按“报价分析会话”做区块分段。

建议一轮完整区块包含：

1. 用户本轮起始询价
2. AI 返回的报价结果
3. 用户围绕这批结果的继续追问
4. AI 对结果的继续分析回答
5. 底部居中显示 `icon + 结束本次对话`

点击结束后：

- 该区块显示 `已结束`
- 后续新问题进入新的一段区块

这样用户更容易分清：

- 哪些消息属于上一票货
- 哪些消息属于下一票货

---

## 9. 前端与后端的交接流程

## 9.1 正常运价查询

流程：

1. 用户输入询价内容
2. 前端发送 `/api/chat`
3. `reset_quote_context = false`
4. 后端返回流式内容
5. 前端逐步渲染 Markdown
6. 收到 `context` 后缓存
7. 收到 `done` 后结束本轮 loading

---

## 9.2 围绕当前结果继续追问

流程：

1. 用户继续问：
   - 最便宜的是哪个
   - 直达有哪些
   - 散货有哪些
2. 前端继续复用同一个 `session_id`
3. 前端继续带当前 `context`
4. `reset_quote_context = false`
5. 后端按当前活跃结果分析会话处理

---

## 9.3 用户中断请求

流程：

1. 用户点击“中断请求”
2. 前端中断网络请求
3. 前端停止继续渲染
4. 当前消息显示为 `已中断`
5. 前端不继续写入本轮未完成的 `context`

---

## 9.4 用户结束本次对话

流程：

1. 用户点击 `icon + 结束本次对话`
2. 前端本地将当前会话块标记为已结束
3. 前端清空当前活跃上下文
4. 用户下一次输入新问题
5. 发请求时增加：
   - `reset_quote_context: true`
6. 后端重置本轮询价上下文，进入新一轮查询

---

## 10. Markdown 渲染要求

这次新后端能力不会改变返回格式，前端仍然需要继续支持 Markdown 渲染。

至少要支持：

- Markdown table
- 段落
- 列表
- 粗体
- 换行

原因：

- 后端结果分析能力很多都会以 Markdown 表格或结构化文本返回
- 如果前端只按纯文本显示，结果可读性会明显下降

---

## 11. 前端接入时的注意事项

### 11.1 `session_id` 不变

同一整个聊天页内，仍然复用同一个 `session_id`。

说明：

- “结束本次对话”不是换 session
- 而是在同一个聊天会话里结束本轮询价上下文

### 11.2 `context` 不能乱改字段名

前端仍然应该：

- 收到什么就按结构缓存什么
- 不自己改字段名
- 不自己造字段值

### 11.3 `reset_quote_context` 是控制信号，不是普通业务字段

这个字段只用于告诉后端：

- 当前这轮运价分析已经结束

不要把它混进普通业务上下文里。

### 11.4 不要把“中断请求”和“结束本次对话”合并成一个按钮

这两个动作含义完全不同：

- `中断请求`
  - 停止当前还没结束的请求

- `结束本次对话`
  - 结束当前这轮报价分析会话

---

## 12. 推荐前端开发顺序

建议前端按以下顺序接入：

### 第一步

- 继续保持现有 SSE 读取逻辑
- 增加 `AbortController`
- 增加“中断请求”按钮

### 第二步

- 增加会话区块概念
- 在 AI 回复后居中显示 `icon + 结束本次对话`
- 增加本地“本轮已结束”状态

### 第三步

- 在下一次新请求中增加 `reset_quote_context: true`
- 确认后端能按新规则重置本轮上下文

---

## 13. 最终总结

这次前后端交接的本质，不是接口大改，而是：

- 在原有 `/api/chat` 基础上
- 增加一个控制字段
- 增加两个前端按钮
- 明确一轮运价分析会话什么时候开始、什么时候结束

一句话总结：

**前端后续要做的不是“继续聊天”，而是“围绕一批报价结果做可中断、可结束、可分段的运价分析会话”。**
