# 2026-06-09 Agent 优化后能力评估

## 目标

基于已有问题分析文档 `2026-06-09_agent-problem-analysis.md`，评估：

1. 如果把这批问题系统性修掉，当前项目 Agent 能达到什么效果
2. 从多个角度，尤其是专业货代运价场景，当前项目处于什么水平
3. 现在是否需要引入 Redis / MySQL，还是应该先把别的事情做好

本文不是改代码方案，而是“能力上限 + 产品定位 + 架构节奏”的判断。

---

## 一、如果把这次问题优化掉，Agent 可以达到什么效果

### 1. 先说结论

如果把这次暴露出的对话路由、上下文污染、查价前校验、业务解释类问法这几类问题修好，当前项目的 Agent 可以达到：

- 不是“全自动货代专家”
- 但可以成为“可用的一线空运询价与业务支持助手”
- 能稳定承接官网 70% 左右的标准化首轮咨询与中低复杂度多轮追问

换句话说，优化后它的定位应该是：

- 前台接待 + 询价收参 + 基础解释 + 结果初筛
- 不是最终的高复杂度方案设计师
- 更不是替代专业操作 / 销售 / 航线经理

---

### 2. 优化后能稳定做到的能力

如果这一轮把问题修透，Agent 应该能稳定做到下面这些事：

#### A. 标准询价场景

- 能识别单始发港、多始发港、全部港口
- 能识别目的港、重量、体积、日期、包装类型、航司、直飞/中转
- 缺参数时先追问，不乱查
- 参数冲突时先确认，不硬查
- 参数明显错误时先纠偏，不直接调工具
- 参数齐全后再调用报价接口

这意味着它会从“动不动就查”变成“先理解，再查，再解释”。

#### B. 多轮补参场景

- 用户第一句只说一部分条件，系统能连续补问
- 用户后面只回一个短答案，比如“上海”“散货”“明天”，系统能正确挂回上一轮槽位
- 用户改条件时，系统能判断是“补充”还是“重开新询价”
- 用户说“那南京呢”“那全部港口呢”“那直飞呢”，系统能在已有条件上局部调整

这块做稳之后，体验会更像业务助理，而不是一问一答表单机。

#### C. 结果解释场景

- 默认只展示当前条件下最有代表性的结果
- 用户要“全部数据”时再展开
- 用户问“为什么是这个价格”“最便宜的是哪条”“有没有托盘/散货差异”，系统能围绕当前结果继续解释
- 用户问“你是不是漏了始发港”“你先别查先确认”，系统能回到确认链路，而不是继续错误推进

这块决定了它会不会像“会话型工具”，还是像“只会吐文本的接口代理”。

#### D. 业务解释与引导场景

- 用户问“全部港口有哪些”，能回答固定白名单口岸范围
- 用户问“你能做什么”，能介绍支持的查询能力
- 用户问“你为什么这样理解”，能解释缺了什么参数、下一步需要什么
- 用户不是在查价，而是在确认流程，系统能解释和引导，而不是一律兜底

这一层补上以后，Agent 才开始具备“业务前台”属性。

#### E. 轻闲聊场景

- “你好”“在吗”“你是谁”能自然接
- 简单非业务对话可以接住
- 但不会把闲聊内容污染进运价槽位

这不是为了做陪聊，而是为了避免用户觉得系统“只会死板挡回去”。

#### F. RAG 业务问答场景

如果现有 RAG 链路继续稳定化，优化后它能承担：

- 锂电池/危险品/普货/报关/系统操作等资料型问答
- 能基于 metadata + 混合检索做基础业务资料回答
- 检索不足时说明边界，而不是瞎编

但这里要强调：

- 它回答的是“已有资料里的规则和说明”
- 不是“实时市场操作经验”
- 更不是“保证 100% 正确的外部监管口径”

---

### 3. 优化后不能指望它做到的事

即使这轮问题修完，也不要把它当成下面这些角色：

- 不能替代资深航线经理做临场定价判断
- 不能替代操作同事判断某票货是否真能接、真能飞、真能过板
- 不能替代销售做客户级让利策略
- 不能替代关务或危品专员做强合规判断

原因很简单：

- 它现在拿到的是报价接口结果，不是完整的舱位、操作、限制、客户政策、历史成交、异常规则系统
- 它有知识资料，但没有足够深的“流程状态 + 风险规则 + 经营策略”数据闭环

所以优化后的合理定位是：

`强前台助手，不是最终业务决策人`

## 二、从各个角度整体评估这个 Agent

下面的评估分两层：

1. 当前项目在“这批问题修完之后”的可达水平
2. 从专业货代空运角度看，它距离成熟产品还差什么

---

### 1. 会话与交互能力

#### 当前优化后可达评价

- 评级：`中上`

如果把这次的路由顺序、上下文复用、meta 问题处理修掉，会话能力能从“容易跑偏”提升到“可稳定接待”。

它会具备：

- 新询价识别
- 多轮追问
- 纠错确认
- 结果跟问
- 基础业务说明

但它仍然不会像通用聊天助手那样自然，因为本质上还是一个强约束业务 Agent。

#### 仍然欠缺的点

- 更细的用户意图层级
- 对反问、模糊表达、业务口语的容忍度
- 更强的“我刚才理解错了，我回退一下”能力

---

### 2. 询价流程完整度

#### 当前优化后可达评价

- 评级：`中上`

从流程完整性看，当前项目其实已经具备了一个标准报价 Agent 的主骨架：

- 意图识别
- 槽位提取
- 追问补参
- 工具调用
- 结果语义化
- context 跨轮延续

这套骨架是对的。现在最大的问题不是“没有流程”，而是“流程优先级和校验层不够硬”。

如果把这轮问题修掉，它至少能成为一个稳定的官网询价入口。

#### 仍然欠缺的点

- 查价前合法性校验还需要更系统
- 条件变更与新询价重开的边界要更稳
- 缺少“确认后再查”的显式对话层

---

### 3. 专业货代运价能力

这一项必须单独看，因为这是项目最核心的价值。

#### 当前优化后可达评价

- 评级：`中等`

为什么不是“高”：

- 它现在擅长的是“把标准化查询条件喂给报价接口”
- 不是“自己理解和推导复杂运价逻辑”

也就是说，它的专业度很大程度来自两部分：

1. 你们已有报价接口的真实质量
2. 它能否把用户条件准确、完整、干净地送到接口

在这个前提下，它优化后会非常适合：

- 普货标准询价
- 官网客户首轮询价
- 多个始发港比价
- 全部港口快速扫价
- 最便宜方案筛选
- 简单条件对比

#### 它已经具备的专业价值

- 支持计费重逻辑
- 支持包装类型区分
- 支持日期和日期相近查找
- 支持航司、直飞/中转等条件
- 能把报价结果做成用户能读懂的话术

这些已经让它不再是一个“空白聊天框”，而是一个真正接了业务接口的工具型 Agent。

#### 它还不够专业的地方

从真正货代一线看，当前还缺下面这些深度：

- 没有件数、尺寸、超长超重、是否可叠、是否带托的更细规则
- 没有危品等级、锂电池分类、是否带磁、是否带电等运价分层判断
- 没有客户级价格策略、合约价、权限价
- 没有舱位可订性、时效波动、航司偏好、甩货风险等运营信息
- 没有起运口岸差异化规则，比如某些口岸可接、某些口岸不可接
- 没有报价有效期管理、市场波动提醒、节假日特殊规则
- 没有把“报价”与“可执行性”拆开

所以它当前更像：

- `报价入口 Agent`

不是：

- `全流程空运业务专家 Agent`

---

### 4. RAG 业务知识能力

#### 当前优化后可达评价

- 评级：`中等`

从代码结构看，RAG 主链路已经不是占位了，已经具备：

- query analyzer
- hybrid retrieval
- generator
- metadata filter
- BM25 + 向量检索架构

这说明项目方向是对的，而且已经过了“只有 PPT 没有实现”的阶段。

但它现在的上限仍然受制于：

- 文档范围只有当前这些资料
- metadata 是否足够细
- 向量检索默认是否真正启用
- 资料更新机制是否稳定
- 回答是否带足够强的边界控制

所以当前优化后，RAG 能承担：

- 内部资料型问答
- 表单/制度/操作说明类问答

但还不适合承担：

- 涉及监管口径变化的强承诺回答
- 超出资料范围的开放性业务判断

---

### 5. 工程稳定性

#### 当前优化后可达评价

- 评级：`中等`

优点：

- 主链路清晰
- 节点边界明确
- Prompt 集中
- 工具调用和 SSE 接口没有混在一起
- RAG 与询价至少在结构上分开了

问题：

- 当前状态机规则较多，分支优先级复杂
- 很多行为依赖局部启发式规则，容易回归
- 自动化测试覆盖还不够
- 运行时状态目前有单机内存依赖

所以现在更像：

- `能跑、能迭代、但还没到高可维护产品化`

---

### 6. 可观测性与排障能力

#### 当前优化后可达评价

- 评级：`中上`

这块其实比表面看起来要好一些，因为项目已经有：

- `history.log`
- 比较完整的节点日志
- 中间状态打印
- SSE 请求链路日志

这意味着它不是“黑盒”。

这很重要，因为 Agent 项目最怕两件事：

1. 出错了不知道为什么
2. 不知道是路由错、槽位错、上下文错，还是工具错

当前项目至少已经有了继续做精细化调试的基础。

## 三、优化后，这个项目 Agent 的合理产品定位

如果本轮问题修透，建议把这个 Agent 的产品定位明确成：

### 定位

`官网空运询价与业务支持助手`

### 主要职责

- 接住用户首轮咨询
- 帮用户把询价条件说清楚
- 快速查出最低价或指定条件结果
- 对结果做基础解释
- 回答已有资料中的业务问题
- 在解决不了时再引导人工

### 不应该承担的职责

- 复杂项目型物流方案设计
- 需要强经验判断的危品接货决策
- 高风险承诺型合规判断
- 客户级商业谈判

这个定位很关键。定位不收口，用户期待就会失真，最后会把所有体验问题都归因到“AI 不专业”。

## 四、Redis / MySQL 现在需不需要加

## 1. 先给结论

### Redis

- `现在不是必须`
- `但中期非常建议加`

### MySQL

- `现在不急`
- `除非你马上要做后台化、留痕、分析、运营闭环`

---

## 2. 为什么 Redis 现在不是必须，但以后大概率要加

当前项目的运行状态里，有一类东西已经明显在走“进程内内存态”，例如：

- 最近一次报价结果
- 结果分析上下文
- 某个 session 的活动状态

这在单实例、低并发、内部调试阶段是可以接受的。

### 现在不必须加 Redis 的原因

- 你当前第一优先级不是扩容，而是把路由和对话质量做稳
- 现在最大的痛点不是性能瓶颈，而是行为正确性
- 过早引入 Redis，只会把错误状态持久化得更稳定，不会让对话更聪明

### 什么时候建议加 Redis

当你出现下面任一需求时，就建议加：

- 服务要多实例部署
- session 不能只存在单机内存里
- 希望报价结果/上下文跨实例共享
- 要做接口缓存，减少重复调用报价接口
- 要做限流、会话超时、幂等控制
- 要把 SSE / 会话状态做得更稳

### Redis 适合放什么

- session 运行态
- 最近一次报价结果缓存
- 短期上下文
- 频繁相同报价条件的短 TTL 缓存
- 限流计数

### 我对 Redis 的建议

- 不是这轮立即加
- 但建议列为下一阶段基础设施优先项
- 优先级高于 MySQL

---

## 3. 为什么 MySQL 现在不急

MySQL 适合解决的是“结构化长期数据”问题，不是当前最核心的“行为正确性”问题。

### 现在不急加 MySQL 的原因

- 当前主价值来自实时询价 + RAG，而不是复杂后台事务
- 已有向量知识库不依赖 MySQL
- 当前还没有很强的“用户、订单、客户、权限、报表、运营后台”闭环需求

### 什么时候建议加 MySQL

当你要做下面这些事时，再加最合理：

- 保存完整会话历史
- 保存询价记录与用户行为数据
- 做报价分析报表
- 做客户维度画像、偏好、权限
- 做人工接管工单
- 做知识库文档管理后台
- 做反馈闭环和标注集

### MySQL 更适合放什么

- 用户与会话元数据
- 询价日志与分析报表
- 客户档案
- 业务反馈
- 人工接管记录
- 配置化规则

### 我对 MySQL 的建议

- 当前阶段不是必须项
- 在“Agent 行为稳定 + 业务量起来”之后再加
- 不建议为了“看起来像正式系统”提前引入

---

## 五、当前阶段最值得补的，不是 Redis/MySQL，而是这几件事

如果从投入产出比排序，我建议优先级如下：

### 第一优先级：把 Agent 行为修正确

- 新询价识别
- 查价前校验
- 多始发港解析
- meta 问题处理
- fallback 分层

这是直接决定用户觉得它“聪明”还是“离谱”的根因。

### 第二优先级：补自动化回归

- 单港询价
- 多港询价
- 全部港口
- 缺参数追问
- 新询价重开
- 结果继续追问
- 业务解释类问法
- 闲聊不污染上下文

否则后面每改一次规则，都可能把别的场景打坏。

### 第三优先级：补观察面板和可配置项

- 更清晰的 intent 命中日志
- query_ready 形成原因
- 为什么追问 / 为什么查价 / 为什么兜底
- 白名单港口、别名词典、业务说明文案配置化

这会显著降低后续维护成本。

### 第四优先级：再考虑 Redis

当行为已经稳定、并发开始上来，再把运行态移出进程内存。

### 第五优先级：最后考虑 MySQL

当你们明确要做后台运营、客户分析、闭环管理时再上。

---

## 六、一个更直白的整体判断

### 1. 当前这个 Agent 值不值得继续做

- 值得

因为它不是从零开始，已经有了：

- 报价主链路
- SSE 接口
- context 多轮机制
- RAG 基础框架
- 可观测日志

这说明它不是方向错了，而是还处于“从能跑到能用”的关键阶段。

### 2. 当前最大问题是不是模型不够强

- 不是

当前最大问题仍然是：

- 路由优先级
- 状态污染
- 校验层缺失
- 对话层过薄

这些不解决，换更强模型也还是会跑偏，只是“更会说错话”。

### 3. 当前优化完之后，它能到什么档位

我的判断是：

- 作为官网第一层接待与标准询价 Agent：`可以达到可上线试运行水平`
- 作为专业货代报价专家：`还不够`
- 作为内部资料问答助手：`可以逐步形成价值`

如果用更产品化的话说：

- `可做业务入口`
- `暂不该做最终决策人`

---

## 七、最终结论

如果把这次暴露出来的问题优化掉，这个项目 Agent 可以从：

- “容易乱跳、容易误查、容易离谱兜底的规则型聊天程序”

提升到：

- “能稳定承接空运询价、多轮补参、结果解释、业务资料问答的一线业务助手”

但它的能力边界也要说清楚：

- 它会是一个强前台 Agent
- 不是资深货代专家替身

关于基础设施，我的结论是：

- `Redis`：中期建议加，但不是当前第一优先级
- `MySQL`：当前不急，等后台化和数据闭环需求明确后再上

当前最值得做的，不是先堆中间件，而是把这四件事彻底做对：

1. 询价前校验
2. 新旧上下文边界
3. 业务解释与闲聊分层
4. 回归测试与可观测性

把这四件事做扎实，这个 Agent 才会真正从“能演示”迈向“能上线”。
