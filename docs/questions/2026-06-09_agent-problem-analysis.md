# 2026-06-09 Agent 问题分析

## 目标

基于 `history.log` 与当前实现代码，分析目前 Agent 为什么会出现：

1. 该追问时不追问，反而直接查价或直接给无结果。
2. 明显不是业务/RAG失败场景，却频繁走兜底或错误意图。
3. 上一轮上下文污染下一轮，导致新询价被误解成“继续解释旧结果”。
4. 非业务问题几乎没有像样的对话能力，只会掉到兜底回复。

本文不直接修代码，先把问题链路、根因、潜在问题和建议方案讲清楚。

## 一、你已经发现的问题，对应根因是什么

### 问题 1：缺始发港时没有追问，直接查成 `LAX -> LAX`

现象：

- 你问：`我有一票货，发往洛杉矶，散货，600公斤，4个立方，今天发多少`
- Agent 没追问始发港，直接进入查询或无结果。
- 你再追问：`你不问我始发港是哪里吗`
- Agent 继续拿错误上下文重复查。

日志证据：

- `2026-06-09 11:14:22`：
  `slot_node` 抽取结果里已经变成 `sfg='lax', mdg='lax'`
- `2026-06-09 11:14:40`：
  实际调用接口时参数是 `sfg=lax&mdg=lax`
- `2026-06-09 11:15:23`：
  你问“你不问我始发港是哪里吗”后，系统继续沿用旧上下文 `sfg=lax, mdg=lax`

根因：

1. 当前 `slot_node` 仍然主要依赖 LLM 槽位抽取，缺少“路线合法性校验”。
2. 对于 `发往洛杉矶` 这种句子，LLM 把目的港 `lax` 同时填进了 `sfg` 和 `mdg`。
3. `_build_missing_required_slots(...)` 只检查 `sfg` 有没有值，不检查这个值是否合理。
4. 一旦 `sfg` 被误填成 `lax`，系统认为必填槽位已齐，就不会再追问始发港。
5. 错误状态被写回 `context` 后，后续对话继续复用这个脏上下文。

本质问题：

当前系统的“是否可以查价”判断，只有“字段非空”，没有“字段正确”。

---

### 问题 2：`你不问我始发港是哪里吗` 被继续查价，而不是进入解释/纠错

现象：

- 用户明确指出系统漏问始发港，Agent 没有理解这是对流程的纠偏，而是继续调用报价接口。

日志证据：

- `2026-06-09 11:15:21`
  `intent_node` 把 `你不问我始发港是哪里吗` 判成了 `rate_query`
- `2026-06-09 11:15:23`
  `slot_node` 没抽出新值，于是直接合并旧上下文，`query_ready=True`
- 然后继续查 `lax -> lax`

根因：

1. 当前意图体系里没有“元问题 / 流程纠错 / 对系统行为的质疑”这一类。
2. 用户这句话既不是标准报价，也不是 RAG，也不是结果解释，但系统只有：
   - `rate_query`
   - `result_analysis`
   - `result_reference`
   - `support_info`
   - `rag`
   - `unknown`
3. 最终这句话被粗暴塞回 `rate_query`。
4. `slot_node` 合并逻辑是“新值没有抽到，就沿用旧值”，导致脏上下文继续有效。

本质问题：

Agent 没有“承认自己刚才理解错了，并回到补槽位”的能力。

### 问题 3：`全部港口有哪些` 被判成 `unknown`

现象：

- 你问：`全部港口有哪些`
- Agent 回复了默认兜底：`您好！我是专业的国际空运报价助手...`

日志证据：

- `2026-06-09 11:17:22`
  `intent_node debug | response.content=unknown | intent=unknown`
- 然后直接走 `fallback`

根因：

1. 当前 `intent_node` 对 `全部港口有哪些` 没有专门规则。
2. 这句话既不是直接查价，也不是 RAG 文档问答，也不属于现有 `support_info_kind`：
   - `capability_intent`
   - `service_info`
3. 所以它被 LLM 判到了 `unknown`。

更深一层的问题：

这句话其实是“业务域内的说明性问题”，不是无关问题。
它至少可以走两种更合理的处理方式：

1. 走 `support_info`，解释“全部港口”会查询哪些分公司口岸。
2. 走报价追问链，先回答港口范围，再引导继续报价。

本质问题：

当前意图设计把“业务域内但不是直接报价”的问题覆盖得太窄。

### 问题 4：正常询价后，再发新询价，系统把整句新询价误判成“解释旧结果”

现象：

- 你已经查过一轮。
- 你再问：`我有一票货，从南京，上海，发往洛杉矶 890 公斤 9 个立方， 托盘，今天发货多少钱`
- Agent 回复：`当前默认展示的结果里，散货最低价合计是 36018.00CNY...`

日志证据：

- `2026-06-09 14:48:18`
  `intent_node rule hit | result_reference=true`
- 这说明系统把整句新询价，当成了“围绕旧报价结果问总价”

根因：

1. 当前只要 `quote_result_active=True`，`intent_node` 会优先尝试 `analyze_result_reference_request(...)`。
2. 新询价句子里带了 `多少钱`，而结果引用规则里把 `多少钱` 直接映射成了 `price_total`。
3. 因为这个判断发生在“新询价识别”之前，所以整句新报价被截胡，进入 `result_reference`。

本质问题：

系统把“多少钱”过度绑定为“解释旧结果价格”，却没有先判断这是不是一条完整新询价。

本质问题：

## 三、这些问题为什么会让 Agent 看起来“像个傻子”

核心不是模型智商，而是状态机设计出了以下结构性问题：

### 1. 路由优先级不对

当前顺序里，很多“结果分析 / 结果引用 / 包装类型短答 / follow-up”规则，优先级高于“新完整询价”。

结果：

- 只要上一轮还残留上下文，新一句完整询价也可能被旧上下文吞掉。

### 2. 只看“字段有没有值”，不看“字段值对不对”

例如：

- `sfg=lax`
- `mdg=lax`

这其实明显不合理，但系统只看到：

- `sfg` 有值
- `mdg` 有值

于是直接放行查询。

### 3. 兜底设计过粗

当前 `unknown` 只返回一个固定提示，不区分：

1. 真无关问题
2. 轻闲聊
3. 业务域内说明问题
4. 用户对系统的纠错/质疑

所以用户体验就会显得“只会背模板”。

### 4. 错误上下文没有被及时打断

一旦抽错，错误状态就会写回前端 `context`。
后续每一轮都在这个错误基础上继续推理。

这就是为什么你问一句“你不问我始发港吗”，系统还能继续拿 `lax -> lax` 查。

## 四、结合你的想法，应该怎么优化

你给的方向是对的，我建议收敛成下面这套优化目标。

### 目标 1：只有“业务和 RAG 解决不了”的问题，才进入人工咨询兜底

这意味着要把现有兜底拆成至少三层：

1. `chat_smalltalk`
   - 轻闲聊、打招呼、简单互动
   - 不更新询价槽位
2. `business_meta`
   - 解释系统行为、确认用户意图、回答“全部港口有哪些”“你为什么这样理解”
   - 允许引导用户继续问业务问题
3. `hard_fallback`
   - 真正无法回答的业务 / RAG 问题
   - 才给人工邮箱

也就是说：

- `您好！我是专业的国际空运报价助手...`
  不应该再承担所有 `unknown`。
- `您可以咨询我司相关人员...`
  只应该用于真正解不了的业务问题。

---

### 目标 2：不要动不动就查询，先确认再调用工具

建议在 `tool_node` 前新增一个“查询前校验 / 纠偏层”：

可以叫：

- `pre_query_validation`

职责：

1. 检查 `sfg` 和 `mdg` 是否同时存在
2. 检查 `sfg != mdg`
3. 检查 `sfg` 是否不应明显等于目的港词
4. 检查多始发港里是否错误混入了目的港
5. 如有问题，不进工具，转成追问或纠错说明

例如：

- `发往洛杉矶`
  不应让 `sfg=lax`
- `从南京，上海，发往洛杉矶`
  不应让 `sfg` 包含 `lax`

---

### 目标 3：先判“是不是一条完整新询价”，再判是不是旧结果 follow-up

这条非常关键。

推荐优先级改成：

1. greeting / 小闲聊
2. 用户是否在明确纠错系统
3. 是否是完整新询价
4. 是否是补槽位回复
5. 是否是旧报价结果分析 / 结果解释
6. 是否是 RAG
7. 再考虑 fallback

这样能避免：

- `我有一票货...多少钱`
  被误判成 `result_reference.price_total`

---

### 目标 4：给 Agent 增加“解释、追问、确认、引导发散”的能力

你说“不要太像个傻子动不动就查询”，本质上是要增加对话型中间态。

建议新增一类轻量节点或意图：

- `conversation_guidance`

它处理的不是查价，也不是 RAG，而是：

1. `你不问我始发港吗`
2. `你是不是理解错了`
3. `全部港口有哪些`
4. `你先别查，先确认我的条件`
5. `你帮我看看还缺什么`

这类问题的正确行为通常是：

1. 承认当前缺失或理解偏差
2. 给出简明解释
3. 回到报价链继续追问

## 五、建议的优化方案

### 第一阶段：先止血

目标：先把“离谱误判”压下去。

建议修改：

1. 在 `tool_node` 前增加路线合法性校验
2. 修正 `origin_parser`，补 `发往 / 发去 / 送往 / 去往`
3. 新询价识别优先级提升到结果引用之前
4. 缩紧 `_looks_like_package_type_reply(...)`
   - 必须是短句
   - 不能是带重量/体积/日期/目的港的完整句子
5. 缩紧 `_has_rate_context(...)`
   - 不再用“两个字段以上”作为强上下文
6. `你不问我始发港吗`
   这类 meta 问题优先走解释/纠错，而不是继续查价

### 第二阶段：把 `unknown` 拆开

建议新增：

1. `chat_smalltalk`
2. `business_meta`
3. `hard_fallback`

这样：

- 非业务闲聊可以自然接一下
- 业务域内说明性问题不再掉兜底
- 真正解决不了的才转人工

### 第三阶段：对话能力升级

建议明确支持：

1. 追问
2. 确认
3. 纠错
4. 引导发散

例如：

- `你不问我始发港吗`
  应答：
  `对，这票目前还缺始发港。我需要先确认您从哪个城市/机场发货。您可以回复一个或多个始发港，例如“上海、南京”；如果要全口岸一起查，也可以直接回复“全部港口”。`

- `全部港口有哪些`
  应答：
  `目前“全部港口”会按分公司固定口岸一起查询，包括：厦门、广州、深圳、香港……如果您要，我也可以直接按全部港口帮您查飞洛杉矶的价格。`

## 六、建议的最终产品行为

### 1. 报价问题

规则：

- 缺参数就追问
- 参数冲突就确认
- 逻辑不合理就纠偏
- 参数齐全再查

### 2. RAG 问题

规则：

- 能检索就回答
- 检索不足就说明边界
- 还不够再引导用户补充信息

### 3. 非业务问题

规则：

- 可以轻闲聊
- 不更新槽位
- 不动不动掉兜底

### 4. 真兜底

只有在：

1. 业务问题超出系统能力
2. RAG 没有依据
3. 无法通过追问解决

这三类场景下，才给人工邮箱。

## 七、结论

目前 Agent 表现“像个傻子”，不是单点 bug，而是四类问题叠加：

1. 意图路由优先级错误
2. 查询前没有做路线合法性校验
3. 错误上下文会被持续复用
4. `unknown` 过粗，缺乏 chat / meta / guidance 三种中间能力

如果只修某一个 Prompt，效果不会根治。

真正要提升，需要把系统从“能查价的状态机”升级成“会确认、会解释、会纠错、会引导的业务 Agent”。

当前最优先的止血项是：

1. 修新询价识别优先级
2. 修 `origin_parser`
3. 修查询前合法性校验
4. 修 `result_reference` 对“多少钱”的误判
5. 给 `unknown` 拆层
