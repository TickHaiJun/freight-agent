# 本地项目容器化与部署实施方案

## 1. 这份方案要解决什么

你现在已经把云服务器基础环境准备好了：

- Ubuntu 系统已就绪
- 数据盘已挂载
- Docker 已安装
- 目录已创建
- 当前还没有配置 Nginx

接下来要做的，不再是服务器环境层，而是“本地项目补齐容器化部署能力”，也就是你列出的这 6 项：

1. 补 `Dockerfile`
2. 补 `.dockerignore`
3. 补 `docker-compose.yml`
4. 补 Nginx 配置
5. 准备 `.env.production`
6. 最后 `docker compose up -d --build`

这份文档的目标，就是把这 6 项拆成一套可落地的实施方案，并明确：

- 我会如何做
- 会改动哪些文件
- 每个文件为什么这样设计
- 上线顺序是什么
- 哪些地方要特别注意

---

基于当前仓库实际情况，现状是：

## 3. 这次我推荐的最终部署形态

我建议最终采用这套结构：

```text
浏览器 / 前端
    ->
公网 IP:80/443
    ->
Nginx 容器
    ->
freight-agent 应用容器
    ->
DeepSeek / DashScope / 运价接口
```

也就是说：

- 应用容器内部监听 `8012`
- 宿主机不开放 `8012`
- 外网统一访问 Nginx
- Docker Compose 负责编排两类容器：
  - `freight-agent-app`
  - `freight-agent-nginx`

## 4. 我会如何落实这 6 项

### 4.1 第 1 项：补 `Dockerfile`

#### 目标

把当前 Python 项目打成一个可运行镜像。

#### 我会怎么做

我会新增一个基础版 `Dockerfile`，设计原则是：

1. 基于官方 Python 镜像
2. 安装 `requirements.txt`
3. 复制项目源码
4. 容器内暴露 `8012`
5. 以 `uvicorn main:app --host 0.0.0.0 --port 8012` 启动

#### 为什么这样做

因为当前项目本质就是一个单进程的 FastAPI 应用，容器化第一版不需要引入更复杂的进程管理器。

#### 关键设计点

1. 容器内监听 `0.0.0.0`
2. 由 Compose 决定是否映射端口
3. 日志写到容器挂载目录
4. 保持当前 `/api/chat` SSE 协议不变

---

### 4.2 第 2 项：补 `.dockerignore`

#### 目标

避免把无关文件打进镜像，减小构建上下文。

#### 我会怎么做

我会新增一个最小但实用的 `.dockerignore`，忽略这些内容：

1. `AiEnv/`
2. `__pycache__/`
3. `.git/`
4. `docs/history/`
5. `data/chroma/`
6. `data/cache/`
7. 本地日志文件
8. 本地临时文件

#### 为什么这样做

因为这些内容不是镜像运行时必须文件：

- 虚拟环境应该在镜像里重新构建
- 索引缓存不该依赖本地开发机产物
- 日志文件不应该被打包进镜像

#### 特别注意

不会忽略这些重要目录：

1. `graph/`
2. `rag/`
3. `tools/`
4. `scripts/`
5. `data/docs/`

尤其 `data/docs/` 不能误忽略，因为知识库原始资料需要进入服务器流程。

---

### 4.3 第 3 项：补 `docker-compose.yml`

#### 目标

让宿主机用一条命令启动应用容器和 Nginx 容器。

#### 我会怎么做

我会新增一个 `docker-compose.yml`，至少定义两个服务：

1. `freight-agent-app`
2. `freight-agent-nginx`

#### 应用容器设计

应用容器负责：

1. 从本地 `Dockerfile` 构建镜像
2. 加载 `.env.production`
3. 挂载知识库数据目录
4. 挂载日志目录
5. 不对公网直接暴露 `8012`

#### Nginx 容器设计

Nginx 容器负责：

1. 对外映射 `80:80`
2. 挂载 Nginx 配置文件
3. 转发 `/health` 和 `/api/chat`
4. 支持 SSE 场景，不做错误缓冲

#### 为什么这样做

因为 Compose 最适合当前这种单机、双容器、低复杂度部署形态。

---

### 4.4 第 4 项：补 Nginx 配置

#### 目标

让外部所有访问统一通过 Nginx，再代理到应用容器。

#### 我会怎么做

我会新增一个 Nginx 配置文件，建议路径：

```text
deploy/nginx/default.conf
```

或者如果你更想放部署目录外，也可以放：

```text
nginx/default.conf
```

#### 我倾向的方案

更推荐：

```text
deploy/nginx/default.conf
```

因为这属于部署层配置，不属于业务代码。

#### 配置核心点

1. `listen 80`
2. `server_name 47.101.141.117`
3. `proxy_pass http://freight-agent-app:8012`
4. SSE 场景关闭缓冲
5. 转发必要头信息

#### 为什么这样做

因为当前最重要的是：

- 先把入口统一
- 先让 Nginx 能稳定代理 SSE
- 先别急着搞 HTTPS

---

### 4.5 第 5 项：准备 `.env.production`

#### 目标

为容器部署准备生产环境变量文件。

#### 我会怎么做

我不会把真正生产密钥直接写进仓库，而是：

1. 新增一个模板文件
2. 例如：

```text
deploy/.env.production.example
```

然后服务器实际使用：

```text
.env.production
```

由你自己填写真实密钥。

#### 为什么这样做

因为：

1. 真实密钥不能进 Git
2. 生产环境路径和本地路径不同
3. Compose 读取生产环境变量更清晰

#### 这份文件里我会重点处理的配置

1. `DEEPSEEK_API_KEY`
2. `FREIGHT_API_BASE`
3. `DASHSCOPE_API_KEY`
4. `CHROMA_PERSIST_DIR`
5. `RAG_DOCS_DIR`
6. `APP_LOG_DIR`

#### 这里一个关键点

宿主机路径和容器路径不一样。

例如：

- 宿主机日志目录：`/data/logs/freight-agent`
- 容器内日志目录：`/app/runtime/logs`

所以生产环境变量必须按容器视角写。

---

### 4.6 第 6 项：最后 `docker compose up -d --build`

#### 目标

把整个服务真正启动起来。

#### 我会怎么做

当上面 5 项都补齐之后，再执行：

```bash
docker compose up -d --build
```

然后按顺序验证：

1. `docker compose ps`
2. `docker compose logs -f`
3. `curl http://127.0.0.1/health`
4. `curl http://47.101.141.117/health`
5. 测试 `/api/chat`
6. 检查 `/data/logs/freight-agent`

#### 为什么最后才做

因为这一步不是“准备工作”，而是“结果验证”。  
前面文件没补全之前直接启动，只会增加无效排障成本。

## 6. 具体设计上的关键判断

### 6.1 `Dockerfile` 是否直接复制整个仓库

我会采用“复制必要项目内容”的方式，而不是把所有东西都粗暴复制进去。

原因：

1. 避免镜像过大
2. 避免把开发文档、日志和无关目录打进去
3. 让镜像更可控

不过第一版我仍然会偏保守，优先保证能跑，再慢慢收紧。

### 6.2 Chroma 与日志放镜像内还是挂载出来

我会明确采用“挂载卷”的方式，不会让这些运行数据只留在容器层。

需要挂载的重点目录：

1. 宿主机 `/data/logs/freight-agent`
2. 宿主机项目中的 `data/docs`
3. 宿主机项目中的 `data/chroma`
4. 宿主机项目中的 `data/cache`

原因：

1. 容器删除后数据不能丢
2. 后续日志平台要直接读宿主机文件
3. RAG 索引不能每次重建

### 6.3 应用是否还需要宿主机 Python 环境

正式容器化后，应用运行本身就不依赖宿主机 Python 环境了。

宿主机主要需要：

1. Docker
2. Docker Compose
3. Nginx 容器配置

但是你如果要在宿主机手工调试脚本，Python 环境仍然可以保留，不冲突。

### 6.4 Nginx 是用宿主机安装还是容器化

我更倾向于：

- 这次和应用一起走 Compose 容器化

原因：

1. 跟部署方案更一致
2. 配置和版本更好迁移
3. 宿主机更干净

也就是说：

- 宿主机安装 Docker
- Nginx 本身跑在容器里

## 7. 实施顺序

如果后续我来正式落代码，我会按这个顺序做。

### 第一步

补部署层文件：

1. `Dockerfile`
2. `.dockerignore`
3. `docker-compose.yml`
4. `deploy/nginx/default.conf`
5. `deploy/.env.production.example`

### 第二步

检查当前配置是否需要最小兼容性调整，重点只看：

1. 日志目录
2. 数据目录
3. 容器启动命令

### 第三步

本地做静态校验：

1. 文件结构正确
2. Compose 语法正确
3. Nginx 配置正确

### 第四步

给你输出服务器实际执行顺序：

1. 上传项目
2. 放 `.env.production`
3. 执行 `docker compose up -d --build`
4. 验证 `/health`
5. 验证日志目录

## 9. 最终上线顺序

后续真正执行时，顺序应该是：

1. 先在本地补齐部署层文件
2. 再把代码上传或同步到服务器
3. 再填写服务器实际用的 `.env.production`
4. 再执行：

```bash
docker compose up -d --build
```

5. 再验证：

```bash
docker compose ps
docker compose logs -f
curl http://127.0.0.1/health
curl http://47.101.141.117/health
```

6. 最后检查：

```bash
ls -lah /data/logs/freight-agent
```
