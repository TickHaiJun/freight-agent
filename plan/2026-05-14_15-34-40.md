# 北京时间 UTC+8 兼容修复记录

## 创建时间
2026-05-14 15:34:40

## 背景
在前端联调过程中，`/api/chat` 请求出现启动即报错的问题。结合 `history.log` 定位后，根因不是运价接口、SSE 协议或结果分析链本身，而是请求初始化阶段创建北京时间对象失败。

关键报错：

- `ModuleNotFoundError: No module named 'tzdata'`
- `ZoneInfoNotFoundError: 'No time zone found with key Asia/Shanghai'`

当前运行环境为 Windows，本地 Python 环境没有可用的 `Asia/Shanghai` zoneinfo 数据，同时也没有安装 `tzdata` 兜底包，导致：

1. `/api/chat` 在进入 `generate()` 后立即失败
2. 还没进入意图识别、槽位提取、运价查询等业务链路

## 这次修复要解决的问题

1. 去掉对 `ZoneInfo("Asia/Shanghai")` 和 `tzdata` 的硬依赖
2. 保证当前项目能稳定获取“请求发生时的北京时间”
3. 不改现有时间业务规则，只修“北京时间基准来源”

## 核心决策

本次采用方案 A：

- 不安装新依赖
- 不继续依赖 `ZoneInfo("Asia/Shanghai")`
- 统一改成后端按 `UTC+8` 手动计算北京时间

这样做的原因：

1. 当前项目只需要“北京时间”这一种固定时区
2. 北京时间没有夏令时，`UTC+8` 计算足够稳定
3. 符合“最小成本改动、优先稳定”的约束
4. 避免不同环境下 `zoneinfo` / `tzdata` 缺失导致再次报错

## 改动范围

### 1. 新增统一时间工具
新增文件：

- `graph/time_utils.py`

职责：

- 提供当前北京时间 `datetime`
- 提供当前北京时间日期字符串 `YYYY-MM-DD`

实现方式：

- 先取当前 UTC 时间
- 再显式转换为 `UTC+8`

### 2. 修改主入口时间初始化
修改文件：

- `main.py`

改动点：

- 移除 `ZoneInfo` 依赖
- 引入 `get_current_beijing_date_str()`
- 在 `/api/chat` 的 `generate()` 中，统一通过工具函数生成 `current_beijing_date`

目的：

- 避免请求一进入就因时区对象创建失败
- 保持 `current_beijing_date` 仍然是每次请求动态计算

### 3. 修改节点内北京时间兜底逻辑
修改文件：

- `graph/nodes.py`

改动点：

- 移除 `ZoneInfo` 依赖
- 引入 `get_current_beijing_datetime()`
- `_parse_current_beijing_date()` 在 `state` 缺少日期时，不再调用 `ZoneInfo("Asia/Shanghai")`
- 改为走统一 UTC+8 工具函数

目的：

- 防止主入口修复后，节点内部兜底路径仍然报同类错误

## 修改文件

- `main.py`
- `graph/nodes.py`
- `graph/time_utils.py`

## 影响范围

本次只影响“北京时间基准”的获取方式，不改变以下逻辑：

- `今天 / 明天 / 后天 / 明后天 / 周末` 的业务规则
- `周一 / 越快越好 / 最近几天` 的追问规则
- 运价查询链路
- 结果分析链路
- SSE 输出协议

## 风险与边界

1. 当前方案只适用于“固定北京时间”场景
2. 如果未来要扩展多国家时区，需要再升级成更通用的时区方案
3. 这次没有改动时间业务词汇规则，只修复了时间基准兼容性

1. `/api/chat` 请求不再因时区初始化报错
2. `今天` 在当前日期下能正常解释为北京时间当天
3. `明天 / 后天 / 明后天 / 周末` 不受本次修复影响
4. 其它运价链路保持稳定

## 摘要

这次修复的本质不是“改日期业务规则”，而是把“北京时间 now”的获取方式从环境依赖型实现，改成了项目内可控的 `UTC+8` 计算。

这样做用最小改动解决了当前 Windows 环境下的 `tzdata / zoneinfo` 兼容问题，同时保留了现有时间查询链路和前后端协议不变。
