# 日期澄清短回复掉入 Fallback 修复记录

## 创建时间
2026-05-14 17:11:54

## 背景

在前端联调中发现一类多轮会话问题：

第一轮用户询价：

- `PVG 到 SYD 1000 公斤 20个立方 周六发多少钱`

系统追问：

- `请问您是指这周六，还是下周六？`

第二轮用户回复：

- `这周六`
或
- `下周六`

系统没有继续走运价查询链，而是直接返回欢迎语：

- `您好！我是专业的国际空运报价助手，可以帮您查询空运运价。请问您有空运报价需求吗？`

## 问题现象

从日志看，第一轮链路是正常的：

1. `intent_node` 命中 `rate_query`
2. `slot_node` 识别到 `周六` 有歧义
3. `ask_node` 追问“这周六还是下周六”

但第二轮短回复出现问题：

1. 用户回复 `这周六`
2. `intent_node` 没把它识别成“对上一轮追问的回答”
3. 大模型将其判成 `unknown`
4. 路由进入 `fallback_node`
5. 没有进入 `slot_node`

## 根因分析

根因分两层。

### 1. 规则覆盖不完整

当前 `intent_node` 的“承接上文继续报价”规则没有稳定覆盖：

- `这周六`
- `本周六`
- `下周六`
- `下下周六`

这类非常短的日期澄清回复。

所以第二轮没有命中 `rate_query` 规则，只能退回开放式意图识别。

### 2. 缺少跨请求的待澄清状态

第一轮虽然已经明确提出了日期澄清问题：

- `请问您是指这周六，还是下周六？`

但第二轮请求进入时，后端并没有显式保留：

- 当前正在等待澄清哪个槽位
- 上一轮的澄清问题是什么

因此第二轮只看到一句：

- `这周六`

系统把它当成“普通新输入”，而不是“对上一轮日期澄清的回答”。

## 修复目标

本次修复目标不是只补一个关键词，而是一起解决两件事：

1. 让 `这周六 / 下周六` 这类短回复能直接回到补槽位链路
2. 让“待澄清状态”可以跨请求保留并参与下一轮路由

## 核心决策

本次采用两条线同时修复：

### 方案一：规则补强

增强 `intent_node` 和时间规则：

- `FOLLOWUP_KEYWORDS` 增加 `这周 / 本周 / 下周 / 下下周`
- 时间解析增加对：
  - `这周六`
  - `本周六`
  - `下周六`
  - `下下周六`
  的直接解析

### 方案二：待澄清状态

新增并跨请求保留：

- `pending_clarify_slot`
- `pending_clarify_message`

用途：

- 上一轮如果系统在追问日期澄清，就把这个状态回传给前端
- 前端下一轮再原样带回
- `intent_node` 据此判断当前是否是“对追问的回答”

## 为什么不是只补一个关键词

如果只补：

- `这周`
- `下周`

虽然能修当前例子，但后续还会继续遇到同类问题：

- `本周三`
- `下下周五`
- 其它槽位的澄清回答

本质上这不是单条规则漏写，而是：

**系统已经会发起澄清，但还没有稳定地消费用户对澄清问题的回答。**

所以必须补“待澄清状态”。

## 改动范围

### 1. 状态定义
修改文件：

- `graph/state.py`

新增字段：

- `pending_clarify_slot`
- `pending_clarify_message`

作用：

- 显式记录当前正在等待用户澄清哪个槽位
- 记录上一轮的澄清问题内容

### 2. 请求初始化与 context 回传
修改文件：

- `main.py`

改动点：

- 从前端 `context` 恢复 `pending_clarify_slot`
- 从前端 `context` 恢复 `pending_clarify_message`
- 在 SSE `context` 回传中把这两个字段带回前端

作用：

- 保证待澄清状态可以跨请求闭环

### 3. 时间规则增强
修改文件：

- `graph/nodes.py`

新增：

- `WEEK_PREFIX_PATTERN`
- `_resolve_prefixed_weekday_date`

支持直接解析：

- `这周六`
- `本周六`
- `下周六`
- `下下周六`

### 4. 意图识别增强
修改文件：

- `graph/nodes.py`

新增：

- `_looks_like_time_clarify_reply`

作用：

- 如果当前会话正在等待日期澄清
- 且用户本轮消息看起来像一个明确日期补充
- 则直接命中 `rate_query`
- 不再重新走开放式意图识别

### 5. 槽位节点与待澄清状态联动
修改文件：

- `graph/nodes.py`

改动点：

- `slot_node` 发现时间歧义时：
  - 继续设置 `time_clarify_message`
  - 同时设置 `pending_clarify_slot = "hbrq"`
  - 同时设置 `pending_clarify_message`
- `tool_node` 成功或失败后，清空待澄清状态

作用：

- 避免待澄清状态残留污染后续会话

## 修改文件

- `graph/state.py`
- `main.py`
- `graph/nodes.py`

## 影响范围

本次改动只影响“澄清追问后的短回复处理”与日期续答识别，不改变：

- 运价查询主链结构
- 工具调用逻辑
- 结果分析链
- SSE 协议

主要提升以下场景：

- `这周六`
- `本周六`
- `下周六`
- `下下周六`

执行语法校验：


结果：

执行关键改动点检查：

```powershell

```

结果：

- 待澄清状态已在状态定义、请求初始化、context 回传、节点逻辑中闭环
- `intent_node` 已增加日期澄清短回复规则
- `slot_node` 已支持 `这周X / 本周X / 下周X / 下下周X`

## 建议回归场景

1. `PVG 到 SYD 1000 公斤 20个立方 周六发多少钱` -> `这周六`
2. `PVG 到 SYD 1000 公斤 20个立方 周六发多少钱` -> `下周六`
3. `PVG 到 SYD 1000 公斤 20个立方 周三发多少钱` -> `本周三`
4. `PVG 到 SYD 1000 公斤 20个立方 周三发多少钱` -> `下下周三`

预期：

- 第二轮不再进入 `fallback`
- 第二轮直接回到 `slot_node`
- `hbrq` 被补齐
- 如果其余参数齐全，则继续查价

## 风险与边界

1. 当前待澄清状态主要先服务于日期槽位 `hbrq`
2. 如果以后要支持“重量澄清”“包装二选一澄清”等，也建议复用这套模式
3. 这次修复依赖前端原样回传新增的：
   - `pending_clarify_slot`
   - `pending_clarify_message`

如果前端过滤掉它们，这条链路仍然会断

## 摘要

这次修复解决的不是“日期规则不会解析”，而是：

**系统会提出日期澄清问题，但之前不会稳定处理用户对澄清问题的短回复。**

通过“规则补强 + 待澄清状态”两条线一起修复后，系统现在不仅能问出：

- `这周六还是下周六？`

还能够在下一轮正确消费：

- `这周六`
- `下周六`

并继续回到运价补槽位链路。
