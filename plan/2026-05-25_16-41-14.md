## 一、需求 1：默认不返回全部数据，改为按包装类型返回最便宜方案

### 1.1 需求理解
当前完整询价成功后，系统默认把当前匹配到的全部报价结果渲染成 Markdown 表格返回。

新需求是：

- 默认不返回所有数据
- 默认只返回：
  - 包装类型为 `散货` 的最便宜 1 条
  - 包装类型为 `托盘` 的最便宜 1 条
- 最便宜的判断依据：`合计`
- 如果同一包装类型下最便宜有多条并列，随机给出 1 条即可
- 同时增加引导提示：
  - 如果客户需要全部数据，可以继续展开展示

### 1.2 我对这个需求的判断
这不是查询问题，也不是结果分析链问题，本质上是“默认结果展示策略”的变化。

也就是说：

- 接口仍然先查全量数据
- `latest_quote_result` 仍然应该保留全量标准结构
- 只是默认返回给用户的首屏内容不再是全部数据

这是正确方向，因为：

- 领导看演示时更关注“可读、可决策”
- 一上来返回十几条报价，信息密度过高
- 但后续结果分析链仍然依赖完整结果，所以后端不能只查两条

### 1.3 推荐方案
建议改造 [graph/nodes.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\graph\nodes.py) 的 `result_node` 默认输出逻辑：

1. 查询成功后仍保留完整 `quotes`
2. 在渲染默认首屏时，先按包装类型分组：
   - `散货`
   - `托盘`
3. 每组按 `price_total` 升序
4. 各取最便宜 1 条
5. 若某组最便宜并列多条：
   - 第一版可直接取排序后第一条
   - 或显式随机取一条
6. 若某组不存在：
   - 只展示存在的那组
7. 表格前加一段简短引导：
   - 示例口径：`先为您展示当前最有参考价值的两类包装最低价方案。如果您需要全部报价明细，我也可以继续为您展开。`

### 1.4 不建议的方案
不建议把“默认只返回散货 / 托盘最便宜”做成 Prompt 控制。

原因：

- 这属于确定性的结构化结果筛选
- 交给模型会增加输出不稳定性
- 很容易出现漏组、排序错位、价格对不上

### 1.5 影响范围
预计后续会改：

- [graph/nodes.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\graph\nodes.py)

原则上不需要改：

- [tools/air_freight.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\tools\air_freight.py)
- [graph/result_handlers.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\graph\result_handlers.py)

### 1.6 风险点
- 接口返回的包装类型可能存在：
  - `散货`
  - `托盘`
  - `散货/托盘`
  - `-1`
- 需要先统一包装类型归类口径，否则“散货/托盘”如何处理会产生歧义

建议第一版先明确：

- `散货` 只归到散货
- `托盘` 只归到托盘
- `散货/托盘` 暂不作为默认首屏候选，除非某一类缺失

## 二、需求 2：“那青岛呢”走兜底是错误的

### 2.1 现象
日志中这条链路非常清楚：

1. 用户先问：
   - `这票货从广州出发的话运费是多少？`
2. 系统进入参数复用确认：
   - `您是想沿用上一票的 500公斤、1.26个立方、2026-05-22 这个条件，查询 CAN 到 LAX 吗？`
3. 用户回复：
   - `是的`
4. 系统正常执行 `CAN -> LAX` 查询，但无结果
5. 用户继续问：
   - `那青岛呢？`
6. 系统直接走：
   - `intent=unknown -> fallback`

### 2.2 根因定位
这个问题卡在 `intent_node`，不是接口问题，也不是 `slot_node` 没抽出来。

从 [history.log](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\history.log) 可以看出：

- `那青岛呢？` 进入时，`context` 里其实还保留着：
  - `sfg=can`
  - `mdg=lax`
  - `inputWeight=500`
  - `inputVol=1.26`
  - `hbrq=2026-05-22`
  - `query_completed=True`
- 但 `intent_node` 最终判成了 `unknown`

根因不是一个，而是两个叠加：

1. 当前 follow-up 规则过于依赖关键词
   - `那青岛呢`
   - 这种“城市 + 呢”的短句没有命中现有 `FOLLOWUP_KEYWORDS`

2. 当前系统没有把“单城市替换起运港/目的港”的短句单独识别成强续问信号

### 2.3 我的判断
这类问题不能只补一个“青岛”关键词，否则后面还会有：

- `那广州呢`
- `深圳呢`
- `北京走呢`
- `那香港呢`

本质上要解决的是：

**在已有完整询价上下文下，用户只说一个城市短句，系统要把它理解为“替换当前航线中的起运港或目的港，再沿用核心参数继续询价”。**

### 2.4 推荐方案
建议在 [graph/nodes.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\graph\nodes.py) 增加一层“城市短回复 follow-up”规则，位置仍放在 `intent_node` 前置规则区。

规则建议：

1. 前提：
   - `query_completed=True`
   - 当前会话已有有效核心参数

2. 当前消息满足任一条件：
   - `那XX呢`
   - `XX呢`
   - `从XX出发呢`
   - `XX走呢`
   - `这票货从XX出发`

3. 且消息内至少识别出一个城市 / 机场候选

4. 则直接判为：
   - `intent=rate_query`

5. 后续由 `slot_node` 做：
   - 新航线识别
   - 参数复用确认或直接复用

### 2.5 这里还要补的一层
仅增强 `intent_node` 还不够，`slot_node` 也要能支持：

- 当前消息只有一个城市
- 默认推断它是在改起运港，还是改目的港

这块不能完全猜，需要基于当前上下文和中文话术位置判断：

- `从青岛出发` -> 起运港
- `到青岛` -> 目的港
- `那青岛呢` -> 第一版建议优先按“替换起运港”处理，如果当前上下文里目的港明显保留

这部分需要在实现前再定细口径，但第一版至少要把这类消息从 `fallback` 拉回 `rate_query`。

### 2.6 影响范围
预计后续会改：

- [graph/nodes.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\graph\nodes.py)
- 可能少量补 [graph/prompts.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\graph\prompts.py)

### 2.7 风险点
- `那青岛呢` 不带介词，本身有歧义：
  - 改起运港
  - 改目的港
  - 甚至可能在问经停

建议第一版按“默认改起运港”落地，并在文档中写清楚。后续如歧义率高，再引入定向追问。

## 三、需求 3：货代行业话术、模糊语义追问、直飞/中转定向输出

这部分实际上包含两个子问题，不能混着看。

### 3.1 子问题 A：货好日期不是航班日期

#### 3.1.1 现象
日志里这组调用链很明确：

1. 用户问：
   - `我有一票500KG,36件，1.26立方的货物要从上海空运到美国洛杉矶，能报一个直飞一个中转的运价吗？`
2. 系统缺航班日期，追问是对的
3. 用户回复：
   - `货5月22日准备好`
4. 系统直接把它当成：
   - `hbrq=2026-05-22`
5. 然后直接发起运价查询

这不符合业务语义。

#### 3.1.2 根因
当前系统对日期相关表达的识别，主要还是：

- 时间词标准化规则
- LLM 槽位抽取

但没有加“货代行业日期语义”的保护层。

也就是说，系统目前会把：

- `货5月22日准备好`

错误等同为：

- `5月22日飞`
- `5月22日航班`

#### 3.1.3 我的判断
这不是单纯补 Prompt 就能稳住的问题。  
要做“行业语义分层”。

我建议按三层处理：

##### 第一层：强确定词，直接落槽位
这类表达可以直接映射：

- `航班日期`
- `起飞日期`
- `走货日期`
- `哪天飞`
- `5月22日出运`

##### 第二层：行业模糊词，触发澄清
这类表达不能直接落到 `hbrq`：

- `货好日期`
- `货准备好`
- `ready date`
- `工厂出货日期`
- `预计备好`
- `货能在5月22日好`

这类应该追问：

- `请问您说的 5月22日 是货物备好日期，还是希望查询 5月22日 的航班价格？`

##### 第三层：完全模糊词，继续缺参提示
例如：

- `尽快安排`
- `越快越好`
- `最近`

这类继续走已有的时间澄清逻辑即可。

#### 3.1.4 推荐方案
建议新增一组“行业时间语义规则”，位置仍放在 [graph/nodes.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\graph\nodes.py) 的时间标准化和槽位提取前后层。

不是直接改 Prompt，而是：

1. 先识别“货好 / 备好 / ready”类表达
2. 如果命中，并且带日期
3. 不直接落 `hbrq`
4. 设置一个新的待澄清状态
5. 追问用户这是：
   - 货好日期
   - 还是航班日期

这套机制和现在的：

- `周六 -> 这周六还是下周六`

本质上是同一类“行业语义澄清”。

### 3.2 子问题 B：一个直飞一个中转，不应该把所有数据返回

#### 3.2.1 需求理解
用户说：

- `能报一个直飞一个中转的运价吗？`

这不是让系统：

- 只查直飞
- 或只查中转

而是让系统：

- 先查整批报价
- 再从原始结果中分别挑出：
  - 一个直飞方案
  - 一个中转方案

#### 3.2.2 我的判断
这不是 `flightType` 查询参数问题，而是“查询后结果定向输出”问题。

也就是说，这类请求不应该提前把 `flightType=直达/中转` 塞进查询工具。  
否则你查不出另一类数据。

正确做法应该是：

1. 按正常完整条件查一次全量结果
2. 查询结果出来后
3. 按 `route_type` 分组
4. 各组取最优一条
5. 按固定结构输出：

```text
直飞：
...

中转：
...
```

#### 3.2.3 推荐方案
建议把它视为“结果展示偏好”或“结果输出模式”，而不是查询参数。

可以在状态里临时增加一个很轻的字段，例如：

- `result_display_mode`

第一版可支持：

- `default`
- `direct_transit_pair`

触发条件：

- 用户原话中同时出现：
  - `直飞`
  - `中转`
  - `一个`
  - `分别`

查询仍走原 `tool_node`，只是在 `result_node` 根据 `result_display_mode` 走不同渲染策略。

#### 3.2.4 为什么不建议放到 result_analysis 链
因为这类问题发生在首次查询前，不是“查完后围绕结果继续分析”。  
所以更适合归到运价主链的结果渲染阶段，而不是 `result_analysis_node`。

  - 货好日期
  - 备好日期

## 四、需求 4：取消 `endDate`，不再做范围查询

### 4.1 现状
当前系统仍然存在明显的区间查询痕迹：

- 状态层：
  - `hbrqBegin`
  - `hbrqEnd`
- 工具层：
  - [tools/air_freight.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\tools\air_freight.py) 的 `AirFreightInput`
- 返回结果：
  - `similar_hbrqBegin`
  - `similar_hbrqEnd`
- 当前无精确数据时的“未来 7 天类似运价”
  - 也是靠 `hbrqBegin/hbrqEnd` 实现的

### 4.2 这是一个大改动
这不是简单删字段。

因为它会影响：

1. 前端 `context`
2. 状态模型
3. 工具 schema
4. 日期澄清逻辑
5. 结果渲染逻辑
6. 类似运价 fallback 策略

### 4.3 两种可选方案

#### 方案 A：对外取消区间字段，对内暂时保留类似运价的内部范围查询
做法：

1. 对用户和前端不再暴露：
   - `hbrqEnd`
   - 区间查询文案
2. 必填日期只保留：
   - 单一 `hbrq`
3. 无精确结果时，后端内部仍可继续用“未来 7 天”做隐藏式 fallback

优点：

- 改动小
- 不影响当前类似运价能力
- 风险最低

缺点：

- 概念上仍有内部区间查询存在

#### 方案 B：完全移除区间语义，包括内部 fallback
做法：

1. 全面删除：
   - `hbrqBegin`
   - `hbrqEnd`
2. 类似运价 fallback 改成：
   - 循环未来 7 天逐日单查
   - 或直接取消类似运价能力

优点：

- 语义彻底统一
- 系统更干净

缺点：

- 改动大
- 风险高
- 可能增加接口调用次数

### 4.4 我的建议
当前阶段更推荐：

- **先做方案 A**

原因：

- 当前系统已经跑通多轮逻辑
- 这轮同时还有 1、2、3 号需求
- 先把“对外字段口径”收掉，比一次性重做内部 fallback 更稳

如果领导后续明确要求：

- 连内部也不允许再用范围查询

再单独升级到方案 B。

## 五、整体实施计划

### 第一阶段：默认结果展示策略调整
目标：

- 首屏只展示：
  - 散货最低价
  - 托盘最低价
- 增加“如需全部报价可继续展开”的引导文案

原因：

- 改动小
- 风险低
- 演示价值高

### 第二阶段：修复“那青岛呢”这类承接式追问
目标：

- 把“城市短回复”从 `fallback` 拉回 `rate_query`
- 让它能进入已有的新航线复用链路

原因：

- 当前体验错误明显
- 业务上非常高频

### 第三阶段：补货代行业话术规则与定向追问
目标：

- 区分：
  - 航班日期
  - 货好日期
  - ready date
- 对模糊表达做定向澄清

原因：

- 这是当前系统最容易“语义看起来对，业务其实错”的区域

### 第四阶段：支持“一个直飞一个中转”的定向输出
目标：

- 查询全量
- 按结果后处理
- 分别输出 1 个直飞、1 个中转

### 第五阶段：收口 `endDate` / 区间查询字段
目标：

- 对外协议取消区间字段
- 内部实现按方案 A 或 B 收敛

## 八、风险与边界

1. “那青岛呢”这类短句天然有歧义
   - 第一版需要人为定义默认解释口径

2. 行业话术规则不可能一次做到 100%
   - 第一版必须优先覆盖高频高风险表达

3. `endDate` 下线是最大改动点
   - 不建议和前 3 个需求一次性混改上线

4. “默认只展示两条最低价”会改变用户认知
   - 必须同时补“可展开全部报价”的引导

---

## 九、当前建议
建议按以下顺序开发，不要并行乱改：

1. 默认最便宜首屏展示
2. “那青岛呢”承接式追问修复
3. 货代日期语义澄清 + 直飞/中转定向输出
4. `endDate` / 区间字段收口

这样能保证每轮改动都有明确收益，同时把回归风险控制在最小范围内。
