## 这份文档解决什么
这份文档用于回答一个更高层的问题：

为什么当前 AI 运价系统“能跑、能查、能追问”，但在真实演示中仍然显得像一套硬规则工作流，而不像一个具备自主理解和推理能力的 Agent。

目标不是否定当前架构，而是：

1. 明确当前系统已经做对了什么
2. 明确“傻”的根因不在某一个 bug，而在架构层的能力边界
3. 给出下一阶段从“工作流驱动”向“受控 Agent 化”演进的方案

## 摘要结论
当前系统本质上是一个：

**规则优先 + 节点固定 + 局部 LLM 提取** 的业务工作流系统。

它的优点是：

- 稳
- 可控
- 易调试
- 对运价工具调用安全

它的缺点是：

- 只有“预设过的场景”处理得好
- 一旦用户的问题超出规则覆盖面，系统就会误路由、误执行、或直接 fallback
- 缺乏“围绕当前上下文先做理解，再决定如何行动”的中间推理层

一句话总结：

**现在的系统不是没有 LLM，而是 LLM 只被当成分类器和槽位抽取器使用了，没有被当成“受控的任务理解器 / 小规划器”使用。**

## 一、当前系统为什么会显得“不智能”

## 1.1 当前架构本质
现在的主链路是：

```text
用户输入
 -> intent_node
 -> slot_node / result_analysis / fallback / rag
 -> tool_node
 -> result_node
```

特点是：

- 每一步的职责很明确
- 每一步的输出字段很固定
- 节点之间没有“反复思考 / 多轮计划 / 动态策略选择”

这是一种标准的 **deterministic workflow**，不是典型的 agentic loop。

## 1.2 为什么这条路一开始是对的
对运价系统来说，一开始做成工作流是正确的：

- 报价工具调用不能乱
- 槽位缺失必须可追踪
- SSE 协议不能乱
- 前端 `context` 必须稳定

所以前期：

- 不让模型“自由发挥”
- 用规则 + 状态保证最小闭环

这是正确工程选择，不是错误。

问题是现在系统进入第二阶段了：

- 用户开始追问更自然
- 场景开始超出既定规则
- 领导开始关注“像不像真人业务员”

这时原来那套“只要规则补一补就行”的方式开始触顶。

## 二、从日志看，当前“傻”的具体表现

## 2.1 例子一：`这是多少号的`
日志里这个问题很有代表性。

用户当前其实是在问：

- “你刚才给我的这批类似运价，是哪一天的？”

这不是新询价，也不是改航线，更不是补槽位。  
它本质上是：

- **对当前结果的指代性追问**

但日志里它被误打成：

- `city_followup_rate_query=true`
- 然后重新走了 `rate_query -> tool_node`

这说明当前系统缺少一层关键能力：

**对“这 / 那 / 上面这条 / 这批结果”这类指代性问题的上下文解释能力。**

也就是说系统没有真正理解：

- 用户是在问结果
- 不是在问查询条件

## 2.2 例子二：`是的`
日志里还有一个典型问题：

- 系统追问：`您是指起运港改成青岛，还是目的港改成青岛？`
- 用户回复：`是的`

然后系统判成了 `unknown`。

这里不是用户问错，而是系统没有做“追问类型约束”。

当前系统会问一个**二选一澄清问题**，但没有形成足够强的“期待回答形状”机制。  
所以用户只要没按剧本回答，就掉出链路。

这就是工作流系统的典型问题：

- 会问
- 但不会容错地理解回答

## 2.3 例子三：规则外问题直接掉兜底
你提到的核心感受是对的：

- 一旦用户问规则外问题
- 系统就“无能为力”
- 只能兜底或误路由

这不是单个规则漏写，而是当前系统没有一层：

- **先理解用户这一句到底在对“什么对象”说话**

对象可能是：

1. 当前询价条件
2. 当前结果集
3. 上一轮澄清问题
4. 某个结果字段
5. 某个日期
6. 某个包装类型

现在系统大多数时候只会判断：

- `rate_query`
- `result_analysis`
- `unknown`

这太粗了。

## 三、当前系统到底缺什么

## 3.1 不缺 LLM，缺“中间推理层”
当前系统不是没有模型，实际上模型已经参与：

- 意图识别
- 槽位抽取
- RAG analyzer

但这些都属于：

- 单步判定
- 单步提取

缺的是：

**在真正执行动作前，先做一层“用户到底在问什么 / 这句话依附于什么上下文 / 我下一步该做什么”的受控推理。**

## 3.2 不缺规则，缺“规则上方的策略层”
现在的问题也不是规则太少，而是：

- 规则已经很多了
- 但没有上层策略协调

结果就是：

- 每个新需求都在 `intent_node` / `slot_node` 里加一条 if
- 最后规则越来越多
- 彼此优先级冲突
- 误判开始增加

这就是典型的“规则堆叠后系统变脆”。

## 3.3 不缺状态，缺“会话对象模型”
当前状态里已经有很多字段：

- 询价槽位
- 结果分析态
- 待澄清态
- 参数复用态

但还缺一个更高层的抽象：

- 当前会话正在围绕哪个“对象”进行交流

比如：

- 当前正在补槽
- 当前正在确认航线角色
- 当前正在确认日期语义
- 当前正在分析上一批报价结果
- 当前正在询问某条结果里的日期

如果没有这个“对象层”，系统就只能靠关键词硬猜。

## 四、我的判断：不要把系统改成“全自由 Agent”

这里需要明确一个边界：

你现在的问题，不意味着应该把系统改成那种：

- LLM 自己决定要不要查价
- 自己决定调什么工具
- 自己决定什么时候结束

那样对运价系统是危险的。

### 为什么不能全自由
运价系统和通用聊天不同：

- 价格查询有工具调用成本
- 业务字段有强约束
- 前端 `context` 协议要稳定
- 错查一次就是错误业务结果

所以正确方向不是：

- 从“工作流”跳到“开放式 agent”

而是：

- 从“纯规则工作流”
- 升级成“**受控 Agent 工作流**”

也就是：

- 关键执行仍然受节点和状态约束
- 但在节点前增加一层小推理器 / 小规划器

## 五、推荐的演进方向：从纯工作流升级到“受控 Agent 工作流”

## 5.1 目标架构
建议未来演进成 4 层：

### 第一层：业务护栏层
这层保留现在的确定性能力：

- 必填字段
- 工具 schema
- SSE 协议
- 报价调用安全边界

这层不能放弃。

### 第二层：会话理解层
这是当前最缺的一层。

职责是先判断：

- 用户当前是在改查询条件
- 还是在问上一批结果
- 还是在回答系统上一次追问
- 还是在做字段指代追问

这层的输出不该直接是 `rate_query / unknown`，而应更细一些，例如：

- `slot_refill`
- `route_update`
- `result_reference_question`
- `clarify_answer`
- `result_filter_request`
- `new_quote_request`

### 第三层：受控执行层
这层仍然由 LangGraph 节点负责：

- `slot_node`
- `tool_node`
- `result_node`
- `result_analysis_node`

但它不再直接面对原始用户输入，而是面对“已理解过的用户意图对象”。

### 第四层：结果解释层
这层负责：

- 结果摘要
- 指代追问回答
- 针对当前结果的再解释

比如：

- `这是多少号的`
- `这个是直飞吗`
- `上面最便宜的是哪家`

这些不应该重新查工具，而应该从当前结果对象中解释出来。

## 六、具体到当前项目，建议怎么升级

## 6.1 第一步：新增“会话语义分类层”
不要再让 `intent_node` 同时承担：

- 新询价识别
- 追问识别
- 澄清回复识别
- 结果分析识别

建议把它拆成两层：

### 现有第一层
- `rate_query`
- `result_analysis`
- `rag`
- `unknown`

### 新增第二层
只在 `rate_query` 内进一步分：

- `new_quote`
- `slot_followup`
- `clarify_reply`
- `route_city_followup`
- `result_reference_question`

这样像：

- `这是多少号的`

就不会再被误塞进“城市短追问”。

## 6.2 第二步：引入“结果指代问答”能力
当前系统能做：

- 最便宜
- 直达有哪些
- 托盘有哪些

但还不能做：

- `这是多少号的`
- `这个是哪个航司`
- `上面这一条是直飞吗`

这类问题不需要重查，只需要：

- 从最新结果里定位目标对象
- 回答目标对象的某个字段

建议未来单独增加一类：

- `result_reference_node`

它只做：

- 指代解析
- 当前结果对象字段解释

## 6.3 第三步：把澄清机制做成统一框架
当前已经有三类澄清：

- 日期歧义
- 航线角色澄清
- 货好日期语义澄清

后面还会继续增加。

建议不要再一个问题加一套特例，而是统一抽象成：

- `clarify_type`
- `clarify_question`
- `clarify_expected_answers`
- `clarify_payload`

这样系统就会更像 Agent，而不是很多 `pending_xxx` 零散拼接。

## 6.4 第四步：让结果分析链支持“自然语言解释”，不只支持筛选
当前 [graph/result_handlers.py](D:\CompanyPlace\AIProject\AiFreightRate\freight-agent\graph\result_handlers.py) 偏结构化：

- filter
- group
- sort

后续应补一类：

- `field_explain`

用于回答：

- 这是多少号的
- 这是直飞还是中转
- 这个包装是什么

这类回答更接近“人理解结果再回复”，会明显提升智能感。

## 七、LangGraph / LangChain 视角下，当前系统缺少哪些知识点

## 7.1 当前已经用了什么
当前系统已经很好地用了这些 LangGraph 思路：

- 明确状态模型 `AgentState`
- 条件路由
- 工具节点与结果节点分离
- 多轮对话 context 回传
- RAG 与报价链分路

这些都没有问题。

## 7.2 当前缺少什么
从 LangGraph / agent 设计角度，当前缺少的是：

### 1. 中间语义状态
现在状态里大多是“业务槽位”，缺少：

- 当前用户问题类型的细粒度语义对象
- 当前回答依附的是哪一个上下文对象

### 2. Planner / Reasoner 层
不是让 LLM 自由规划多步工具，而是：

- 先用一个小规划器判断：
  - 该查工具
  - 该回答结果
  - 该继续澄清
  - 该解释上一条结果

### 3. 对象级记忆
当前记的是：

- 槽位值
- 最新结果集

后续更需要记的是：

- 当前活跃结果块
- 当前被指代的结果项
- 当前澄清问题的期望回答类型

### 4. 执行前反思
例如：

- 用户说的是不是“货好日期”
- 这句话是不是在问结果，不是在改航线

现在系统是在执行前直接路由，缺少一个 very cheap 的“反思一步”。

## 八、推荐的下一阶段计划

## 阶段 A：修正明显误判，不先大改架构
先补两类现象级问题：

1. `这是多少号的` 不应再落到 `city_followup_rate_query`
2. `是的` 回答澄清问题时，不应直接掉 `unknown`

这一阶段目标是止血。

## 阶段 B：引入“会话语义子类型”
把 `rate_query` 内部再细分。

目标：

- 不再只分“查价 / 不查价”
- 而是分“改条件 / 回答澄清 / 问结果 / 问字段”

这一阶段最关键。

## 阶段 C：增加“结果指代问答”能力
专门解决：

- 这是多少号的
- 这个是直飞吗
- 上面这一条是哪家

让系统真正开始表现出“看懂了当前结果”的能力。

## 阶段 D：统一澄清框架
把现在散落的：

- `pending_clarify_slot`
- `pending_clarify_context`
- `pending_reuse_confirmation`

逐步收敛成统一的会话澄清对象。

## 九、当前建议
如果只给一句结论：

**当前系统不是不够智能，而是现在的智能只被用在“分类和抽槽位”，还没被用在“理解这句话是在对哪一个对象说话”。**

所以后续正确方向不是把整个系统变成开放式 Agent，而是：

- 保留当前业务护栏
- 补一层“会话语义理解 + 小规划器”
- 让系统先理解，再执行

这样既不会把报价链路搞乱，又能明显减少“看起来很傻”的感觉。
