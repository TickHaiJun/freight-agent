## 这份文档解决什么
这份文档只覆盖两个当前已确认的问题：

1. 用户明确要求“只要一条数据”时，系统仍输出多条排序结果
2. 一轮询价完成后，用户输入闲聊短句如“你好”，系统误触发继续询价

这份文档不直接改代码，只输出实现方案和边界。

---

## 摘要结论

### 问题一：单条结果失效
根因已经明确：

- `result_analysis` 已经命中
- `analysis_intent` 也已经进入 `lowest / filter_list / summary`
- 但“只要一条”的约束没有稳定贯彻到最终渲染

所以本轮的重点不是再补更多“最便宜”规则，而是把：

- `quantity_mode = single | multi`

从语义识别层稳定传递到：

- 过滤
- 排序
- 截断
- 输出模板

### 问题二：`你好` 误触发询价
根因也已经明确：

- `intent_node` 中的 `_looks_like_city_followup(...)` 规则过宽
- 只要上下文仍在、消息很短，就有概率把闲聊短句也吸进 follow-up 询价链

所以本轮的重点不是继续增强 follow-up，而是先增加：

- 闲聊短句前置排除层
- 更严格的城市短追问判定

---

## 本轮总体策略

本轮继续保持当前架构原则：

1. 保留现有工作流骨架
2. 不改 SSE 协议
3. 不改前端协议
4. 不动 `tools/air_freight.py`
5. 只在中间层收口：
   - `graph/nodes.py`
   - `graph/result_handlers.py`
   - 必要时少量 `graph/state.py`

也就是说：

- 不去动报价接口
- 不去动前端交互
- 不去动 RAG

只修中间的语义层和结果层。

## 已确认的实现边界

### 边界 1：单条结果只基于最近一次完整报价结果
本轮仍然只基于：

- `latest_quote_result`

做单条结果输出，不支持：

- 跨多轮历史结果引用
- 上一票、上上票混合比较

这样最稳。

### 边界 2：闲聊短句不进入结果分析，也不进入继续询价
像下面这些第一版统一视为闲聊 / 非业务短句：

- `你好`
- `您好`
- `在吗`
- `谢谢`
- `好的`
- `收到`
- `嗯`

这类短句在有旧报价上下文时，也不应默认触发：

- `city_followup`
- `result_analysis`
- `result_reference`

### 边界 3：单条结果的输出样式优先“总结 + 单条表格”
当用户明确要一条时，默认输出模式不是：

- 纯总结
- 多条表格

而是：

- 一句简短总结
- 一条结果表格

这样既满足“只要一条”，又保留报价细节。

### 边界 4：本轮不处理“为什么问你好时该如何更自然回复”
本轮先解决“不误触发继续询价”。

也就是说：

- 第一版先保证它不要错进运价链
- 至于是走 `support_info`、`fallback` 还是更自然寒暄，后续再优化

当前最稳的收口是：

- 不触发运价 follow-up
- 落回普通 LLM 分类或 fallback

### 边界 5：`一条` 优先级高于 `lowest/filter_list/summary`
当消息中明确表达：

- `一条`
- `只要一条`
- `推荐一条`
- `来一条`

则数量约束优先于默认的列表型渲染。

例如：

- `给出航司为CK且直达的一条数据`

应该解释为：

- 条件筛选：`carrier=CK, route_type=direct`
- 结果数量：`single`

而不是：

- “筛出一组 CK 直达数据并按顺序全部输出”

---

## 本轮不处理的内容

为了控制风险，这一轮明确不处理：

1. 多窗口 / 多 `chat_id`
2. `reset_quote_context` 机制调整
3. `hbrqBegin / hbrqEnd` 下线
4. 结果引用跨多批历史结果
5. RAG
6. 前端按钮交互
7. 更深的行业语义层扩展

## 代码改动计划

## 阶段一：先修闲聊误触发

### 目标
确保一轮完整询价后，用户输入：

- `你好`
- `您好`
- `谢谢`
- `好的`
- `在吗`

不会再进入：

- `city_followup_rate_query`

### 改动点
主要修改：

- `graph/nodes.py`

### 计划做法

#### 1. 增加闲聊短句前置判定函数
建议新增：

```python
def _looks_like_small_talk(message: str) -> bool:
    ...
```

作用：

- 在 follow-up 规则前先拦掉明显闲聊

#### 2. 收紧 `_looks_like_city_followup(...)`
当前这条规则要从：

- “有上下文 + 消息短”

收紧成更明确的：

- 明确出现城市名 / 机场名 / 承接词
- 且不是常见闲聊短句
- 且不是结果解释类问题

#### 3. 在 `intent_node` 中调整优先级
建议顺序变成：

1. `support_info`
2. `result_reference`
3. `result_analysis`
4. `small_talk guard`
5. `city_followup`
6. `rate_followup`

这样能避免 `你好` 在太早阶段就被吸进城市追问链。

### 风险
- 如果闲聊短句集合写太宽，可能误伤极短业务问法
- 所以第一版建议只收极稳定的闲聊词，不做泛化

---

## 阶段二：稳定单条结果数量控制

### 目标
让下面这些问法稳定只返回一条：

- `给我最便宜的一条`
- `只要一条直飞的`
- `给出航司为CK且直达的一条数据`
- `推荐一条托盘的`

### 改动点
主要修改：

- `graph/result_handlers.py`
- 少量 `graph/nodes.py`

### 计划做法

#### 1. 统一数量模式识别
当前 `_detect_quantity_mode(...)` 已存在，但需要做两件事：

- 增强命中词
- 强化日志输出，确认它真正生效

建议后续日志至少打印：

- `analysis_intent`
- `quantity_mode`
- `response_mode`

#### 2. 把数量裁剪前移成统一步骤
当前渲染逻辑里已经有 `single_result` 分支，但还不够统一。

建议收敛成固定顺序：

1. 原始结果集
2. 条件过滤
3. 排序
4. 数量裁剪
5. 输出模式渲染

而不是不同 `analysis_intent` 各自散落处理。

#### 3. 明确 `single_result` 的统一输出函数
保持：

```python
_render_single_result_message(...)
```

作为唯一单条输出入口。

要求：

- 无论是 `lowest`
- 还是 `filter_list + single`
- 还是 `summary + single`

最终都走这一条渲染路径。

#### 4. 在 `analyze_result_request(...)` 中强化优先级
确保：

- 当 `quantity_mode == "single"` 且存在过滤条件时

优先解释成：

- “在筛选后的结果里选一条”

而不是：

- “筛选出一组”

### 风险
- 如果 `single` 与 `top_n` 同时出现，要定义优先级
- 第一版建议：
  - `single` 优先于 `top_n`

---

## 阶段三：统一输出模式

### 目标
避免系统在“明确只要一条”时还输出列表表格。

### 改动点
主要修改：

- `graph/result_handlers.py`

### 计划做法
把当前输出模式进一步收紧成：

- `summary_only`
- `summary_plus_table`
- `summary_plus_clarify`
- `single_result`

并规定：

- `quantity_mode == single` 时，默认输出 `single_result`
- 除非明确是澄清问题，才输出 `summary_plus_clarify`

### 风险
- 这一层如果和 `analysis_intent` 纠缠太深，会继续让逻辑变重
- 所以建议保持：
  - `analysis_intent` 决定“做什么”
  - `response_mode` 决定“怎么展示”
  - `quantity_mode` 决定“返回几条”

## 阶段四：补日志与回归观察点

### 目标
让后续调试不再靠猜。

### 改动点
主要修改：

- `graph/nodes.py`
- `graph/result_handlers.py`

增加关键日志：

- 是否命中 `city_followup`

这样下一轮如果还有问题，可以直接从日志判断：

- 语义没识别到
- 还是渲染没收口

## 我建议你确认的边界

下面这些边界如果提前确认，后续落代码会更稳：

### 1. 闲聊短句的目标行为
像：

- `你好`
- `您好`
- `谢谢`
- `在吗`

你希望第一版改完后：

- 统一走普通 fallback
还是
- 返回一个更自然的简短寒暄回复

我当前建议：

- 第一版先保证“不误触发询价”
- 输出先接受 `fallback / support_info / unknown` 体系，不新增寒暄模板

### 2. `single` 和 `top_n` 同时出现时的优先级
例如：

- `给我最便宜的一条 top3`

这类问法很少见，但边界上要定。

我建议：

- 第一版 `single` 优先

### 3. 单条结果默认展示形式
第一版我建议固定为：

- 一句总结
- 一条 Markdown table

而不是：

- 纯自然语言总结

因为报价细节还是需要用户看到。

### 4. 闲聊短句是否要清空活跃结果分析状态
当前建议：

- 不清空
- 只是不触发 follow-up

这样用户说完 `你好`，后面继续问报价结果相关问题仍可接上。

## 验证方案

### 场景一：闲聊不应误触发询价

前置：
- 完成一轮完整询价

验证问法：
- `你好`
- `您好`
- `谢谢`
- `好的`
- `在吗`

预期：
- 不命中 `city_followup_rate_query`
- 不重新进入运价查询主链

### 场景二：明确单条结果

前置：
- 已有一批完整报价结果

验证问法：
- `给我最便宜的一条`
- `只要一条直飞的`
- `给出航司为CK且直达的一条数据`
- `推荐一条托盘的`

预期：
- 最终只输出 1 条
- 不再输出完整排序列表

### 场景三：不影响多条结果

验证问法：
- `最便宜的有哪些`
- `展示全部数据`
- `直达的有哪些`

预期：
- 仍然按多条模式输出
- 不被单条逻辑误伤

## 最终结论
这轮实现的核心不是“再补几条规则”，而是把两层问题收口成可稳定维护的结构：

1. **闲聊误触发问题**
   - 在 `intent_node` 前半段补一个稳的闲聊排除层
   - 收紧 `city_followup` 规则

2. **单条结果失效问题**
   - 让 `quantity_mode` 真正贯穿：
     - 识别
     - 过滤
     - 排序
     - 截断
     - 输出

这两个点修完后，系统观感会比现在明显自然：

- 用户只要一条，就真的给一条
- 用户只是打个招呼，不会再被系统强行拉回报价工作流
