## 这份文档解决什么
这份文档只覆盖本轮已经明确确认的三类改造：

1. `packageType` 升级为正式必填参数，标准值只有：
   - `散货`
   - `托盘`
2. 日期行业语义澄清升级：
   - 遇到“货好日期 / 航班日期”这类模糊表达时尽量追问确认
   - 澄清后继续承接上一轮已识别的报价上下文，不重新从零补参
3. 报价域 follow-up 语义识别升级：
   - 保留现有规则
   - 对规则未覆盖的表达，增加 LLM 结构化补充识别
   - 重点解决“展示全部”这类结果分析表达误走 `unknown`

这份文档不直接改代码，只输出改造方案与边界。

## 已确认的业务决策

### 决策 1：包装类型进入报价必填
当前系统里的 `packageType` 已经存在于：

- `state`
- 槽位提取 Prompt
- 报价工具入参
- 结果标准化结构

但尚未进入真正的“必填控制流”。本轮改造后应明确：

- 用户未提供包装类型时，必须追问
- 包装类型只接受：
  - `散货`
  - `托盘`
- 结果输出中的“最低参考”必须以用户当前选择的包装类型为约束条件

### 决策 2：模糊日期语义优先追问
对如下表达，优先澄清，而不是直接猜测执行：

- `货好日期为明天`
- `货5月22日准备好`
- `按航班日期`
- `按货好日期`

澄清原则：

- 能明确追问时，优先追问
- 用户回答若是规则明确可识别的短语，按规则执行
- 用户回答若不是规则关键词，则再用 LLM 做结构化语义判断

### 决策 3：报价域 follow-up 采用“规则 + LLM”混合识别
当前很多 follow-up 语义仍然依赖硬编码规则，这会导致：

- 规则没覆盖时直接掉到 `unknown`
- 已有上下文却重新走全局三分类
- 结果分析与重新询价容易混淆

本轮确认采用：

- 规则优先
- 报价域内 LLM 结构化识别补盲区
- 不再让所有 follow-up 都先走全局 `rate_query / rag / unknown`

## 日志中已暴露的两个核心问题

## 问题一：日期语义澄清后丢失原始报价上下文
`History.log` 已明确表现出如下问题：

### 场景
用户先说：

- `上海到AMS的运价 100kg 1个立方 货好日期为明天`

系统触发了：

- `pending_action_type = hbrq_semantic`

随后用户回复：

- `按航班日期`

系统却又重新追问：

- `sfg`
- `mdg`
- `inputWeight`
- `inputVol`

### 根因
根因不是“规则太少”，而是状态机没有保住澄清前已识别的询价槽位。

也就是说，系统当前做到了：

- 识别出这是日期语义澄清问题

但没有做到：

- 在挂起 `pending_action` 前，把已经识别出的 `sfg/mdg/inputWeight/inputVol` 稳定写入当前上下文
- 在 `pending_action_payload` 中保存本轮已识别的报价槽位快照

结果就是：

- 用户第二轮只回答了“日期语义”
- 系统却把整票货当成一条新的不完整输入重新处理

### 本轮目标
本轮必须确保：

- 澄清前先固化已识别槽位
- `pending_action_payload` 除了候选日期，还要带本轮已识别的上下文快照
- 用户回复 `按航班日期 / 按货好日期` 后，只解决这一个歧义，不重问其他已识别参数

---

## 问题二：结果分析依赖硬编码词表，导致 `展示全部` 掉到兜底
日志中另一条非常典型：

- `全部数据` 命中 `result_analysis`
- `展示全部` 走 `unknown`

### 根因
当前系统的问题不是结果分析链不存在，而是：

1. 现有规则只覆盖了：
   - `全部数据`
   - `所有数据`
   - `全部报价`
   - `所有报价`
   - `全部明细`
   - `完整报价`
2. 当规则没有命中时，系统会退回到全局 intent 分类
3. 全局 intent Prompt 只有：
   - `rate_query`
   - `rag`
   - `unknown`

所以：

- `展示全部` 并不是没法理解
- 而是当前全局意图层根本没有 `result_analysis` 这个输出选项

### 本轮目标
本轮不再单纯继续补词表，而是要增加一个“报价域 follow-up 语义识别层”：

- 当前已有报价上下文时，不先走全局三分类
- 先判断它是不是：
  - 结果分析
  - 结果引用
  - 条件修改重查
  - 日期语义答复
  - 包装类型补充
  - 联系方式 / 能力说明
- 规则没命中时，再让 LLM 做结构化补充识别

## 本轮总体策略

本轮继续保持已有架构原则：

1. 不改 `/api/chat` 协议
2. 不改 SSE 返回事件结构
3. 不动前端 `context` 对接方式
4. 不推翻现有 LangGraph 主骨架
5. 优先在中间层收口：
   - `graph/nodes.py`
   - `graph/result_handlers.py`
   - `graph/state.py`
   - `graph/prompts.py`
   - `tools/air_freight.py` 只做最小必要配合

总体目标不是“增加更多散点规则”，而是把当前系统升级为：

- `状态机兜底`
- `规则优先`
- `LLM 补盲区`
- `动作执行与语义识别分层`

## 改造后的目标语义分层

## 第一层：对话阶段识别
不先看用户这句话字面像什么，而是先看当前处于哪个业务阶段：

- `quote_building`
  - 正在补齐询价参数
- `quote_clarifying`
  - 正在等待用户回答某个澄清问题
- `quote_completed`
  - 最近一次报价已完成，后续话术大概率属于结果分析 / 条件修改 / 结果解释
- `rag`
  - 知识库问答链
- `unknown`
  - 非业务内容

这一步优先依赖状态，不依赖大模型。

## 第二层：报价域内动作识别
当系统已经判断用户仍在报价域时，再进一步识别它属于哪种 follow-up：

- `slot_fill`
  - 补充缺失槽位
- `slot_update`
  - 修改已有槽位，例如改城市、改日期、改包装
- `date_semantic_reply`
  - 回答“航班日期还是货好日期”
- `result_analysis`
  - 查看全部、看最便宜、按航司分组、按包装筛选
- `result_reference`
  - 这是多少号的、这是哪个航司、这个包装是什么
- `support_info`
  - 问联系方式、问支持内容
- `clarify_needed`
  - 仍然不够明确，需要追问

这一层采用：

- 规则优先
- 规则不确定时再让 LLM 输出结构化标签

## 第三层：动作执行
语义识别完成后，再进入对应链路：

- 更新 state
- 清理 / 延续 pending action
- 重新查价
- 做结果分析
- 做结果解释
- 输出澄清问题

这一步不负责理解用户，只负责执行已确定动作。

## 代码改造计划

## 阶段一：把 `packageType` 提升为真正的必填报价槽位

### 目标
让包装类型从“可选条件”变成“正式必填业务参数”。

### 影响范围
- `graph/nodes.py`
- `graph/state.py`
- `graph/prompts.py`
- `tools/air_freight.py`
- `graph/result_handlers.py`

### 修改点
1. 修改必填字段定义：
   - `REQUIRED_RATE_FIELDS` 增加 `packageType`
2. 增加包装类型标准化：
   - 只接受 `散货 / 托盘`
   - 同义表达统一映射，例如：
     - `托`
     - `托盘货`
     - `板货`
     - `散`
     - `散货`
3. 补充追问逻辑：
   - 当前缺少包装类型时，追问：
     - `请问这票货的包装类型是散货还是托盘？`
4. 查询执行层：
   - 报价接口仍通过 `packageType` 传入
5. 结果层：
   - “最低参考”只能基于当前包装类型过滤后的结果
   - 即使接口返回混合结果，也应在结果层做一次防御性收口

### 风险点
- 历史上下文里已有未带 `packageType` 的旧会话，可能会触发新的补参
- 用户包装表达不规范时，需要一个稳定的标准化层，否则会过度追问

---

## 阶段二：修复日期语义澄清时的上下文丢失问题

### 目标
确保：

- 用户说“货好日期为明天”
- 系统追问“这是货好日期还是航班日期”
- 用户回“按航班日期”

之后系统只解决这一个歧义，不重问已经给过的起运港、目的港、重量、体积。

### 影响范围
- `graph/nodes.py`
- `graph/state.py`

### 修改点
1. 在触发 `hbrq_semantic` 澄清前：
   - 先把本轮已识别出的报价槽位写入 state
2. 扩展 `pending_action_payload`：
   - 除日期候选值外，再保存 `resolved_slots_snapshot`
3. 用户回复 `按航班日期 / 按货好日期` 时：
   - 从 `pending_action_payload` 恢复报价上下文
   - 只更新 `hbrq` 相关字段
   - 不清空已识别的其他槽位
4. 当用户回答不是规则关键词时：
   - 启动 LLM 语义判断
   - 结构化输出：
     - `flight_date`
     - `cargo_ready`
     - `unknown`
5. 若判定仍不明确：
   - 继续追问，不要执行模糊动作

### 风险点
- `pending_action` 和旧 `pending_clarify_*` 共存时，兼容层必须保持一致
- 不能因为恢复上下文而把上一轮旧值误带入全新询价

---

## 阶段三：新增“报价域 follow-up 语义识别器”

### 目标
不再让已有报价上下文下的 follow-up 全部退回到全局三分类。

### 影响范围
- `graph/nodes.py`
- `graph/prompts.py`

### 修改点
1. 在 `intent_node` 内增加前置分流：
   - 若当前有活跃报价上下文或待澄清动作
   - 优先进入“报价域 follow-up 识别”
2. 规则优先识别高确定性表达：
   - `展示全部 / 全部展开 / 全部列出来 / 把所有方案给我看`
   - `按航班日期 / 按货好日期`
   - `只看托盘 / 只看散货`
   - `最便宜的一条 / 全部数据 / 哪个航司 / 这是多少号的`
3. 当规则无法确定时：
   - 调用一个新的轻量 LLM Prompt
   - 输出结构化结果，例如：
```json
{
  "domain": "quote_followup",
  "intent": "result_analysis",
  "sub_intent": "all_list",
  "confidence": 0.84
}
```
4. 只有在 follow-up 识别器也无法处理时，才退回全局 intent 识别

### 这一步解决什么
重点解决：

- `展示全部` 不再掉 `unknown`
- `全部展开`
- `都列出来`
- `按航班日期`
- `按货好日期`
- `按包装看`

这类表达不再完全依赖关键词白名单。

### 风险点
- follow-up 识别器如果设计过宽，可能误把真正的新询价吃进去
- 所以必须先依赖“当前是否有活跃报价上下文”来决定是否进入该层

---

## 阶段四：结果分析链补充包装维度与更多自然表达

### 目标
在不破坏现有结果分析链的前提下，增强：

- 包装维度筛选
- 全量展示自然表达
- 包装约束下的最低参考

### 影响范围
- `graph/result_handlers.py`

### 修改点
1. `is_result_analysis_request(...)`
   - 保留现有高频关键词
   - 补强“展示 / 展开 / 列出来 / 给我看所有”一类表达
2. `analyze_result_request(...)`
   - 增加“包装类型”优先级
   - 当会话中已存在 `packageType` 时，默认沿用该约束
3. `render_result_analysis_message(...)`
   - 当用户已经指定包装类型时：
     - `lowest`
     - `summary`
     - `all_list`
     都只基于该包装类型集合输出

### 风险点
- 要避免“用户指定了包装类型，但结果展示又回到混合列表”
- 要避免“全量展示”时把包装维度过滤丢掉

---

## 阶段五：Prompt 升级但不改主协议

### 目标
给 LLM 一个更清晰的职责边界，让它做“域内语义补盲”，而不是直接替代状态机。

### 影响范围
- `graph/prompts.py`

### 修改点
1. 保留现有全局 intent Prompt
2. 新增一个“报价域 follow-up 识别 Prompt”
3. 新增一个“日期语义澄清回复识别 Prompt”
4. 明确要求：
   - 只输出结构化标签
   - 不直接生成客服回复
   - 不直接改写用户意图

### 原则
LLM 在本轮只做两类事：

1. 补规则盲区
2. 输出结构化语义标签

它不负责：

- 直接改 state
- 直接调用工具
- 直接生成最终动作

## 验证方案

## 验证一：包装类型必填
用例：

1. `上海到AMS 100kg 1个立方 明天`
   - 预期：追问包装类型，不直接查价
2. `托盘`
   - 预期：继续原询价，直接查价
3. `散货`
   - 预期：继续原询价，直接查价

## 验证二：日期语义澄清不中断上下文
用例：

1. `上海到AMS的运价 100kg 1个立方 货好日期为明天`
   - 预期：只追问日期语义，不追问其他槽位
2. `按航班日期`
   - 预期：继续沿用上海/AMS/100kg/1个立方，不重新追问
3. `按货好日期`
   - 预期：继续追问真正需要查询的航班日期，不丢失其余槽位

## 验证三：follow-up 自然表达不再掉兜底
用例：

1. 完成一轮询价后输入：
   - `展示全部`
   - `全部展开`
   - `把所有方案给我看`
   - `都列出来`
   - 预期：进入 `result_analysis -> all_list`
2. 完成一轮询价后输入：
   - `只看托盘`
   - `只看散货`
   - 预期：进入 `result_analysis`，并只展示对应包装

## 验证四：最低参考必须受包装约束
用例：

1. 先查一轮返回散货和托盘混合结果
2. 用户指定：
   - `只看托盘`
   - `给我最便宜的一条`
3. 预期：
   - 最低参考只能来自托盘结果集

## 实施顺序建议

建议严格按下面顺序做，避免边修边乱：

1. 先改 `packageType` 必填链路
2. 再修 `hbrq_semantic` 的上下文承接
3. 再加“报价域 follow-up 识别器”
4. 最后收口 `result_handlers` 的包装维度和自然表达

理由：

- `packageType` 是明确业务规则，先固化边界
- 日期语义上下文丢失是状态机 bug，优先级高
- follow-up 识别器是能力增强，要建立在状态机稳定之上
- 结果展示层应最后改，避免前面状态还不稳定时反复返工

---

## 一句话总结

本轮不是继续给现有规则打补丁，而是把当前 AI 运价系统从“散点规则驱动”升级成“状态机兜底 + 规则优先 + LLM 补盲区”的报价域语义系统。第一优先级是修复语义澄清时的上下文承接，第二优先级是把 `packageType` 提升为正式必填参数，第三优先级才是增强 `展示全部` 这类自然 follow-up 的识别能力。
