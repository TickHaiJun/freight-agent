# GitHub 管理、Docker 与部署方式建议

## 1. 目标

这份文档回答 4 个问题：

1. 当前项目适合怎样做 GitHub 管理
2. 云服务器部署时是否建议使用 Docker
3. 手动部署和自动化部署分别怎么理解、优缺点是什么
4. 结合你当前阶段，推荐采用什么组合方案

这次只做方案，不动代码。

---

## 2. 先给结论

基于当前 `freight-agent` 项目的形态，我的建议是：

1. 项目代码管理建议尽快统一到 GitHub
2. 云服务器部署建议分阶段做，不要一上来把复杂度拉满
3. 当前阶段更推荐：
   - `GitHub + Linux 服务器 + 手动部署`
   - 或者 `GitHub + Linux 服务器 + 半自动部署脚本`
4. Docker 不是不能上，但不建议作为你第一阶段的强依赖
5. GitHub Actions 这类 CI/CD 很值得做，但更适合在你把“单机部署、环境变量、日志、RAG 建库”跑顺之后再接

一句话收口：

**现在优先把 GitHub 管理和单机生产部署跑稳；Docker 与 CI/CD 作为第二阶段升级，不建议这周就全部上。**

## 3. 结合当前项目特征，为什么不能只泛泛谈部署

这个项目不是简单静态网站，也不是纯 CRUD 服务，它有几个明显特征：

1. `FastAPI + SSE` 流式输出
2. LangGraph 编排，多节点状态机
3. 运价查询依赖下游报价接口
4. RAG 依赖本地文档、Chroma、BM25 缓存
5. 日志后续要做平台化
6. 后面你明确还要跑 `4-5` 个 AI 项目

这意味着部署方案不只是“能跑起来”，还要考虑：

1. 环境一致性
2. 文档和索引资产管理
3. 日志目录规划
4. 回滚能力
5. 后续多项目共机治理

所以我的建议会更偏工程可持续，不只看首发成本。

## 4. GitHub 管理方式建议

## 4.1 我推荐你用 GitHub

我认为这个项目非常适合用 GitHub 管理，而且应该尽快统一。

原因：

1. 你现在已经开始进入“产品化”和“部署化”阶段，不再只是本地试验
2. 后续日志、RAG、前端平台、部署脚本都会逐渐增多
3. 你明确还有多个 AI 项目，后面统一管理方式会越来越重要

## 4.2 GitHub 对你当前项目的价值

### 第一层：代码版本管理

这是最基础但最重要的：

1. 知道每次改了什么
2. 知道哪个版本部署到了服务器
3. 出问题时能快速回滚

### 第二层：协作与变更记录

即使短期还是你自己为主，也有价值：

1. 后面前端、运维、同事参与时更顺
2. 可以通过 PR 留下需求背景和变更记录
3. 部署版本能和 issue / 文档 / 需求对应起来

### 第三层：自动化基础

后面不管是：

1. 自动测试
2. 自动构建
3. 自动打包
4. 自动部署

前提都需要先把代码托管标准化。

## 4.3 GitHub 管理的推荐做法

建议至少做到这些：

1. 一个项目一个独立仓库
2. `main` 分支只放可部署版本
3. 功能开发走短分支，例如：
   - `feature/logging-platform`
   - `feature/rag-docx-refresh`
4. 重要变更通过 PR 合并，即使只有你一个人，也建议保留合并记录
5. `.env`、`data/chroma`、`data/cache`、服务器日志目录不进仓库

## 4.4 仓库里应该放什么，不该放什么

### 应该进 GitHub 的

1. 业务代码
2. `requirements.txt`
3. 部署文档
4. 初始化脚本
5. RAG 的 metadata 配置
6. 前端日志平台相关的数据结构定义

### 不应该进 GitHub 的

1. `.env`
2. `data/chroma`
3. `data/cache/bm25.pkl`
4. 运行日志
5. 服务器本地上传目录
6. 本地虚拟环境

特别说明：

如果 `data/docs` 里是公司内部业务资料，是否进仓库要谨慎。  
我的建议是：

1. 如果资料敏感，不要直接进公共仓库
2. 如果 GitHub 是私有仓库，也仍然建议单独评估
3. 更稳妥的方式通常是“仓库放代码和 metadata，资料文件走服务器受控上传”

## 5. 云服务器是否使用 Docker

## 5.1 结论先说

当前阶段：

**Docker 不是必须，但未来值得上。**

更准确地说：

1. 第一阶段可以不用 Docker
2. 第二阶段建议上 Docker

## 5.2 现在不用 Docker 的理由

你当前项目虽然已经不小，但还处于“链路稳定优先”的阶段。

此时如果一上来加 Docker，会额外增加这些复杂度：

1. Dockerfile 维护
2. 镜像构建和调试
3. 容器里的 `.env` 管理
4. 容器里的 `data/docs`、`data/chroma`、`data/cache` 挂载
5. 日志目录映射
6. Windows 本地与 Linux 服务器之间的路径差异

而你当前真正更重要的是：

1. 运价链路稳定
2. RAG 建库流程稳定
3. 日志落盘稳定
4. 服务器目录规划清晰

也就是说：

**现在最大的风险不是“没有容器”，而是“业务链路和运维边界还没完全收敛”。**

## 5.3 未来上 Docker 的价值

当你后面要跑多个 AI 项目时，Docker 的价值会明显变大。

主要体现在：

1. 环境一致
2. 部署标准化
3. 版本切换更容易
4. 多项目隔离更好
5. 更容易接 GitHub Actions / Registry / 自动化部署

## 5.4 Docker 的优点

### 优点 1：环境一致性强

本地、测试、生产使用同一套镜像逻辑，更少出现：

- 本地能跑，服务器不能跑
- 依赖版本不同
- Python 环境不一致

### 优点 2：适合多项目共机

你后面要跑 `4-5` 个 AI 项目，这种情况下容器化会比“每个项目自己一套裸环境”更好管理。

### 优点 3：更适合自动化部署

后续如果你要做：

- GitHub Actions 构建镜像
- 推送镜像到阿里云镜像仓库
- 服务器拉镜像重启

Docker 会非常顺。

## 5.5 Docker 的缺点

### 缺点 1：初期复杂度更高

你现在不是不能学，而是没必要把第一阶段问题和第二阶段问题绑一起解决。

### 缺点 2：RAG 本地资产管理更麻烦

这个项目有：

1. `data/docs`
2. `data/chroma`
3. `data/cache`
4. 日志目录

这些都不是纯无状态 API 服务的思路。  
只要上 Docker，就要认真设计 volume 映射。

### 缺点 3：调试心智负担更高

首发阶段你可能还会经常：

1. 看实时日志
2. 临时修改配置
3. 重建 RAG
4. 手动验证文档

这时容器化会让排查路径多一层。

## 6. 手动部署 vs 自动化部署

## 6.1 手动部署是什么

这里的“手动部署”不是瞎传文件，而是有明确步骤的人工部署，例如：

1. 从 GitHub 拉代码
2. 激活虚拟环境
3. 安装依赖
4. 配置 `.env`
5. 必要时执行 `python -m scripts.rebuild_kb`
6. 启动或重启服务

这仍然是工程化部署，只是触发方式是人工。

## 6.2 自动化部署是什么

你理解的 GitHub CI/CD 基本是对的。

典型形式是：

1. 代码 push 到 GitHub
2. GitHub Actions 自动执行测试 / 构建
3. 自动把代码或镜像发布到服务器
4. 自动重启服务

有时还会附带：

1. 部署前检查
2. 健康检查
3. 自动回滚

---

## 7. 两种部署方式的详细对比

## 7.1 手动部署的优点

### 优点 1：最容易起步

你当前项目最适合先跑通的是手动部署，因为你还在不断确认：

1. RAG 文档形态
2. 日志目录与日志平台方向
3. 报价 API 的网络连通
4. 服务器资源规格

### 优点 2：问题定位更直接

出了问题你会更容易判断是：

1. 代码问题
2. 环境变量问题
3. 依赖问题
4. 文档路径问题
5. 日志权限问题

### 优点 3：适合首发和单机验证

对当前 `freight-agent` 这种“先把一个核心 Agent 稳定跑起来”的阶段，手动部署并不丢人，反而更务实。

## 7.2 手动部署的缺点

### 缺点 1：容易遗漏步骤

例如：

1. 忘了拉最新代码
2. 忘了装依赖
3. 忘了 rebuild RAG
4. 忘了改 `.env`
5. 忘了重启服务

### 缺点 2：可重复性差

时间长了以后，你会越来越依赖“我记得上次是这么弄的”。

### 缺点 3：后续多项目会越来越累

当同一台服务器跑多个 AI 项目时，纯手工维护成本会迅速上升。

---

## 7.3 自动化部署的优点

### 优点 1：可重复

每次部署的步骤是固定的，不依赖记忆。

### 优点 2：多人协作友好

前端、后端、运维或其他同事协作时，部署流程清晰很多。

### 优点 3：适合多项目统一治理

你后面明确还有多个 AI 项目，这时候 CI/CD 价值会快速上升。

### 优点 4：更容易标准化回滚

如果你后面按 tag、版本号或镜像 digest 部署，回滚会比手工覆盖轻松很多。

## 7.4 自动化部署的缺点

### 缺点 1：前期搭建成本高

它不是写一个 YAML 就结束，而是要一起解决：

1. 凭证管理
2. 服务器登录方式
3. 环境变量注入
4. RAG 文档与索引策略
5. 部署失败后的处置

### 缺点 2：你当前项目还不是完全无状态

只要还涉及：

1. 本地文档
2. 向量库
3. BM25 缓存
4. 日志目录

自动化部署就不能像纯前端站点那样简单。

### 缺点 3：如果基础流程没跑顺，会把问题藏起来

最怕的是你现在基础部署流程还没完全定型，就先上 CI/CD。  
这样将来一旦失败，排查路径会更绕。

## 8. 我对你当前阶段的推荐

## 8.1 第一阶段推荐方案

我推荐你当前采用：

### 方案 A：GitHub + Linux 服务器 + 手动部署

适用现在这个阶段。

为什么：

1. 复杂度最低
2. 对当前项目最稳
3. 最适合排查报价接口、RAG、日志这些基础问题

### 方案 B：GitHub + Linux 服务器 + 半自动部署脚本

如果你想比纯手动再进一步，我更推荐这个，而不是立刻全 CI/CD。

所谓半自动，是指：

1. 代码还是人工触发部署
2. 但服务器上有固定脚本，例如：
   - 更新代码
   - 安装依赖
   - 重启服务
   - 可选 rebuild RAG

这样你能得到：

1. 重复性更好
2. 风险仍然可控
3. 排查比 CI/CD 容易

## 8.2 第二阶段推荐方案

当你满足下面条件时，再上 Docker + CI/CD 更合适：

1. 单机部署已经稳定
2. RAG 建库与日志目录规则稳定
3. `.env` 和服务器目录策略不再频繁变化
4. 你要开始跑多个 AI 项目
5. 你希望减少人工部署

这时推荐升级到：

### 方案 C：GitHub + Docker + GitHub Actions + 服务器拉镜像部署

这是更长期的工程化方案。

## 10. Docker 和非 Docker 的推荐对比

### 当前阶段更推荐：非 Docker

原因：

1. 你更需要把部署链路理解透
2. 你项目还在快速变化
3. RAG 和日志的本地资产还在调整

### 中期更推荐：Docker

原因：

1. 多项目共机时更清晰
2. 配合 GitHub Actions 更舒服
3. 运维标准化更好

## 11. 最终建议

如果你现在让我替你拍板，我会给你这样的建议：

### 当前就做

1. 用 GitHub 托管项目
2. 用私有仓库
3. 服务器优先选 Linux
4. 先做手动部署或半自动部署
5. 暂时不把 Docker 和 CI/CD 绑在首发阶段

### 1 到 2 周后再考虑

1. Docker 化
2. GitHub Actions 自动化部署
3. 镜像仓库
4. 自动回滚

### 不建议当前就做的

1. 一上来 Docker + CI/CD + 多项目统一编排全上
2. 在基础部署链路还没稳定时做过早自动化

## 12. 风险点

### 风险 1

如果现在直接上 Docker + CI/CD，而基础部署规则还没稳定，问题会被放大，不会被解决。

### 风险 2

如果继续不用 GitHub，后面日志平台、前端项目、部署脚本一多，变更管理会迅速混乱。

### 风险 3

如果未来多个 AI 项目还都走纯裸机手动部署，运维成本会明显上升。

---

## 13. 建议你现在的动作顺序

1. 先把当前项目整理到 GitHub 私有仓库
2. 先按现有部署文档把 Linux 单机部署跑通
3. 先把日志目录、RAG 建库、服务启动方式定型
4. 再决定是否 Docker 化
5. 最后再接 GitHub Actions 自动化部署

---

## 14. 一句话总结

对你当前这个 `freight-agent` 项目，我推荐的顺序不是：

`GitHub -> Docker -> CI/CD -> 再想怎么部署`

而是：

`GitHub -> 先把 Linux 单机部署跑稳 -> 半自动化 -> Docker -> CI/CD`

这个顺序更务实，也更适合你当前这个 Agent 项目正在经历的阶段。
