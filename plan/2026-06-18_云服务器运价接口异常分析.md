# 云服务器运价接口异常分析

## 1. 问题现象

当前线上现象是：

- **RAG 问答正常**
- **AI 运价查询失败**
- 前端/用户侧看到的回复是：

```text
抱歉，运价查询系统暂时异常，请稍后重试。
```

根据你提供的服务器运行日志 `cloundHistory.log`，这个问题不是：

1. Docker 容器没起来
2. `/api/chat` 服务挂了
3. RAG 链路异常
4. 槽位没抽出来

而是 **运价工具调用外部报价接口时，真实收到了 `404 Not Found`**。

---

## 2. 已确认的调用链

从日志和当前代码可以确认，完整调用链如下：

```text
用户请求 /api/chat
-> intent = rate_query
-> slot_node 抽取完毕，query_ready = true
-> tool_node 调用 search_air_freight_rate
-> tools/air_freight.py 拼接报价接口 URL
-> GET https://www.wecanintl.com/freightprice//fee/api/airFreightFee/getUnitPrice
-> 远端返回 HTTP 404
-> air_freight 返回 success = false, error = HTTP_ERROR
-> graph/nodes.py 把 HTTP_ERROR 统一映射成：
   “抱歉，运价查询系统暂时异常，请稍后重试。”
```

也就是说：

> 现在的用户提示虽然像“系统异常”，但根因其实已经很具体：报价接口 URL 访问到了一个不存在的路径。

---

## 3. 日志里已经能证明的事实

日志里已经出现了这条关键信息：

```text
HTTP Request: GET https://www.wecanintl.com/freightprice//fee/api/airFreightFee/getUnitPrice?... "HTTP/1.1 404 Not Found"
```

以及：

```text
air_freight api http error | params=... | error=Client error '404 Not Found'
```

并且随后：

```text
tool_failed
error_type = HTTP_ERROR
error_message = 抱歉，运价查询系统暂时异常，请稍后重试。
```

说明：

1. 运价参数已经正常抽取完成
2. `tool_node` 已经正常进入
3. 外部 HTTP 请求真实发出去了
4. 失败不是超时，而是 **明确的 404**

这比“怀疑接口没部署”更进一步，因为现在可以明确说：

> 不是接口连不上，而是当前配置出来的 URL 路径不对，或者远端站点没有这条路由。

---

## 4. 根因判断

当前最可能的根因，不是单一一个点，而是下面两类中的一个或两个同时存在。

## 根因 A：`FREIGHT_API_BASE` 配置成了官网地址，但官网没有代理这条报价接口路径

当前代码里报价工具会这样拼 URL：

```python
url = f"{settings.freight_api_base}/fee/api/airFreightFee/getUnitPrice"
```

而日志里实际请求出来的是：

```text
https://www.wecanintl.com/freightprice//fee/api/airFreightFee/getUnitPrice
```

这说明服务器上的 `FREIGHT_API_BASE` 很可能被配成了类似：

```text
https://www.wecanintl.com/freightprice/
```

这样会导致最终 URL 变成：

```text
https://www.wecanintl.com/freightprice//fee/api/airFreightFee/getUnitPrice
```

如果官网 Nginx 或后端并没有把：

```text
/freightprice/fee/api/airFreightFee/getUnitPrice
```

这条路径转发到真实报价服务，就会直接返回 404。

### 这类问题的典型含义

不是运价服务“完全没部署”，而是：

- 官网站点已经存在
- 当前 `FREIGHT_API_BASE` 指到了官网域名
- 但官网域名下 **并没有配置报价服务路由**

---

## 根因 B：`FREIGHT_API_BASE` 尾部带了 `/`，导致拼出了双斜杠 `//`

当前日志中的 URL 有一段是：

```text
/freightprice//fee/api/...
```

这说明 `freight_api_base` 很可能尾部已经带了 `/`，而代码又固定拼了：

```python
... + "/fee/api/airFreightFee/getUnitPrice"
```

于是出现：

```text
.../freightprice//fee/api/...
```

### 需要注意

双斜杠 **不一定** 是 404 的唯一根因，因为有些网关会自动归一化路径，有些不会。

所以这里的判断是：

- **双斜杠是一个明显风险点**
- 但真正更大的问题仍然是：**当前基础地址对应的站点/网关未必真的提供了这条报价接口**

也就是说：

> 双斜杠是次级问题，错误的 base 路由或缺失的服务代理更像主因。

---

## 5. 现阶段最合理的结论

结合日志、代码和现象，我当前的结论是：

### 结论 1：Agent 本身没坏

因为：

- intent 正常识别为 `rate_query`
- slot 正常抽取
- tool 正常进入
- 错误被统一兜底返回

说明 Agent 主链路是通的。

### 结论 2：问题出在“运价外部接口访问配置”这一层

也就是：

- `FREIGHT_API_BASE` 配置值
- 官网域名网关/Nginx 路由
- 报价服务是否挂在该路径之下

### 结论 3：当前更像“外部报价服务地址配置错误 / 路由未部署”，不是 RAG、不是 Docker、不是 SSE 问题

---

## 6. 服务器上建议立即验证的内容

下面这些检查，不需要改代码，优先做。

## 第一步：看容器里实际生效的 `FREIGHT_API_BASE`

在服务器执行：

```bash
cd /data/apps/freight-agent
docker compose exec freight-agent-app printenv | grep FREIGHT_API_BASE
```

你需要确认它到底是不是：

```text
https://www.wecanintl.com/freightprice/
```

或者其他值。

如果它是官网地址，那问题基本就落在“官网路由是否代理报价服务”这一层。

---

## 第二步：在容器里直接 curl 真实报价 URL

执行：

```bash
docker compose exec freight-agent-app curl -i "https://www.wecanintl.com/freightprice//fee/api/airFreightFee/getUnitPrice?sfg=pvg&mdg=lax&inputWeight=500&inputVol=3&hbrq=2026-06-19&packageType=%E6%95%A3%E8%B4%A7&cargoType=%E6%99%AE%E8%B4%A7"
```

如果这里仍然是 `404`，说明问题不在 Agent，而在远端服务地址或网关路由。

---

## 第三步：分别验证 3 种可能的基础路径

你现在最该确认的是，真实报价服务到底暴露在哪个 base 下。

建议在服务器里分别试：

### 方案 1

```bash
curl -i "https://www.wecanintl.com/fee/api/airFreightFee/getUnitPrice?sfg=pvg&mdg=lax&inputWeight=500&inputVol=3"
```

### 方案 2

```bash
curl -i "https://www.wecanintl.com/freightprice/fee/api/airFreightFee/getUnitPrice?sfg=pvg&mdg=lax&inputWeight=500&inputVol=3"
```

### 方案 3

如果你们原始内部服务本来就是：

```text
http://192.168.0.186:9000
```

那就要确认：

```bash
curl -i "http://192.168.0.186:9000/fee/api/airFreightFee/getUnitPrice?sfg=pvg&mdg=lax&inputWeight=500&inputVol=3"
```

如果这个能通，而官网域名那条不通，就说明：

> 报价服务本身可能存在，但没有被官网域名正确代理出来。

---

## 7. 两种主要修复方向

## 方向 A：修服务器配置，不改代码

这是当前最优先建议的方向。

### 适用场景

如果你们真实报价服务已经存在，只是 `FREIGHT_API_BASE` 配错了，或者官网没有把报价服务代理出来。

### 处理方式

1. 确认真实报价服务地址
2. 把服务器 `deploy/.env.production` 里的 `FREIGHT_API_BASE` 改成正确值
3. 重新创建应用容器，让环境变量生效

例如：

```bash
docker compose up -d --force-recreate freight-agent-app
```

### 这是当前最可能的一步到位修复

因为日志已经表明：Agent 能正常走到工具调用层，真正错的是它访问的地址。

---

## 方向 B：后续补代码防御，但不是当前首要动作

即使最终根因在配置，代码层后续仍然建议补两个防御点。

### 防御点 1：拼接 URL 前做 base 标准化

例如把：

```python
settings.freight_api_base
```

先做：

```python
settings.freight_api_base.rstrip("/")
```

避免最终生成双斜杠路径。

### 防御点 2：把 404 错误打得更明确

当前前端看到的是统一文案：

```text
抱歉，运价查询系统暂时异常，请稍后重试。
```

这对用户可以，但对运维排查不够直观。

后续可以考虑在日志里继续保留：

- base URL
- 最终请求 URL
- HTTP 状态码

这个你现在日志里其实已经打出来了，这点是对的。

---

## 8. 当前优先级建议

当前不建议先改 Python 代码，也不建议先怀疑 Docker。

建议优先顺序如下：

1. **先确认容器内 `FREIGHT_API_BASE` 实际值**
2. **直接 curl 当前日志里的真实 URL**
3. **验证报价服务真实暴露路径到底是什么**
4. **修正服务器 `FREIGHT_API_BASE` 或官网代理路由**
5. **重新启动应用容器**
6. **再测一条运价查询**

---

## 9. 一句话结论

这次线上“运价查询系统暂时异常”的根因，已经可以明确定位为：

> **运价外部接口请求真实返回了 404，问题在报价服务地址配置或官网代理路由，不在 RAG、不在 SSE、不在 Agent 主链路。**

当前最该查的是：

- `FREIGHT_API_BASE` 服务器实际值
- 官网域名是否真的代理了 `/freightprice/fee/api/airFreightFee/getUnitPrice`
- 是否应该改回真实报价服务地址，而不是当前官网地址
