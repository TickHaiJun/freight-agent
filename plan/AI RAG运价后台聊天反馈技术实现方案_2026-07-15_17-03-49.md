# AI RAG 运价后台聊天反馈技术实现方案

> 方案时间：2026-07-15 17:03:49  
> 适用项目：`freight-agent`（AI RAG 运价后台）  
> 实施状态：仅技术设计，当前不修改代码

## 1. 目标

在不改变既有 `/api/chat` SSE 事件类型、请求结构和运价/RAG 调用链的前提下，增加独立的聊天反馈闭环：前端可对某一轮回答提交不满意反馈；后台先可靠保存原始反馈，再进行可失败的 AI 归因；Next 日志监控平台随后读取并聚合这些反馈事件。

本期不引入数据库、消息队列或新第三方依赖。复用现有 FastAPI、Pydantic、DeepSeek `ChatOpenAI`、`Settings`、统一 `log_event` 和 JSONL 日志设计。

## 2. 已确认的现状与改造切入点

当前调用链如下：

```text
POST /api/chat
  → main.chat()
  → 生成 request_id（build_request_id）
  → agent.invoke(initial_state)
  → 逐字 text SSE
  → context SSE（_build_context_data）
  → done SSE
  → request_started / agent_finished / request_completed JSONL 日志
```

现有实现中：

- `request_id` 已在 `main.py` 内生成，并记录在应用 JSONL 日志中；
- `_build_context_data()` 是对前端回传上下文的唯一集中位置；
- `logging_schema.py` 已提供 `log_event()`、`summarize_retrieved_docs()`、`summarize_latest_quote_result()`；
- `config.py` 的 `Settings` 是配置唯一入口；
- `rag/generator.py` 已使用 `ChatOpenAI(settings.deepseek_*)` 调用 DeepSeek，可复用同一模型接入方式；
- 当前 `SESSION_RUNTIME_STORE` 只保存最近报价结果，不能作为请求级历史追溯来源。

因此最小闭环不修改 LangGraph 节点和 RAG 模块，不向 `AgentState` 新增反馈字段。反馈模块从 `/api/chat` 的 `context` 获取请求关联，从既有运行日志抽取链路摘要。

## 3. 总体架构与调用链

```text
原生 JS 前端
  ├─ POST /api/chat
  │   └─ SSE context.context.request_id（新增可选字段）
  └─ POST /api/chat-feedback
       ↓
FastAPI FeedbackRouter/Endpoint
  → Pydantic 校验与长度限制
  → 基础脱敏
  → 根据 request_id 查当日应用 JSONL，构建 trace_snapshot
  → FeedbackStore.append(feedback: pending)
  → FeedbackAnalyzer（DeepSeek，8 秒超时）
  → FeedbackStore.append(feedback_enrichment: completed/failed)
  → 返回 accepted + AI 状态
       ↓
data/feedback/chat-feedback.jsonl
       ↓
Next 日志监控平台服务端读取并按 feedback_id 聚合
```

关键可靠性原则：`feedback` 原始事件必须在模型调用前追加成功。AI 分析是增强步骤，任何超时、异常或格式问题都不能回滚原始反馈。

## 4. 接口与兼容策略

### 4.1 `/api/chat` 的最小改动

修改 `_build_context_data(final_state, request_id)` 或在生成 `context` SSE 时对其结果补充 `request_id`。推荐前者，确保字段构造集中：

```python
def _build_context_data(final_state: AgentState, request_id: str | None = None) -> dict:
    return {
        # 保留全部既有字段
        ...,
        "request_id": request_id,
    }
```

生成 SSE 时传入当前生成器局部变量 `request_id`。只新增 `context.context.request_id`，不改动 `text`、`context`、`done`、`error` 的事件名称、顺序和既有字段含义；旧前端可直接忽略。

### 4.2 `POST /api/chat-feedback`

新增普通 JSON 接口，不使用 SSE：

```json
{
  "session_id": "session_xxx",
  "request_id": "req_xxx",
  "feedback_text": "我问的是含电池货物，回答却按普货给建议。",
  "dissatisfaction_types": ["misunderstood_question", "incomplete_answer"],
  "user_question": "锂电池货物怎么订舱？",
  "assistant_answer": "……",
  "conversation_excerpt": [],
  "allow_context_for_review": true
}
```

返回：

```json
{
  "feedback_id": "fb_<uuid>",
  "status": "accepted",
  "ai_analysis_status": "completed"
}
```

`request_id` 在模型层可定义为可空以兼容尚未升级的前端，但记录 `trace_found=false`；不得以“查不到请求日志”为理由拒绝有效反馈。

接口请求模型不定义、也不接受 `page_url`、`client_version`、`client_feedback_id`。使用 Pydantic 默认的 `extra="forbid"`，让误传字段得到明确 422，而非被静默保存。

## 5. 建议文件划分与职责

| 文件 | 职责 | 主要内容 |
|---|---|---|
| `main.py` | API 接入 | `request_id` 回传、注册/定义反馈 POST 端点。 |
| `config.py` | 配置唯一入口 | 增加反馈路径、长度、开关、模型超时和保留期配置。 |
| `feedback/__init__.py` | 包导出 | 仅暴露服务入口及必要模型。 |
| `feedback/models.py` | 数据契约 | 请求模型、枚举、JSONL 事件模型、AI 分析模型、响应模型。 |
| `feedback/store.py` | 存储层 | 路径解析、追加写入、单进程锁、逐行读取、按 request_id 查日志。 |
| `feedback/sanitize.py` | 数据安全 | 长度截断和基础敏感信息掩码。 |
| `feedback/service.py` | 编排层 | 生成 ID/时间、链路摘要、原始落盘、AI 分析、增强落盘。 |
| `feedback/prompts.py` | Prompt 集中管理 | AI 归因系统提示词和用户模板。 |
| `tests/test_chat_feedback.py` | 回归与可靠性 | 见第 10 节。 |

第一版推荐在 `main.py` 直接定义端点，避免为单一接口提前引入 router 分层；当后台接口数量增长后再迁移 `feedback/router.py`。不修改 `graph/`、`rag/` 业务节点、`tools/` 和现有日志 formatter。

## 6. 数据模型与存储设计

### 6.1 请求模型

`ChatFeedbackRequest` 使用严格 Pydantic 校验：

- `session_id`：去空白后非空，最大 128；
- `request_id`：可空，最大 128；
- `feedback_text`：5～配置上限，去首尾空白；
- `dissatisfaction_types`：非空列表、去重，元素为固定枚举；
- `user_question`：1～2,000；
- `assistant_answer`：1～配置上限；
- `conversation_excerpt`：可选 `list[str]`，限制轮数和总长度；
- `allow_context_for_review`：默认 `true`。

当 `allow_context_for_review=false` 时，仍可保存用户反馈与关联键，但 `conversation_snapshot` 不写问题、答案和历史摘录，或仅写固定的“用户未授权保存上下文”标记；该行为需由前后端共同固定。

### 6.2 JSONL 事件

文件路径由 `CHAT_FEEDBACK_DIR` 与 `CHAT_FEEDBACK_FILE_PREFIX` 组合，例如 `data/feedback/chat-feedback.jsonl`。每一行是完整 UTF-8 JSON，对象必须带 `schema_version=1` 和 `record_type`。

原始事件：

```json
{
  "record_type": "feedback",
  "schema_version": 1,
  "feedback_id": "fb_...",
  "created_at": "2026-07-15T17:00:00+08:00",
  "source": "web_chat",
  "session_id": "session_xxx",
  "request_id": "req_xxx",
  "user_feedback": {"dissatisfaction_types": ["..."], "feedback_text": "..."},
  "conversation_snapshot": {"user_question": "...", "assistant_answer": "...", "conversation_excerpt": []},
  "trace_snapshot": {"trace_found": true, "intent": "rag"},
  "ai_analysis": {"status": "pending"},
  "workflow": {"status": "new", "owner": null}
}
```

增强事件只包含与 AI 分析相关的增量字段和同一 `feedback_id`；人工操作后追加 `feedback_update`。禁止修改或重写旧行，使 Next 平台可按事件顺序稳定聚合。

### 6.3 写入一致性与读取容错

`FeedbackStore.append_event()` 在单进程内使用 `threading.Lock`，通过 `open(..., "a", encoding="utf-8")` 写入一行、换行、`flush()`；生产需要强持久保证时按配置再 `os.fsync()`。当前项目只有单服务实例，第一版不处理跨容器锁；部署扩展至多 worker/多实例前，必须采用 Linux `flock` 或按实例/日期分片，否则可能出现行交叉。

`read_events()` 逐行解析，坏行只记录警告事件并跳过，不使反馈接口或 Next 读取整体失败。按日轮转与 `CHAT_FEEDBACK_RETENTION_DAYS` 清理在后续运维任务实现，不放入本接口首版。

## 7. 链路摘要提取

新增 `FeedbackStore.find_request_trace(request_id)`，优先读取 `settings.app_log_dir / f"{prefix}-app.jsonl"` 当日文件。匹配 `request_id` 后合并：

| 日志事件 | 可提取字段 |
|---|---|
| `request_started` | 请求路径、重置上下文标记、提交时间。 |
| `agent_finished` | `intent`、`query_ready`、待补槽位、RAG 检索条件与命中文档数。 |
| `request_completed` | 总耗时、展示模式、报价摘要、检索摘要。 |
| `request_failed` | `error_stage`、`error_type`、错误摘要、耗时。 |

产出的是轻量扁平快照，保留 `trace_found`，不复制 `message_text`、完整回答、堆栈或整条日志，以避免敏感信息扩散和反馈文件膨胀。当前 `main.py` 的 `request_completed` 缺少 `intent`，实施时可补充该日志字段；这是增强追溯质量的局部日志改动，不影响聊天协议。

## 8. AI 分析设计

`FeedbackAnalyzer.analyze()` 复用 `ChatOpenAI`，但独立于 RAG 回答 Prompt，放在 `feedback/prompts.py`。建议用 `asyncio.to_thread()` 包裹同步 `.invoke()`，并用 `asyncio.wait_for(..., timeout=settings.chat_feedback_ai_timeout_seconds)` 限时，避免阻塞 FastAPI 事件循环。

模型仅接收脱敏后的反馈、允许保存时的快照、轻量链路摘要及严格 JSON Schema。输出字段：

- `summary`；
- `quality_tags`；
- `business_domain`；
- `pipeline_stage`；
- `root_cause_hypothesis`；
- `severity`；
- `confidence`；
- `recommended_action`；
- `needs_human_review`。

利用 Pydantic 校验模型输出；非 JSON、未知枚举、空内容、超时均转为 `status=failed`，写入安全的 `error_type`/摘要。根因提示词要求使用“可能、需复核”，不把用户主观不满判断成系统事实。首版在原始落盘后同步等待最多 8 秒；响应超时即返回 `accepted/pending`，后续再考虑后台任务或重分析接口。

## 9. 配置与日志变更

在 `Settings` 增加以下字段并提供兼容默认值：

```env
CHAT_FEEDBACK_ENABLED=true
CHAT_FEEDBACK_DIR=./data/feedback
CHAT_FEEDBACK_FILE_PREFIX=chat-feedback
CHAT_FEEDBACK_MAX_TEXT_LENGTH=1000
CHAT_FEEDBACK_MAX_ANSWER_LENGTH=6000
CHAT_FEEDBACK_AI_ENABLED=true
CHAT_FEEDBACK_AI_TIMEOUT_SECONDS=8
CHAT_FEEDBACK_RETENTION_DAYS=180
```

新增应用事件：

- `chat_feedback_received`：只记录 `feedback_id`、`session_id`、`request_id`、类型数量和关联状态；
- `chat_feedback_ai_completed`：记录 `feedback_id`、业务域、阶段、严重度、置信度；
- `chat_feedback_ai_failed`：记录 `feedback_id` 和安全错误摘要。

不可将 `feedback_text`、问答全文或模型原始输出写入应用运行日志。当前 `logging_schema.sanitize_event_fields()` 会截断文本，但这不是保存敏感原文的授权依据。

## 10. 测试与验收

新增 `tests/test_chat_feedback.py`，采用临时反馈目录和 mock AI 调用，覆盖：

1. 正常提交：返回 `accepted`，产生可解析 `feedback` 与 `feedback_enrichment` 两行；
2. 非法枚举、空文本、超长文本、误传三个排除字段：返回 422；
3. `request_id` 命中/未命中运行日志时，分别写入正确的 `trace_found`；
4. AI 超时、异常、非法 JSON：原始反馈仍存在，增强状态为 `failed` 或响应为 `pending`；
5. `allow_context_for_review=false`：不保存未经授权的问答快照；
6. 单进程并发提交：每一行均可独立 JSON 解析；
7. `/api/chat` 回归：SSE 仍依次输出既有事件，且 `context` 中新增可选 `request_id`；
8. RAG 问答、运价追问、报价结果分析等现有场景回归，确认反馈模块未参与其业务分支。

手工验证还需使用真实 DeepSeek 凭据测试 AI 结构化输出、8 秒超时与日志文件权限。

## 11. 影响范围

### 直接影响

| 范围 | 影响 | 兼容性 |
|---|---|---|
| `main.py` | 新端点；context 新增 `request_id`。 | SSE 事件与既有字段不变，向后兼容。 |
| `config.py` / `.env` | 增加反馈配置。 | 默认值保证未配置时本地可运行；生产需显式指定受控目录。 |
| `data/feedback/` 或生产日志目录 | 新增反馈 JSONL 文件。 | 与原 app 日志隔离。 |
| `logging_schema.py` | 增加反馈应用事件；可补 `request_completed.intent`。 | 仅增加日志字段。 |
| 测试 | 新增反馈测试并扩展 chat 回归。 | 不改变既有用例意图。 |

### 间接影响

- 原生 JS 前端需读取 `context.request_id`、缓存该轮问答并调用新接口，但其实现不属于本后台改动。
- Next 平台需按 `feedback_id` 聚合反馈事件，并以服务端身份读取受控路径。
- 运维需创建反馈目录、设置权限、配置备份/归档；多实例前须完成写入方案升级。
- DeepSeek 调用量与接口响应时间会因同步 AI 分析增加；通过开关与 8 秒超时控制，且不影响 `/api/chat`。

### 明确不受影响

- LangGraph 意图识别、槽位追问、运价工具调用、RAG 检索与 RAG 生成逻辑；
- `/api/chat` 的 URL、请求模型、SSE 事件名称及前端现有 context 字段；
- 现有报价结果内存态和运行日志文件格式（只会新增字段/事件）。

## 12. 实施顺序、风险与决策点

建议按以下顺序实施：

1. 建立 `feedback` 模型、脱敏与 JSONL Store，并完成纯单元测试；
2. 在 `main.py` 增加 request_id 回传和 feedback 端点，完成 API 测试；
3. 接入运行日志摘要查询与反馈应用事件；
4. 接入 AI 归因与超时/失败处理；
5. 与原生 JS 前端联调；
6. 由 Next 平台读取真实 JSONL 样本进行聚合验证。

主要风险：

- **AI 调用失败或变慢**：先落盘、超时转换 pending/failed，永不阻塞 `/api/chat`。
- **未经授权的上下文留存**：以 `allow_context_for_review` 决定是否写入问答快照，并先脱敏。
- **日志文件格式/轮转差异**：摘要查询只依赖稳定事件和字段，查不到则降级。
- **多实例并发写入**：当前单实例可用进程锁；扩容前必须增加跨进程/跨节点方案或迁库。
- **重复反馈**：用户要求不使用 `client_feedback_id`，本期不承诺重试幂等；前端提交态是唯一防重复手段。

需要在编码前最终确认的唯一产品规则是：用户关闭“允许带上本轮问答内容用于排查”后，是完全不保存问答快照，还是允许保存脱敏摘要。本文默认采用更保守的“完全不保存问答快照”。
