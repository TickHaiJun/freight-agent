# AI 反馈日志平台可视化方案

## 1. 文档目标与边界

### 1.1 目标

基于 `log/chat-feedback.mock-100.jsonl` 建设一个面向业务、运营和研发共同使用的 AI 反馈日志平台。平台的核心不是“展示很多图”，而是让不同角色在几分钟内回答四个问题：
UI风格类似 以 `asstes/目录下的图片为风格`


1. 用户最近主要不满意什么？
2. 问题发生在哪个 AI 链路阶段？
3. 哪些问题最需要优先处理，处理后是否改善？
4. 能否从某个问题一路追到原始反馈、对话、链路摘要和处理状态？

技术前提：前端采用 **Next.js 16.2.9 + shadcn + tailwindcss-animate + tailwindcss + ECharts**；Next.js 同时提供 BFF（Backend for Frontend）接口；当前阶段**不连接数据库**，直接读取 JSONL 模拟日志文件。
这个日志文件可以配置路径，例如服务器指定目录 或者当前项目目录， 例如当前面项目log 下存放JSONL 文件。

### 1.2 当前数据事实

模拟数据一共 100 行事件，构成为 50 组反馈闭环：

| 事件类型 | 数量 | 用途 |
| --- | ---: | --- |
| `feedback` | 50 | 用户提交的原始反馈、授权状态、对话片段、链路摘要 |
| `feedback_enrichment` | 50 | AI 自动归因；44 条完成、6 条模拟失败 |

当前样本中的高频反馈类型包括：`slow_response`、`quote_result_issue`、`incomplete_answer`、`misunderstood_question`、`display_issue` 与 `other`。AI 归因标签主要集中在 `missing_information`、`tool_or_data_issue`、`knowledge_gap`、`display_issue`、`latency`、`misunderstanding`。

### 1.3 不在本期范围

- 不修改现有 `/api/chat` SSE 协议与运价查询链路。
- 不接入数据库、登录权限、真实工单系统或实时消息队列。
- 不把 AI 分析结论自动当作事实；所有“根因”仍需人工复核。

## 2. 设计定位

### 2.1 用户与单一职责

| 用户 | 进入平台后最关心的问题 | 首屏应给出的答案 |
| --- | --- | --- |
| 业务负责人/管理者 | 用户体验是否在恶化，优先投入在哪里 | 问题量、严重度、趋势、Top 问题 |
| 运营/产品 | 哪类用户反馈最多，如何分派和跟进 | 标签、业务域、流程阶段、待处理清单 |
| 开发/算法 | 具体在哪段链路出现问题，是否可复现 | 请求 ID、耗时、工具状态、对话和 AI 假设 |

### 2.2 视觉与交互方向

产品定位为“AI 服务质量驾驶舱”，而不是传统后台表格。建议采用克制的航运调度语言：深海蓝作为信息底色，舱单米白用于阅读区，信号橙仅用于高风险告警，港口青用于“已改善/正常”。

建议 Token：

| 名称 | 色值 | 作用 |
| --- | --- | --- |
| 深海蓝 | `#102A43` | 顶栏、导航、核心数据标题 |
| 港口青 | `#007C91` | 正常、完成、可行动状态 |
| 舱单米白 | `#F7F4EE` | 页面背景与阅读面板 |
| 信号橙 | `#D96C21` | 高严重度、超时、需处理 |
| 海雾灰 | `#D9E2EC` | 分隔线、次级信息 |
| 墨黑 | `#243B53` | 正文与数据标签 |

独特交互元素：在首页使用一条“**问题航线图**”而非普通漏斗。它用五个 AI 流程节点（理解 → 补参 → 报价工具 → 知识检索 → 结果展示）展示问题从哪一段集中涌出；节点大小代表问题数，颜色代表严重度。它是业务人员理解“AI 到底哪里出问题”的最短路径。

可访问性要求：颜色不能作为唯一状态标识；高/中/低严重度必须配文字；图表均有表格/列表替代视图；支持键盘焦点和减少动画。

## 3. 无数据库阶段的数据架构

### 3.1 数据读取与归并

JSONL 是事件流，不是最终业务实体。页面不能直接逐行展示，必须在 Next.js 服务端按 `feedback_id` 做归并：

```text
chat-feedback.mock-100.jsonl
  ├─ feedback              ─┐
  └─ feedback_enrichment  ─┼─ 按 feedback_id 合并
                            ↓
                       FeedbackCase（页面使用的案例对象）
                            ↓
                Dashboard 聚合 / 列表过滤 / 详情查看
```

`FeedbackCase` 建议包含：

```ts
type FeedbackCase = {
  feedbackId: string;
  createdAt: string;
  sessionId: string;
  requestId?: string;
  feedbackText: string;
  dissatisfactionTypes: string[];
  contextAllowed: boolean;
  conversationSnapshot?: { userQuestion?: string; assistantAnswer?: string; conversationExcerpt?: string[] };
  trace: { traceFound: boolean; intent?: string; toolStatus?: string; totalElapsedMs?: number; retrievedDocsCount?: number };
  aiAnalysis: { status: 'completed' | 'failed' | 'pending'; qualityTags?: string[]; businessDomain?: string; pipelineStage?: string; severity?: string; confidence?: number; summary?: string; rootCauseHypothesis?: string; recommendedAction?: string; needsHumanReview?: boolean };
};
```

### 3.2 Next.js BFF API

浏览器不直接读取仓库文件。由 Next.js Route Handler 在服务端读取 JSONL、完成归并和聚合，然后返回经过裁剪的 JSON。

建议接口：

| 接口 | 用途 | 主要参数 |
| --- | --- | --- |
| `GET /api/feedback/overview` | 首页 KPI、趋势、问题航线图 | `from`、`to` |
| `GET /api/feedback/cases` | 可筛选、分页的反馈案例列表 | `page`、`pageSize`、`type`、`tag`、`stage`、`severity`、`status`、`q` |
| `GET /api/feedback/cases/:id` | 单条案例详情 | 无 |
| `GET /api/feedback/facets` | 筛选项、标签和计数 | 同筛选条件 |
| `GET /api/feedback/insights` | AI 或规则生成的阶段性洞察 | `from`、`to` |

实现原则：

- 文件路径仅在服务端环境变量中配置，例如 `CHAT_FEEDBACK_LOG_PATH`。
- 每次读取时校验 JSON，坏行跳过并记录服务端日志，不能让一个坏行导致整页 500。
- 100 条日志可每次读取；后续数据增大后，再增加 30～60 秒的内存缓存。
- API 返回列表摘要；完整对话只由详情 API 按需返回，减少隐私暴露和页面体积。
- 当前阶段“处理状态”仅在前端临时状态或 URL 中展示，不承诺刷新后保留；需要持久流转时再引入数据库。

## 4. 页面信息架构

建议第一期建设 6 个页面。导航按用户任务命名，避免出现“Enrichment”“Trace”等开发术语。

```text
质量总览
问题地图
反馈记录
反馈详情
数据与规则
```

### 页面一：质量总览

**目标：** 让负责人在 30 秒内看清服务质量、问题优先级和近期变化。

页面从上到下的结构：

```text
筛选栏：时间范围 | 业务域 | 是否仅看高风险 | 刷新时间
────────────────────────────────────────────────────────
本期质量摘要：总反馈 | 高优先级 | AI 分析覆盖率 | 平均响应耗时
────────────────────────────────────────────────────────
问题趋势（左，宽）                     问题类型占比（右）
────────────────────────────────────────────────────────
问题航线图：理解 → 补参 → 报价工具 → 知识检索 → 结果展示
────────────────────────────────────────────────────────
最需处理的 5 个问题                    最近反馈动态
```

#### KPI 卡片

| 卡片 | 计算 | 解读 |
| --- | --- | --- |
| 用户反馈数 | `feedback` 案例数 | 量级，不等同于失败数 |
| 高优先级问题 | `severity=high/critical` | 需要人工优先介入 |
| AI 归因覆盖率 | `completed / 全部案例` | 分析能力可用性；失败不等于无问题 |
| 平均响应耗时 | `total_elapsed_ms` 平均值 | 仅在存在链路耗时时展示 |

#### 图表

1. **反馈趋势折线图**：按天/周统计总反馈，叠加高优先级数量。用于发现回归或发布后的异常波峰。
2. **反馈类型横向条形图**：展示 `dissatisfaction_types`。支持多标签计数，图表标题需注明“同一条反馈可计入多个类型”。
3. **问题航线图**：ECharts `graph` 或固定 SVG + ECharts 提示层。节点对应 `pipeline_stage`，边不表示真实调用次数，而表示“问题定位到的阶段”；必须在说明中明确。
4. **优先处理清单**：不是图表，按“严重度 → 出现次数 → 最近时间”排序，展示标签、阶段、建议行动。

点击任一图表元素，自动带筛选条件跳转到“反馈记录”。

### 页面二：问题地图

**目标：** 让运营和产品判断“问题是什么、影响哪里、先做什么”。

该页采用二维矩阵，不用饼图堆砌：横轴是 AI 流程阶段，纵轴是质量标签，单元格颜色代表案例数，单元格内显示高优先级数。

```text
                理解  补参  报价工具  知识检索  回答生成  页面展示
数据/工具问题      ·    ·      9        ·        ·        ·
知识缺口          ·    ·      ·        8        8        ·
理解偏差          ·    5      ·        ·        ·        ·
响应慢            ·    6      ·        ·        ·        ·
显示问题          ·    ·      ·        ·        ·        8
```

#### 组件

- **问题矩阵（ECharts heatmap）**：单元格点击跳转列表。
- **严重度堆叠柱图**：按阶段展示低/中/高严重度构成，避免只看数量忽略风险。
- **AI 归因可靠性卡片**：完成、失败、待处理数量；对失败项目提供“一键查看”入口。
- **Top 标签说明卡**：把内部标签翻译为业务语言，例如 `tool_or_data_issue` 显示为“报价接口或数据异常”。

### 页面三：反馈记录

**目标：** 将 50 组案例变成可筛、可找、可交接的工作台。

#### 筛选与搜索

筛选项应使用中文业务名称，URL 中再映射到内部枚举：

- 时间范围：今天、近 7 天、近 30 天、自定义。
- 问题类型：响应慢、报价异常、回答不完整、理解错误、展示问题、其他。
- AI 判断标签：数据/工具异常、知识缺口、缺少信息、理解偏差、显示问题、响应慢。
- 发生环节：理解问题、补参、报价查询、知识检索、回答生成、页面展示。
- 严重程度：低、中、高、紧急。
- AI 分析状态：已完成、分析失败、待分析。
- 是否需要人工复核、是否授权查看上下文。
- 关键词：匹配反馈原文、用户问题、回答摘要、请求 ID。

#### 表格字段

| 列 | 面向非技术人员的显示 | 开发人员扩展信息 |
| --- | --- | --- |
| 时间 | `7月16日 16:29` | 精确时区 ISO 时间 |
| 用户反馈 | 前 60 字，可展开 | 完整原文 |
| 问题标签 | 中文 Tag Chip | 原始枚举值仅在 hover 中显示 |
| AI 判断 | 一句摘要 | 置信度、根因假设 |
| 发生环节 | 中文阶段名 | `pipeline_stage` |
| 风险 | 高/中/低 | `severity` |
| 链路状态 | 耗时、工具成功/失败 | intent、检索文档数 |
| 处理建议 | 建议行动 | 是否人工复核 |

默认排序：高风险优先，其次 AI 分析失败，再按时间倒序。每行点击进入详情；支持复制请求 ID 和导出当前筛选的 CSV（仅前端生成）。

### 页面四：反馈详情

**目标：** 给开发人员一条可复盘、可交接的完整证据链，同时向非技术人员解释技术信息。

```text
返回反馈记录     反馈 #fb_mock_0012        [高风险] [需要人工复核]
────────────────────────────────────────────────────────
用户说了什么                AI 判断了什么
反馈原文                    归因摘要 / 质量标签 / 置信度
用户问题                    可能原因（明确标为“AI 假设”）
助手回答                    建议行动
────────────────────────────────────────────────────────
对话与隐私                  链路摘要
最近对话片段                意图 | 耗时 | 工具状态 | 检索文档数
未授权时显示“用户未授权查看上下文”  请求 ID（可复制）
────────────────────────────────────────────────────────
处理记录（一期可仅显示，不持久保存）
待处理 → 已确认 → 处理中 → 已解决 / 无法复现
```

关键规则：

- `allow_context_for_review=false` 时，绝不尝试补齐用户对话；只显示“用户未授权查看上下文”。
- “可能原因”必须以 AI 假设标签显示，不能与真实日志证据混排。
- 展示原始字段时给出中文解释。例如 `tool_status=failed` 旁显示“报价工具调用失败”。
- “跳转关联请求”暂不实现真实链路页，可复制 `request_id` 用于服务日志平台搜索。





### 页面五：数据与规则

**目标：** 建立信任，让非技术人员理解报表口径，让开发人员能排查数据质量。

内容包括：

- 数据源文件、最近读取时间、有效行数、跳过坏行数。
- 事件配对逻辑：`feedback_id` 对应一条原始反馈和一条 AI 补充分析。
- 字段字典：内部枚举与中文名称映射。
- 指标口径：多标签反馈如何计数、AI 失败是否计入覆盖率、未授权上下文如何处理。
- 数据限制：当前为模拟文件、无持久化处理状态、不是实时流。
- AI 结论说明：AI 只提供分类、摘要和假设，不自动判定真实根因。

## 5. ECharts 图表配置建议

| 图表 | ECharts 类型 | 关键交互 | 常见误区 |
| --- | --- | --- | --- |
| 反馈趋势 | `line` + `bar` | 缩放、点击日期过滤 | 样本太少时不要过度解读趋势 |
| 问题类型 Top N | `bar` 横向条形 | 点击类型进入列表 | 多标签会使合计大于案例数，需说明 |
| 严重度构成 | `bar` 堆叠 | 点击阶段或色块 | 不建议用饼图比较多个小比例 |
| 问题矩阵 | `heatmap` | 单元格筛选 | 0 值也应保留，避免误解为缺数据 |
| 问题航线图 | `graph` | 节点 hover 展示问题与案例 | 不能误称为调用拓扑或因果链 |
| 响应耗时分布 | `bar` 分桶/直方图 | 按区间筛选 | 没有耗时字段的案例不能强行计为 0 |

图表通用规范：所有 Tooltip 必须显示“案例数”“高风险数”“筛选范围”；响应式容器使用 `ResizeObserver`；在移动端切换为单列，热力图改为可滚动表格或分阶段列表。

## 6. 面向不同角色的中文映射

避免让非技术人员直接面对英文枚举。建议建立固定映射：

| 原始值 | 平台显示 |
| --- | --- |
| `slow_response` | 响应慢 |
| `quote_result_issue` | 报价结果异常 |
| `incomplete_answer` | 回答不完整 |
| `misunderstood_question` | 未理解用户问题 |
| `tool_or_data_issue` | 报价工具或数据异常 |
| `knowledge_gap` | 业务知识缺口 |
| `slot_extraction` | 识别货运条件 |
| `freight_tool` | 查询报价接口 |
| `rag_retrieval` | 检索业务资料 |
| `rag_generation` | 生成业务回答 |
| `frontend_display` | 页面展示 |

严重度文案：

- 高：影响用户获得正确业务结果，或同类问题反复出现。
- 中：影响体验或回答完整性，但可通过人工补充处理。
- 低：展示、措辞等体验问题，暂不阻断业务。

## 7. AI 可以应用的地方

### 7.1 第一优先级：反馈自动归因（已有基础，需可视化）

当前 `feedback_enrichment` 已包含 AI 的分类、阶段、严重度、置信度、假设和建议行动。平台应把它转为可审核的辅助结论，而不是黑盒评分。

- 自动总结长反馈为一句“问题摘要”。
- 将用户语言映射为统一标签与流程阶段。
- 给出带不确定性措辞的根因假设。
- 当置信度低、严重度高、或 AI 分析失败时自动标记“需要人工复核”。

### 7.2 第二优先级：相似问题聚类

当日志量变大后，可对 `feedback_text + user_question + AI summary` 做 embedding 聚类，形成“问题主题”，例如“PVG-LAX 运价工具失败”“锂电池资料回答不完整”。

输出应是：主题名称、最近出现次数、代表案例、涉及阶段、变化趋势。必须允许运营人员修改主题名称或拆分误聚类。

### 7.3 第三优先级：周报与异常说明

让 AI 基于已聚合的**脱敏统计数据**生成周报草稿：

```text
本周反馈量较上期增加 18%。主要增量来自“报价结果异常”，集中在报价工具阶段。
高优先级案例中，4 条请求存在工具失败或数据异常信号；建议优先核验上游接口状态。
```

AI 不能自行编造“上升”“下降”或具体根因；系统必须把计算出的时间范围、分子/分母和原始指标作为输入。周报需人工确认后再发布。

### 7.4 第四优先级：开发辅助排查

对单个案例，AI 可将“用户问题、助手回答、链路摘要、相似历史案例”整理成研发可读的排查卡：

- 可复现步骤；
- 预期与实际行为；
- 关联流程阶段；
- 可能需要查看的日志关键词；
- 建议验证命令/测试方向。

这里 AI 只生成排查建议，不执行生产操作，也不替代研发结论。

### 7.5 暂不建议 AI 自动化的事项

- 自动关闭问题或自动标记“已解决”。
- 根据单个用户反馈直接修改 Prompt、路由或报价规则。
- 将包含未授权对话的内容送入模型。
- 把 AI 的根因假设直接计入业务 KPI。

## 8. 交付分期

### Phase 1：可浏览、可定位（建议首期）

1. 服务端 JSONL 读取、容错解析、按 `feedback_id` 合并。
2. 质量总览、问题地图、反馈记录、反馈详情四个页面。
3. 基础筛选、URL 状态同步、CSV 导出、中文枚举映射。
4. ECharts 趋势、条形、热力图和问题航线图。
5. 明确展示“模拟数据/无数据库持久化”的状态。

验收标准：业务人员能从 Top 问题点击进入案例；开发人员能复制 request ID、看到链路摘要和 AI 假设；未授权上下文不泄露。

### Phase 2：可跟踪、可复盘

1. 接入数据库，存储处理状态、负责人、版本和解决结论。
2. 改进跟踪页从演示状态变成真实工单台账。
3. 增加发布版本与反馈趋势对比。
4. 引入相似反馈聚类、AI 周报草稿和人工审核流。

### Phase 3：可预警、可主动发现

1. 从 JSONL 文件切换到实时日志流或分析仓库。
2. 当某标签、阶段或高严重度比例超过基线时推送预警。
3. 对报价工具失败、RAG 空召回、模型超时设置独立 SLO。
4. 将“问题主题 → 研发任务 → 发布版本 → 改善效果”串成闭环。

## 9. 风险与注意事项

| 风险 | 影响 | 处理原则 |
| --- | --- | --- |
| JSONL 读写并发 | 读到半行或坏 JSON | 服务端逐行容错、原子写入、坏行计数 |
| AI 归因失败 | 看板出现缺失 | 单独展示“分析失败”，不丢弃原始反馈 |
| 多标签计数 | 类型总和大于案例数 | 图表标题和数据说明明确标注 |
| 小样本趋势 | 误导决策 | 标注样本量，少于阈值不做变化结论 |
| 隐私上下文 | 对话内容泄露 | 严格按 `allow_context_for_review` 显示与传递 |
| 无数据库状态 | 处理动作刷新即丢失 | 首期仅提供浏览/筛选，不承诺协同闭环 |

## 11. 下一步决策

建议先实现 Phase 1，并先确认以下三项口径：

1. 首页默认时间范围：建议近 30 天；模拟数据不足时按全量展示并提示。
2. 高优先级定义：建议先采用 `severity=high/critical`，不要用 AI 置信度代替严重度。
3. “处理状态”是否首期展示：建议展示为“仅浏览演示”，不提供保存按钮，避免用户误认为已进入真实工单流程。

这样可以在不引入数据库的前提下，先获得真正可用的质量观察与问题定位能力；等数据量、使用频率和协作流程稳定后，再进入持久化与自动化阶段。
