# AI询价方案记录

日期：2026-05-06

## 目标

在不改变现有 `/api/chat` SSE 协议的前提下，升级 AI 运价查询能力，覆盖以下需求：

1. 支持接口新增字段：
   - 必填：`sfg`、`mdg`、`inputWeight`、`inputVol`、`hbrq` 或 `hbrqBegin+hbrqEnd`
   - 可选：`flightType`、`packageType`、`cargoType`、`twoCode`、`gid`
2. 结果输出改为稳定的 Markdown 文本：
   - 有精确结果：日期 + Markdown table + 内容总结
   - 无精确结果但有类似结果：提示说明 + 类似日期 Markdown table
   - 完全无结果：人工引导文案
3. 缺少必填参数时，一次性告诉用户缺哪些必填项
4. 可选字段默认不追问，优先从用户语义中抽取，缺失则先查
5. 当上一轮已完成查询并返回结果后，如果用户发起新的完整询价，则清空旧状态
6. 类似运价自动查，且第一版仅向后放宽 7 天，不跨目的港，不放宽重量体积

## 产品口径

### 新的完整询价判定

仅在上一轮已完成查询的前提下启用。

满足以下条件时，视为新询价并清空旧状态：

1. 当前消息重新出现新的起运港/目的港线索
2. 同时带了重量/体积/日期中的至少两项

### 类似运价

精确查询无结果时，自动执行类似运价查询：

1. 保持以下条件不变：
   - `sfg`
   - `mdg`
   - `inputWeight`
   - `inputVol`
   - 其他已抽取的可选筛选项
2. 仅放宽日期，范围为：
   - 单日期查询时，从目标日期次日起，向后 7 天
3. 区间查询不再自动做类似日期扩展

### 无数据分层

1. 无精确结果，但有类似结果：返回相近日期参考价
2. 精确结果和类似结果都没有：返回人工引导文案

## 技术设计

### 状态扩展

新增以下会话字段：

- `hbrqBegin`
- `hbrqEnd`
- `flightType`
- `packageType`
- `cargoType`
- `twoCode`
- `gid`
- `query_completed`

### 节点策略

#### intent_node

1. 保留现有意图识别
2. 增加“上一轮已完成 + 新完整询价”的会话重置规则
3. 保留“承接上文的继续报价”优先命中 `rate_query`

#### slot_node

1. 扩展槽位抽取 Prompt，支持新增字段
2. 保留“新值 > 明确清空/未知 > 旧 context”的合并优先级
3. 日期支持：
   - 单日期 `hbrq`
   - 区间 `hbrqBegin+hbrqEnd`
4. 缺参判断改为：
   - `sfg`
   - `mdg`
   - `inputWeight`
   - `inputVol`
   - `hbrq` 或 `hbrqBegin+hbrqEnd`

#### ask_node

改为代码生成追问文案，一次性告诉用户当前缺失的所有必填项。

#### tool_node

调用运价接口时支持：

1. 精确查询
2. 单日期无结果时，自动查询类似日期

并向上游返回：

- `search_mode=exact`
- `search_mode=similar`
- `search_mode=none`

#### result_node

改为代码生成稳定 Markdown 输出：

1. 精确结果：日期分组 + 表格 + 总结
2. 类似结果：说明 + 日期分组 + 表格
3. 完全无结果：人工引导

## 风险点

1. 新完整询价判定过宽，可能误清空上下文
2. 槽位抽取对区间日期的稳定性依赖 LLM
3. 类似运价仅按日期扩展，若接口返回量大，需要后续考虑排序与截断
4. Markdown 表格输出需要与前端最终展示效果联调

## 验证重点

1. 新完整询价是否正确清空旧状态
2. 续问型短句是否仍能承接前文
3. 一次性缺参提示是否稳定
4. 单日期无结果时是否自动查未来 7 天
5. 类似运价是否严格保持航线与重量体积不变
6. 区间查询是否不触发类似运价
7. Markdown 表格是否结构稳定
