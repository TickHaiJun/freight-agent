## 问题 1：没有先把“卡点”定位到具体节点，就容易围着表象反复试

**Tag**
`#根因定位` `#日志分析` `#不要猜`

**今天的现象**
- RAG 问题一发起，请求长期卡住。
- 日志总是停在：
  `rag.embeddings parsed`
- 一开始容易怀疑是：
  - DeepSeek 意图识别慢
  - embedding 接口慢
  - LangGraph 流程写错

**最后确认的根因**
- 真正的卡点不在意图识别，也不在 embedding HTTP 请求。
- 卡点在 `Chroma similarity_search_with_score(...)` 返回阶段。
- 也就是说，embedding 已经成功回来了，但本地向量库查询没有正常返回。

**为什么今天会反复**
- 早期看到“最后一条日志”就容易把它当成根因。
- 但“最后一条日志”只说明程序执行到了那里，不代表问题出在那里。
- 没把调用链完整串起来时，修改会变成碰运气。

**对我的影响**
- 时间被消耗在错误方向上，排障节奏被打乱。
- 情绪上容易着急，开始怀疑很多其实没问题的模块。
- 会让后续每次改动都缺少信心，觉得“是不是哪里都可能有问题”。

**以后怎么避免**
- 先画调用链，再看日志：
  `intent -> query_analyzer -> retriever -> vector/bm25 -> answer`
- 明确回答两个问题后再改代码：
  1. 最后一条成功日志是什么
  2. 下一条本该出现但没有出现的日志是什么
- 只有确定“卡在 A 和 B 之间”，才动对应代码。

**一句话提醒**
- 日志停在哪里，不等于问题就在哪里。

## 问题 2：把不稳定组件放在主链路前面，导致整个接口被拖死

**Tag**
`#系统设计` `#主链路稳定性` `#降级策略`

**今天的现象**
- 知识库问题会卡住整个 `/api/chat`。
- SSE 没有继续返回文本，也没有及时失败。
- 用户侧看到的不是“结果不准”，而是“系统没反应”。

**最后确认的根因**
- RAG 主链路默认先走向量检索。
- 而本地 Chroma collection 读操作已经不健康，甚至 `count()` 都能卡住。
- 结果是：一个不稳定依赖把整个请求同步拖死。

**为什么今天会反复**
- 一开始默认认为“向量检索是主方案，BM25 是补充”。
- 但在当前环境里，实际应该反过来：
  - 先保证 BM25 可返回
  - 再把向量检索作为增强能力

**对项目的影响**
- 接口可用性下降，比“答得不够准”更严重。
- 前端会误以为是 SSE 或后端整体故障。
- 排障时很难区分“检索质量问题”和“系统稳定性问题”。

**对我的影响**
- 容易陷入“继续修向量库也许下一次就好了”的思路。
- 忽略了一个更高优先级目标：先让系统稳定返回。

**以后怎么避免**
- 所有新能力接入前先问：
  “这个组件如果挂了，主链路还能不能返回？”
- 对检索链路强制设计三层：
  1. 主路径
  2. 超时/熔断
  3. 降级路径
- 当前项目里，应该优先遵守：
  - `/api/chat` 不能因为 RAG 子模块挂掉而整体卡死

**一句话提醒**
- 不稳定模块不能站在主链路最前面。

## 问题 3：环境能力没有先确认，导致方案在本机上根本跑不通

**Tag**
`#环境依赖` `#本机约束` `#先验证再设计`

**今天的现象**
- `.doc` 文件解析一直不稳定。
- 后来验证发现本机没有可用的 `Word.Application` COM 环境。
- 报错：
  `无效的类字符串`

**最后确认的根因**
- 方案里默认使用 Windows + Word COM 读取旧版 `.doc`。
- 但当前机器上这个前提并不成立。
- 所以不是代码逻辑错了，而是运行条件不满足。

**为什么今天会反复**
- 先写了依赖 Word 的方案，再在本机上验证。
- 但这类文件解析能力，本来应该在方案落地前先确认。

**对项目的影响**
- BM25 现场构建无法完整覆盖 4 个关键锂电池 `.doc` 文件。
- 知识库内容不完整，导致即使系统返回了，也可能答不到点上。

**对我的影响**
- 会误以为“是检索写得不对”，其实是资料根本没进库。
- 容易在代码层过度排障，忽略运行环境这个更基础的层面。

**以后怎么避免**
- 只要涉及文件解析、COM、OCR、本地软件依赖，先做环境检查。
- 先验证：
  - 是否安装 Word
  - 是否能成功调用 COM
  - 是否能实际读到 `.doc`
- 如果环境不确定，优先选更稳定的输入格式：
  - `.docx`
  - `.pdf`
  - `.txt`

**一句话提醒**
- 方案能不能跑，先看环境，不要先看代码。

## 问题 4：离线入库没有真正跑通，却默认认为“知识库已经可用”

**Tag**
`#数据准备` `#知识库建设` `#不要假设`

**今天的现象**
- `data/chroma` 目录里有文件，看起来像是已经建库了。
- 但 `data/cache/bm25.pkl`、`data/exports/chunks.json` 之前其实都没有。
- 说明“完整离线入库”并没有真正成功完成。

**最后确认的根因**
- 看到目录存在，就下意识以为索引已经建好。
- 但真正代表知识库可用的，不是目录存在，而是：
  - 文档成功解析
  - chunk 成功导出
  - BM25 成功建索引
  - 向量库可正常读写

**为什么今天会反复**
- 把“有目录/有 sqlite 文件”和“知识库可用”混为一谈。
- 缺少一份明确的建库完成判定标准。

**对项目的影响**
- 检索链路调了很多次，但基础数据其实没准备好。
- 会导致线上排障和离线建库问题交织在一起，越查越乱。

**对我的影响**
- 排障心智模型失真：
  以为是在调召回质量，实际上还卡在“有没有数据”阶段。

**以后怎么避免**
- 每次建库后固定检查 4 个输出：
  1. `data/exports/chunks.json`
  2. `data/cache/bm25.pkl`
  3. collection 可 `count()`
  4. 抽样 query 能命中
- 把“知识库初始化成功”的定义写死，不靠感觉判断。

**一句话提醒**
- 目录存在，不代表知识库可用。

## 问题 5：没有先确认“服务是不是已经加载了新代码”，导致同一问题反复看日志

**Tag**
`#发布验证` `#服务重启` `#版本确认`

**今天的现象**
- 代码改了多轮，但日志还是继续出现：
  - `rag.vector_store`
  - `rag embeddings request start`
  - `rag embeddings parsed`
- 容易怀疑“是不是改动没生效”。

**最后确认的根因**
- 改代码和运行服务是两件事。
- 如果服务进程没有重启，或者没加载最新代码，日志当然还是旧行为。

**为什么今天会反复**
- 改完后更关注“结果是不是好了”，没先确认“运行的是不是新版本”。

**对项目的影响**
- 会把“代码问题”和“部署/运行问题”混在一起。
- 调试时间被白白拉长。

**对我的影响**
- 容易产生挫败感：
  明明改了，为什么一点反应都没有。

**以后怎么避免**
- 每次修改关键链路后，先验证 3 件事：
  1. 服务已重启
  2. 新日志关键字是否出现
  3. 旧日志关键字是否消失
- 例如今天这个场景，改完“关闭向量检索”后，应该立即确认：
  - 出现：`skip vector search`
  - 不再出现：`rag.embeddings`

**一句话提醒**
- 先确认跑的是新代码，再判断代码有没有效果。

## 问题 6：过早追求“完整 RAG”，而不是先拿到最小可用闭环

**Tag**
`#最小可行解` `#工程节奏` `#不要贪全`

**今天的现象**
- 一开始想同时把：
  - Chroma
  - embedding
  - BM25
  - `.doc` 解析
  - generator
  都一起跑通。

**最后确认的结论**
- 当前最重要的不是“完整技术路线全部上线”，而是：
  - 用户发知识库问题时系统能返回
  - 不要卡死
  - 有基本可解释的结果

**为什么今天会反复**
- 容易把“最终目标”直接当成“今天必须一次到位的目标”。
- 这样一旦某个基础条件没准备好，整体就一起受阻。

**对项目的影响**
- 推进效率低。
- 任何一个子环节不稳定，都会拖住整个阶段目标。

**对我的影响**
- 容易焦虑，觉得进展很慢。
- 实际上问题不是慢，而是目标设定过满。

**以后怎么避免**
- 拆成分层目标：
  1. 能返回
  2. 能稳定
  3. 能答准
  4. 再优化召回和排序
- 今天这类问题，正确顺序应是：
  - 先 BM25-only 跑通
  - 再恢复向量检索
  - 最后再做 hybrid 和 rerank

**一句话提醒**
- 先闭环，再完整；先能用，再优秀。
